Comprehensive audit completed mapping pane's actor framework
against Fowler / Hu's EAct calculus (Maty / Maty_zap).

**Why:** Lane requested an exhaustive audit to verify pane
faithfully implements EAct guarantees, as the architecture spec
claims par + EAct as the formal foundation.

**How to apply:** Use these findings when evaluating any
changes to the looper, dispatch, destruction sequence, or
session layer. The three conditions for soundness are
load-bearing.

## Key findings

### 1. Verdict: Conditionally sound

Three conditions must hold:

- **I2 / I3 enforced** (watchdog timer) for Global Progress
  — **NOW HOLDS:** heartbeat watchdog landed in
  `architecture/looper`, post-2026-04-06.
- **Active-phase state-machine tracking absent** (deliberate,
  consistent with EAct's own Scala impl which checks linearity
  dynamically) — still by design.
- **Bilateral-only sessions** (multiparty compliance by
  design review, not automated projection) — still by design.

### 2. Property mapping (EAct → pane)

- **Preservation** (Thm 4): handshake phase via par typestate,
  active phase via exhaustive enum match. State-sequencing gap
  is real but by design.
- **Progress** (Thm 5): depends on I2 / I3 (now enforced via
  watchdog). I6 is structural.
- **Global Progress** (Cor 1): process isolation gives stronger
  Lemma 1 than EAct. Same I2 / I3 caveat (now satisfied).
- **Failure preservation** (Thm 6): destruction sequence maps
  to E-RaiseS + cascading. `dispatch.clear()` is more
  aggressive than E-CancelH (no callback for non-primary
  entries).
- **Failure progress** (Thm 7 / 8): sound, backstopped by fd
  hangup.

### 3. Drop compensation is the correct affine-gap strategy

Fowler / Hu explicitly endorse this approach (§4, citing
Mostrous / Vasconcelos 2018, Harvey et al. 2021,
Lagaillardie et al. 2022, Fowler et al. 2019).
`ReplyPort` → `ReplyFailed` is the pane analog of Maty_zap's
failure continuation.

### 4. Gap priorities (PARTIALLY STALE — annotated)

- ~~MODERATE: No active-phase state-machine enforcement (fixable via protocol_handler macro runtime checks)~~ — still open as a deliberate design choice
- ~~MODERATE: I2 / I3 not enforced (fixable via watchdog timer)~~ — **RESOLVED** by heartbeat watchdog (post-2026-04-06, see `architecture/looper`)
- LOW: No multiparty support (by design)
- LOW: `dispatch.clear()` drops without callbacks (backstopped
  by fd hangup)
- MINIMAL: `CancelHandle` Drop no-op (correct semantics)

Additional resolutions since 2026-04-05:

- **I8 (`send_and_wait` panics from looper thread)** — now
  enforced via stored ThreadId, see `decision/wire_framing` and
  `architecture/looper`
- **S3 (six-phase batch ordering)** — implemented in calloop
  Looper

### 5. Extensions (optics, pane-fs) are orthogonal

They operate on handler internal state, not session
communication. No EAct property is affected.
