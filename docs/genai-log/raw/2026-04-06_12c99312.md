> ## 1. MonadicLens as the Duploid Bridge
> 
> **The formal setup (Clarke et al. Def. 4.6 / Prop. 4.7).** MndLens_Ψ((A,B),(S,T)) := W(S,A) x W(S x B, ΨT), which is isomorphic to Optic_(x, ⋊) where:
> - Left action (x) : W x W -&gt; W is the cartesian product, acting on the "pure" category W
> - Right action (⋊) : W x Kl(Ψ) -&gt; Kl(Ψ) is the Kleisli action induced by the strength of Ψ
> 
> In pane's concrete encoding:
> - `view: fn(&amp;S) -&gt; A` is a morphism S -&gt; A in W (pure)
> - `set: fn(&amp;mut S, A) -&gt; Vec&lt;Effect&gt;` is a morphism S x A -&gt; ΨS in Kl(Ψ) (effectful), where Ψ wraps state mutation + effect list production
> 
> **The polarity question.** In a duploid dupl_{L,R} arising from an adjunction L ⊣ R between categories C (positive) and D (negative), morphisms between positive and negative objects are where non-associativity lives. Here:
> 
> - **S is negative.** The handler state is defined by its behavior under message dispatch. You don't inspect it directly from outside — you observe it through projections (attr reads, compositor draws). This is exactly the "defined by elimination" character of negative types in the CBN subcategory.
> 
> - **A is positive.** The focused value is data — a usize, a String, a bool. It has a canonical form. You construct it, pattern match on it, serialize it. This is the "defined by introduction" character of positive types in the CBV subcategory.
> 
> **So MonadicLens is a cross-polarity morphism.** The view component S -&gt; A goes from negative to positive (extracting data from behavior). The set component S x A -&gt; ΨS goes from (negative x positive) to negative, with the Kleisli wrapper adding the effect dimension.
> 
> In the duploid picture, a morphism from a negative object to a positive object is precisely a "cross-polarity" morphism, and the claim in the duploid literature (Mangel, Mellies, Munch-Maccagnoni) is that these are where associativity may fail. However, the direction matters:
> 
> - **Negative-to-positive (view):** This corresponds to ↓ (positive shift / thunk evaluation) — extracting a value from a computation. In duploid terms, this is well-behaved (it's a "value" morphism, always associative).
> 
> - **Positive-to-negative (set):** This corresponds to ↑ (negative shift / suspension) — embedding data into a computation. The Kleisli action ⋊ is doing exactly this: taking pure data (the new value A) and producing an effectful state transformation.
> 
> **The non-associativity question.** The MonadicLens *itself* is not a single duploid morphism — it is a *pair* of morphisms with opposite polarity directions. The view component lives entirely in the co-Kleisli/pure subcategory. The set component lives in the Kleisli/effectful subcategory. The monadic lens bundles them together as a mixed optic. The fact that it is *mixed* — Optic_(x, ⋊) with two different actions — is precisely the signature of a cross-polarity construct. The non-associativity that duploids warn about would arise if we tried to compose morphisms that go positive→negative→positive in an uncontrolled way, which the optic framework prevents by keeping the two actions separate.

> **Assessment:** MonadicLens genuinely sits at the polarity boundary in the duploid. (S = negative, A = positive) is not a labeling choice — it follows from the categorical structure. But the mixed optic framework already handles this junction without needing duploid machinery explicitly.

> ## 3. Profunctor Encoding of Duploid Morphisms
> 
> **Clarke et al. Theorem 4.9 (the profunctor representation theorem):**
> 
> ∫_{P ∈ Tamb} V(P(A,B), P(S,T)) ≅ Optic((A,B),(S,T))
> 
> For MonadicLens specifically, the Tambara constraint is for profunctors that are simultaneously Cartesian (preserving the x action) and "Kleisli-strong" (preserving the ⋊ action). The profunctor representation would be:
> 
> ```
> type MonadicLensP Ψ s t a b = forall p. (Cartesian p, KleisliStrong Ψ p) =&gt; p a b -&gt; p s t
> ```
> 
> where `KleisliStrong Ψ p` expresses that p preserves the Kleisli action ⋊.
> 
> **Is this the profunctor encoding of a duploid morphism?** The answer depends on what we mean. Clarke et al.'s theorem gives us:
> 
> 1. **Composition is function composition.** MonadicLens profunctor optics compose via `.`, just like ordinary profunctor optics. The constraint sets merge under composition.
> 
> 2. **The two monoidal actions are explicit.** The Optic_(x, ⋊) formulation makes the asymmetry between view (cartesian action) and set (Kleisli action) visible at the type level. This is the "mixed" character.
> 
> 3. **The representation theorem applies to mixed optics.** Clarke et al. explicitly extend Pastro-Street to the mixed case. This is their main contribution.
> 
> Now, in a duploid, a morphism from positive A to negative B would be an element of D(A, B) in the duploid category. The duploid's composition rule for cross-polarity morphisms uses the shift operators to mediate. The profunctor encoding wraps this mediation into the Tambara constraint: the requirement that P be both Cartesian and KleisliStrong *is* the encoding of the shift mediation.
> 
> **Concrete consequence from Theorem 4.9:** The profunctor representation gives us:
> 
> - **Composability.** MonadicLens composes with ordinary Lenses (inheriting the Cartesian constraint) and with Kleisli morphisms (inheriting the KleisliStrong constraint). The composed optic's constraints are the union.
> 
> - **The Tambara module structure tells us what "bridges" are available.** A Tambara module for (x, ⋊) must respect both actions simultaneously. The number of such modules determines the expressiveness of the optic — what operations you can perform through it. For MonadicLens, you get view and set, but not (for example) traverse, because Monoidal is not in the constraint set.
> 
> **What this does NOT give us:** The profunctor representation does not directly tell us about the *non-associativity* of duploid morphisms. The profunctor category Tamb is always a category (composition is associative). The non-associativity in the duploid lives at a different level — it's about composing the *underlying* morphisms without the profunctor wrapping. The profunctor encoding *resolves* the non-associativity by imposing the Tambara constraint.

> ## Framing
> 
> The claim is that Plan 9's design lives primarily in the co-Kleisli subcategory (CBN/comonadic/negative polarity) of a duploid. Before classifying individual abstractions, I need to be precise about what the polarities mean in this context:
> 
> - **Negative/CBN/co-Kleisli**: Defined by *elimination forms* (what you can do with it). The value doesn't exist until demanded. A negative type is a computation waiting for a consumer. The canonical example: a function type `A -&gt; B` — you can't inspect it, you can only apply it.
> 
> - **Positive/CBV/Kleisli**: Defined by *introduction forms* (what it contains). The value exists independently of any consumer. A positive type is data waiting to be examined. The canonical example: a sum type `A + B` — you pattern-match on it, it tells you which variant it is.
> 
> - **Shifts** mediate between the two subcategories. A shift-down (thunk) suspends a positive value into a negative computation. A shift-up (force/return) reifies a negative computation into a positive value.
> 
> ## 1. Classification of Plan 9 Abstractions
> 
> ### Files in General: **Primarily Negative, with a Positive Cache Layer**
> 
> The claim that file reads are negative is correct and well-supported by the primary sources. From intro(5):
> 
> &gt; "A client transmits requests (T-messages) to a server, which subsequently returns replies (R-messages) to the client."
> 
> `read(fd, buf, n)` is demand-driven. The client provides the demand (offset, count) and a continuation (the buffer to fill, the tag to reply to). The server computes the response. This is CBN: the content of the file is not a stored value presented to the client — it's a computation the server performs in response to a demand.
> 
> But here's where it gets interesting. Plan 9 has the `-C` flag on bind/mount (from bind(1)):
> 
> &gt; `-C`: "cache — kernel may cache read data; currency verified at each open"
> 
> The cache transforms a negative type into a positive one. A cached read returns stored data (positive: here is the data, pattern-match on it) rather than demanding computation from the server. The `open` acts as a freshness check — a shift from the cached positive value back to the authoritative negative computation.
> 
> **Classification**: Files are negative by default. The `-C` cache flag is an explicit shift-up (force), converting the negative computation into a positive cached value. The staleness-check-on-open is the re-thunking that moves back.
> 
> This is a clean polarity structure. Plan 9's design makes the shift *explicit and optional*, which is exactly right.
> 
> ### Synthetic Filesystems (/proc, /net, /srv, /env): **Strongly Negative**
> 
> The names paper is explicit:
> 
> &gt; "Their contents are synthesized on demand when read; when written, they cause modifications to kernel data structures."
> 
> `/proc/N/status` is the clearest negative type in Plan 9. It has no stored value at all. The "file" is defined entirely by its behavior under read (returns 12 formatted fields of process state), write (not applicable — it's read-only), and stat (returns metadata). This is negative polarity by definition: defined by elimination, not introduction.
> 
> `/proc/N/ctl` is negative-accepting-commands: write "stop" and the process stops. Write "kill" and it dies. The file is an interface (a collection of eliminators), not a container.
> 
> `/proc/N/wait` is a negative type with a blocking elimination: "If the process has no extant children, living or exited, a read of wait will block." The read *is* the computation — it blocks until there's something to report. There is no pre-existing value to extract.
> 
> `/env` is more interesting. From env(3):
> 
> &gt; "The env device serves a one-level directory containing files with arbitrary names and contents."
> 
> Environment variables are stored. They have values independent of any consumer. When you `read /env/PATH`, you get stored bytes. This is a **positive type presented through a negative interface**. The file protocol wraps it negatively (you demand via read), but the underlying entity — a stored string — is positive. The fork semantics confirm this: `rfork(RFENVG)` copies the values (positive duplication), and from that point "changes made in one process group cannot be noticed by the other."
> 
> `/srv` is a peculiar case. From srv(3):
> 
> &gt; "srv is a bulletin board on which processes may post open file descriptors to make them available to other processes."
> 
> The entries in `/srv` are *references to capabilities* (open file descriptors). A reference is neither purely positive nor purely negative — it's a pointer to a negative interface. You open `/srv/myserv` and you get a file descriptor to the posted service. The srv entry itself is a positive datum (it exists, it holds a reference), but what it refers to is negative (a server ready to accept operations).
> 
> **Classification**:
> - `/proc/*`: Strongly negative. Synthesized on demand.
> - `/net/*`: Strongly negative. Network stack state computed on access.
> - `/env/*`: Positive values behind a negative interface. The shift is the file protocol itself.
> - `/srv/*`: Positive references to negative interfaces. A capability store.
> 
> ### 9P Messages: T-messages vs R-messages
> 
> This is where your question about whether T/R is the positive/negative split or the cut gets precise.
> 
> Looking at the wire format (from intro(5)):
> 
> **T-messages carry demands.** `Tread(tag, fid, offset, count)` says "I want `count` bytes at `offset` from `fid`, reply to `tag`." The tag IS the continuation — it's how the client correlates the response. The T-message is a *negative action*: it's an elimination applied to the server's file tree.
> 
> **R-messages carry data.** `Rread(tag, count, data)` delivers the bytes. This is a positive value — here is the data, take it. `Rstat(tag, stat)` delivers a struct — positive.
> 
> But this isn't the whole story. `Rerror(tag, ename)` is also positive — it's a sum type response. `Rread | Rerror` is exactly `data + error`, a sum/coproduct, which is the canonical positive construction.
> 
> **However**: the T-message is *not* the left side of a cut. A cut in sequent calculus eliminates a formula that appears on both sides. T/R is more precisely a **function application**: the T-message is an argument applied to the server (which is a negative function-like thing), and the R-message is the return value (positive data).
> 
> The T/R pair together form a single interaction, which in the duploid is a morphism in the co-Kleisli category: `!A -&gt; B` where `!A` is the server resource (comonadic — available for repeated use) and `B` is the response data.
> 
> Actually, let me refine this. The server is the comonad. It's a resource that can be repeatedly queried without being consumed. `!A` in linear logic means "as many copies of A as you want" — and that's exactly what a file server is. You can read the same file at the same offset any number of times. The fid represents a comonadic reference: it grants repeated access to a computation.
> 
> **Classification**: T-messages are co-Kleisli morphisms (demands applied to the comonadic server). R-messages are the return values of those morphisms (positive data). The T/R pair is not a cut — it's the evaluation of a co-Kleisli map.
> 
> ### Fids: **Comonadic References**
> 
> A fid is a 32-bit handle chosen by the client (from intro(5)):
> 
> &gt; "Fids are somewhat like file descriptors in a user process, but they are not restricted to files open for I/O: directories being examined, files being accessed by stat calls, and so on — all files being manipulated by the operating system — are identified by fids."
> 
> A fid is a handle to a comonadic resource. You can use it multiple times (read at different offsets, stat it, walk from it). It's not consumed by use. `clunk` explicitly releases the comonadic reference.
> 
> This maps directly to the comonad's `!A` in linear logic. The fid is the token that witnesses `!A` — it grants the right to use the resource arbitrarily many times. `clunk` is the dereliction that drops the comonadic modality.
> 
> One crucial detail from walk(5): `Twalk(fid, newfid, wnames...)` takes an existing fid and produces a new fid pointing to a descendant. This is the comonad's `duplicate : !A -&gt; !!A` — from one comonadic reference, derive another. The original fid is not consumed.
> 
> **Classification**: Strongly comonadic/negative. Fids are `!`-typed references.
> 
> ### Qids: **Positive Identifiers**
> 
> From intro(5):
> 
> &gt; "The qid represents the server's unique identification for the file being accessed: two files on the same server hierarchy are the same if and only if their qids are the same."
> 
> A qid is a triple (type: byte, version: u32, path: u64). It's pure data. You can compare qids, inspect the type byte, check the version. It's defined by its structure, not by what you can do with it.
> 
> **Classification**: Positive. A qid is data. It's the positive content returned by negative operations (attach, walk, open, create all return qids in their R-messages).
> 
> ### Pipes: **The Cut — but with an Asymmetry**
> 
> From pipe(3):
> 
> &gt; "An attach of this device allocates two new cross-connected I/O streams, `dir/data` and `dir/data1`. Data written to one channel becomes available for reading at the other."
> 
> A pipe connects a writer (producer of positive data) to a reader (consumer demanding negative computation). This IS the cut rule: it juxtaposes two derivations that share a formula, one on the left (producing it) and one on the right (consuming it).
> 
> In `a | b`, process `a` writes to its stdout (introduces positive data), process `b` reads from its stdin (eliminates, consuming the data). The pipe mediates: the cut eliminates the shared formula.
> 
> But Plan 9 pipes are *bidirectional*. From pipe(3): both `data` and `data1` are open `ORDWR`. This is not a simple cut — it's two cuts on two formulas, one in each direction. In linear logic terms:
> 
> ```
> a : Γ ⊢ A, Δ        b : A, Γ' ⊢ Δ'
> ─────────────────────────────────────  (cut)
>            Γ, Γ' ⊢ Δ, Δ'
> ```
> 
> For a unidirectional pipe, this is clean. For a bidirectional pipe (the Plan 9 default), you have two cuts — one for each direction. Which is what you'd expect: a bidirectional channel is a pair of unidirectional channels.
> 
> The shell's `|` is specifically a unidirectional cut (stdout of left -&gt; stdin of right). But the underlying pipe(3) device is more general.
> 
> **Classification**: Pipes are cuts. Unidirectional pipes (`|` in rc) are simple cuts. Bidirectional pipes (the kernel device) are tensor products of two cuts (`A ⊗ A^⊥`).
> 
> ### Per-Process Namespace: **The Context (Γ) — With Structural Rules**
> 
> This is where the correspondence gets rich. A per-process namespace is the set of name-to-server bindings available to the process. In sequent calculus, Γ is the set of assumptions available to a proof.
> 
> From the names paper:
> 
> &gt; "a user or process assembles a private view of the system by constructing a file name space that connects these resources."
> 
> The namespace IS Γ. Each entry in the mount table is an assumption: "the name `/mnt/wsys` is bound to this file server." The process can use any name in its namespace — it can use any assumption in its context.
> 
> Now, the structural rules.
> 
> **Weakening** (you can ignore an assumption): Yes, trivially. Having `/mnt/foo` in your namespace doesn't require you to use it. The namespace can contain entries that are never accessed. This is implicit weakening — you don't need to declare it.
> 
> **Contraction** (you can use an assumption multiple times): Yes. A file in the namespace can be opened multiple times, walked multiple times, used by any number of system calls. Names in the namespace are implicitly contractable. This is the `!` modality at work — the namespace gives you comonadic access to its contents.
> 
> **Exchange** (reordering assumptions doesn't matter): Mostly yes, but union directories break this. In a union directory with `bind -b new old`, the order matters: `new` is searched before `old`. This is a *failure of exchange* — the order of bindings is semantically significant. This is interesting because it means Plan 9's context is not a set but an ordered list (a context in intuitionistic logic, not classical).
> 
> Now for the specific operations:
> 
> **`bind -b new old`** (MBEFORE — prepend to union): This is adding an assumption to the front of the context. Not weakening — it's an extension of the context with a new assumption that shadows existing ones in the union.
> 
> **`bind -a new old`** (MAFTER — append to union): Adding an assumption to the back. The existing assumptions take priority.
> 
> **`bind new old`** (REPLACE — no flag): Substitution. Remove the old assumption, add the new one. This is contraction + weakening in one step, or more precisely, it's just replacement of a hypothesis.
> 
> **`unmount`**: Removes an assumption from the context. This is the *structural* operation of discarding an assumption. In the absence of weakening for that particular assumption, this would be significant; with implicit weakening, it's just cleanup.
> 
> **`rfork(RFNAMEG)`**: Copies the entire context. From env(3)'s analogy and namespace(6): the child gets a duplicate of the parent's mount table. Modifications in one don't affect the other. This is precisely the **copying** of a context — if Γ is the parent's context, the child gets its own copy Γ' = Γ that can be independently modified.
> 
> **Union directories and additives vs multiplicatives**: This is the subtlest point. A union directory presents multiple file servers at the same path. When you walk into a union, the kernel searches each element until it finds a match. This is *additive* (the `&amp;` / with connective): you get ONE of the available options, chosen by the system. It's not multiplicative (`⊗` / tensor) because you don't get all of them simultaneously — you get the first match.
> 
> But the `-c` flag on bind complicates this: "permit file creation in this union element." When creating a file, the kernel picks the first union element with the `-c` flag. This is a focused choice — a specific element of the additive conjunction is selected by the operation type.
> 
> **Classification**: The per-process namespace is an ordered context Γ with implicit weakening and contraction. It fails exchange in union directories. `bind` extends the context, `unmount` contracts it, `rfork(RFNAMEG)` duplicates it. Union directories are additive conjunctions with ordered focusing.
> 
> ### /srv: **A Positive Store of Negative Capabilities**
> 
> Already analyzed above. The srv device is a mutable store (positive: it contains entries) of file descriptor references (negative: each entry is a handle to a server). Creating an entry in `/srv` is introducing a positive value (the stored reference). Opening it is eliminating that value to obtain the negative capability.
> 
> This is a **shift**: srv stores shifted-down values. `create + write fd` is the shift-down (thunk a live server connection into a stored name). `open` is the shift-up (force the thunk to obtain the live connection).
> 
> **Classification**: Positive store of shifted negative values. The shift boundary is explicit.
> 
> ### /env: **Positive Values in Negative Clothing**
> 
> Already analyzed. Environment variables are stored strings (positive data) served through the file protocol (negative interface). The file protocol is the shift.
> 
> There's an interesting structural point: `rfork(RFENVG)` copies the positive values. This is value duplication, which is natural for positive types (they're data, data can be copied). The fact that you can fork the env group and get independent copies is evidence of positive polarity — negative types (computations) don't copy the same way.
> 
> **Classification**: Positive. The file interface is a shift-down wrapping.
> 
> ### Channels (Limbo/Inferno): **Cuts with Linear Discipline**
> 
> I don't have Inferno sources vendored, but I know the semantics well enough. Limbo channels are typed, synchronous (by default), and can be buffered. `chan of T` carries values of type T. A send on a channel produces a value; a recv consumes it. This is a cut: the channel connects a producer and consumer, eliminating the shared type.
> 
> Synchronous channels are the purest cut: send blocks until recv is ready, and vice versa. The two sides must meet. This is exactly cut elimination — the computation doesn't proceed until both sides of the cut are present.
> 
> Buffered channels introduce a positive store between the two sides — the buffer holds values (positive) that haven't been consumed yet. This is a shift: the immediate cut becomes a mediated cut with a positive buffer in between.
> 
> The `alt` statement (from thread(2)): "If at least one Alt structure can proceed, one of them is chosen at random to be executed." This is additive choice — select one of several available channels. It's the `&amp;` (with) elimination.
> 
> **Classification**: Channels are cuts. Synchronous channels are pure cuts. Buffered channels are cuts with a positive buffer (shift). `alt` is additive choice on cuts.
> 
> ### rc's Variables: **Positive, Backed by /env**
> 
> rc variables are lists of strings (positive data). From rc(1), they're stored in `/env` as NUL-separated byte sequences. Setting a variable writes to `/env/varname`; reading it reads the file.
> 
> Lists are the canonical positive type (introduced by enumeration, eliminated by iteration). rc's list operations — indexing, counting, concatenation — are all operations on positive data.
> 
> **Classification**: Positive. rc variables are data.
> 
> ### rc's Pipeline Model: **Cuts Composed Sequentially**
> 
> `a | b | c` is a composition of cuts. The intermediate types are byte streams (untyped positive data flowing through unidirectional cuts). rc adds nothing novel beyond the standard pipe/cut model — it just provides syntax for composing them.
> 
> The process substitution `&lt;{cmd}` from the rc paper is more interesting: it creates a pipe, runs `cmd` with stdout connected to one end, and gives the other end to the parent as a file path (`/dev/fd/N`). This reifies the cut into a name — it takes a computation (negative) and gives it a name in the namespace (positive reference to a negative channel). This is exactly a shift-down: thunk the computation into a value (the fd path).
> 
> **Classification**: Pipelines are composed cuts. Process substitution is a shift-down from computation to name.
> 
> ## 2. Is Plan 9 Primarily Co-Kleisli?
> 
> **Yes, but with important qualifications.**
> 
> The dominant pattern in Plan 9 is: resources are negative types (servers, defined by behavior), accessed through comonadic references (fids), in a context (namespace) that provides implicit weakening and contraction. The entire 9P interaction model is co-Kleisli: demand from client, supply from server, through comonadic handles.
> 
> The positive components are real but secondary:
> - **Data returned by reads** (Rread payloads, qids, stat results) — these are positive values extracted from negative computations.
> - **Environment variables** — positive values in negative wrappers.
> - **Pipe buffers** — positive intermediate storage in cuts.
> - **/srv entries** — positive references to negative capabilities.
> - **rc variables** — positive data.
> 
> The structure is: **negative infrastructure holding and mediating positive data**. The system is organized around servers (negative), not around data stores (positive). Even when Plan 9 stores data (fossils, file servers), the interface is negative — you access it through a protocol of demands.
> 
> This is not accidental. The names paper makes it architectural:
> 
> &gt; "every resource in the system, either local or remote, is represented by a hierarchical file system"
> 
> File system = negative interface. "Represented by" = the negative interface IS the resource, not a wrapper around a positive store.
> 
> **Confidence: High.** Plan 9 is primarily co-Kleisli. The design systematically presents everything through negative/demand-driven interfaces, using the 9P protocol as the universal co-Kleisli morphism structure.
> 
> ## 3. T-messages and R-messages: Not Cut, but Co-Kleisli Evaluation
> 
> Already covered above, but to be precise:
> 
> A T-message is a **co-Kleisli morphism** applied to a comonadic resource. The `tag` serves as the return continuation, the `fid` identifies the comonadic resource, and the operation (read/write/walk/stat) specifies the elimination.
> 
> An R-message is the **result** of evaluating that morphism — positive data returned to the caller.
> 
> The T/R pair is NOT a cut because the two sides are not symmetric. The client is the active party (applying an elimination), the server is the passive party (computing the result). In a cut, both sides are active computations that meet. Here, one side demands and the other supplies. That's function application, not cut elimination.
> 
> The multiplexing structure (multiple outstanding T-messages with different tags, R-messages returned out of order) is exactly the structure of co-Kleisli composition in a concurrent setting: multiple independent demands issued against the same comonadic resource, each with its own continuation.
> 
> `Tflush` is interesting: it cancels an outstanding demand. In co-Kleisli terms, this is withdrawing a morphism before it's evaluated. There's no clean sequent-calculus analogue — it's a pragmatic concurrency operation. The protocol's BUGS: you must wait for the `Rflush` even if the original R-message arrives first. This is a linearization requirement: the protocol insists on a total order of tag lifecycle events.
> 
> ## 4. Structural Rules in the Namespace
> 
> Already detailed above, but let me be more precise about what works and what doesn't in the Γ analogy.
> 
> **What works well:**
> - **Weakening**: Implicit. Unused namespace entries don't cause errors.
> - **Contraction**: Implicit via fids. Once a name is in the namespace, it can be resolved and used arbitrarily many times.
> - **Context extension**: `bind` and `mount` extend Γ with new assumptions.
> - **Context copying**: `rfork(RFNAMEG)` duplicates Γ. The copies evolve independently.
> - **Substitution**: `bind new old` (replace mode) substitutes one assumption for another.
> 
> **What's problematic:**
> - **Exchange fails in unions**: The order of `bind -b` and `bind -a` is semantically significant. Γ is an ordered context, not a set. This is a real departure from standard sequent calculus where exchange is a structural rule. You're in a non-commutative logic when union directories are involved.
> - **No linearity guarantee**: The namespace never enforces that a resource is used exactly once. Everything is implicitly contractable. There is no `!`-free zone. This is consistent with co-Kleisli (where the comonad `!` is everywhere) but means Plan 9 can't express linear resource consumption through the namespace alone.
> - **No structural rule for `unmount`**: `unmount` doesn't correspond to a clean structural rule. It's closer to an explicit context restriction — removing an assumption. In sequent calculus, you can only add assumptions (weakening), not remove them. `unmount` is an operation on the metalevel — modifying the context itself, not using it.
> 
> **Union directories as additive conjunction:**
> 
> A union directory `{A, B, C}` presents three servers at one name. A walk into the union tries A first, then B, then C, returning the first success. This is the with-connective `A &amp; B &amp; C` — the client doesn't choose which one; the *content of the demand* (the filename being walked) determines which element responds.
> 
> Actually, this is more subtle. In the additive `&amp;`, the *consumer* chooses which projection to use (π₁ or π₂). In a union directory, the *name being walked* determines which element responds. If the file exists in A, you get A's version. If it only exists in B, you get B's. This is closer to pattern matching with fallthrough — an ordered disjunction with implicit prioritization.
> 
> The `-c` flag adds another dimension: create operations go to the first union element marked with `-c`. This is explicit focusing — the type of operation (create vs walk) selects different elements of the union.
> 
> I'd classify union directories as **ordered additive conjunctions with operation-dependent focusing**. Not standard additive connectives, but recognizably in the additive family.
> 
> ## 5. Where Plan 9 Crosses Between Subcategories
> 
> These are the interesting cases where something changes polarity:
> 
> ### Cache (-C flag): Negative to Positive
> 
> Already discussed. The cache flag explicitly forces the server's negative computation into a positive cached value. Open re-validates. This is a controlled shift with explicit freshness semantics.
> 
> ### /env: Positive as Negative
> 
> Environment variables are positive values (stored strings) presented through the negative file interface. Every read returns the stored value; there's no computation. The file protocol is a shift-down that's effectively a no-op — the "elimination" just returns the stored introduction.
> 
> But fork semantics reveal the positive nature: `rfork(RFENVG)` copies the values. A truly negative resource (a running server) doesn't copy this way — you'd need to fork the server process too.
> 
> ### /srv: Negative as Positive (Reification)
> 
> When you post a file descriptor to `/srv`, you take a live negative resource (a running server's channel) and store it as a named positive entry. The entry persists even if no process has it open. This is reification — freezing a live computation into a stored reference.
> 
> The known failure mode is exactly what you'd predict from the polarity analysis: if the server process dies but the `/srv` entry persists, the positive reference points to a dead negative computation. The entry is stale. srv(3) has no mechanism to detect this. The positive store can't tell whether the negative computation behind it is still live.
> 
> This is a general principle: **shifts from negative to positive lose liveness information**. A stored reference to a computation doesn't know whether the computation is still running. Plan 9 never solved this cleanly — the remedy was manual cleanup or reboot (which cleared `/srv`).
> 
> ### Blocking reads: Negative demanding synchronization
> 
> `/proc/N/wait` blocks until a child exits. The read *is* the synchronization. This is a negative type that refuses to produce a positive result until a condition is met. There's no pre-existing value; there's a computation that depends on external events.
> 
> This is actually the cleanest negative type in the system: the read is an elimination that may not terminate (it blocks), its result depends on when (or whether) the condition is satisfied, and the result is a positive value (the wait record text) produced by the negative computation.
> 
> ### Write as positive-into-negative
> 
> `Twrite(fid, offset, count, data)` carries positive data (the bytes to write) and applies it to a negative resource (the file server). The write operation is a *positive value being consumed by a negative computation*. In duploid terms, this is a morphism from the Kleisli side into the co-Kleisli side: positive data crosses the polarity boundary.
> 
> This is significant because it means 9P is not purely co-Kleisli. The `Twrite` message carries a Kleisli component (the data payload is positive, introduced by the client). The protocol mediates between the two subcategories: `Tread` is purely co-Kleisli (demand from client, supply from server), but `Twrite` is a mixed morphism (positive data from client applied to negative server).
> 
> ## 6. Plan 9 as Co-Kleisli vs Be as Kleisli
> 
> The claim that Plan 9 is primarily co-Kleisli and Be is primarily Kleisli is **substantially correct** and illuminating.
> 
> **BeOS/Haiku's Kleisli character:**
> 
> From the BeOS internals in the project memory:
> 
> - **BMessage is positive data.** It's a self-describing typed data structure with named fields. You construct it (introduction), send it, and the receiver pattern-matches on the `what` code and extracts fields (elimination of a positive type). The message exists as a value independent of any consumer.
> 
> - **BRoster is a positive registry.** Applications register and are stored. You query the roster and get back data (application info structs). The roster is a data store, not a computation server.
> 
> - **BClipboard is push-based (positive).** Data is uploaded at copy time and persists even if the source exits. This is in explicit contrast to X11's pull-based (negative) model. The clipboard holds positive data, not a reference to a computation.
> 
> - **The wire protocol is positive framing.** LinkSender batches multiple messages and flushes them — positive data accumulated and transmitted. The app_server receives positive data packets and processes them.
> 
> - **Loopers are event consumers.** The BLooper message loop receives positive BMessages and dispatches them. The looper doesn't demand — it receives. It's a consumer of positive data, which is the Kleisli pattern: values flow toward consumers.
> 
> **The complementarity:**
> 
> Plan 9: Resources are negative servers. You demand from them. The infrastructure mediates demands. Data flows from server to client in response to client demands. **Pull-based. Demand-driven. Co-Kleisli.**
> 
> BeOS: Resources are positive messages. You receive them. The infrastructure delivers messages. Data flows from sender to receiver as a pushed value. **Push-based. Supply-driven. Kleisli.**
> 
> This is a genuine complementarity, and it corresponds to the duploid structure: the two subcategories (Kleisli and co-Kleisli) coexist within the same framework, connected by shifts.
> 
> **Where the complementarity matters for pane:**
> 
> Pane inherits Be's push-based messaging (BMessage -&gt; Message trait, BLooper -&gt; calloop event loop) but also adopts Plan 9's demand-driven file interface (pane-fs synthetic filesystem). This means pane naturally lives at the **shift boundary** between the two subcategories:
> 
> - The **protocol layer** (pane-session, message dispatch) is Kleisli: positive messages flowing between loopers.
> - The **filesystem layer** (pane-fs, FUSE) is co-Kleisli: demand-driven reads against synthetic files.
> - The **shift** between them is the bridge: when a FUSE read arrives (co-Kleisli demand), pane-fs translates it into a query against the pane store and returns positive data; when a protocol message arrives (Kleisli value), the handler processes it and may update state that the filesystem will later serve.
>

> Quick orientation for the analysis below. In a duploid (Mangel-Mellies-Munch-Maccagnoni):
> 
> - **Positive types** are characterized by their *introduction* (constructors). You know them by what they *contain*. They're values: data-at-rest. > ## 2. The Pattern
> 
> The split is **not** simply data-positive / computation-negative. It's more precise:
> 
> **Things that cross boundaries are positive.** BMessage, BMessenger, BPropertyInfo, entry_ref, node_ref — anything that gets serialized into a port write, embedded in another message, or stored in the filesystem. Positive types are the *lingua franca* because positivity gives you serializability, copyability, and referential transparency.
> 
> **Things that sit at boundaries are negative.** BHandler, BLooper, BWindow, BView, BQuery (live), node monitor watchpoints — anything that *waits for and responds to* positive values. Negative types are the *endpoints* because negativity gives you reactive behavior, encapsulation, and the ability to be substituted (any handler that responds to the right messages is interchangeable).
> 
> **The connective tissue between them is shifts.** BMessenger is ↑(handler) — a positive token for a negative endpoint. reply_port is ↑(continuation) — a positive triple of scalars for a negative one-shot demand. `be_app` and `be_roster` are ↑ shifts at the language level (global pointers to negative objects). `SendMessage` is ↓ — forcing the positive reference to make a demand on the negative endpoint.
> 
> The axis is **boundary-crossing**. Things that stay put are negative. Things that travel are positive. Shifts are what let negative things be referenced across boundaries without losing their identity.

> ## 3. Where Be's Design Crosses Between Subcategories

> ## 4. pane's Separation of Message from ReplyPort
> 
> The prior analysis was correct, and the duploid framing makes it precise.
> 
> BeOS BMessage was a *compound type*: `positive_data × ↑(negative_continuation)`. But it was not decomposed — the reply_port fields were embedded in the same struct as the message data. Consequences:
> 
> - BMessage was **not safely copyable** despite appearing to be, because the copy would share the same reply_port (the continuation must be used *once*, but now two objects reference it)
> - BMessage's `operator=` copied the reply fields, creating aliased continuations
> - REPLY_DONE was mutable state that made BMessage thread-unsafe around the reply path
> - The positive half and the negative half had different lifetime requirements but were bound to the same object lifetime
> 
> pane's design decomposes this:
> 
> - `Message` trait (`Clone + Serialize + DeserializeOwned + Send + 'static`) — purely positive. No continuation, no reply state. Freely copyable, serializable, projectable. Lives in the Kleisli subcategory without compromise.

> - `ReplyPort&lt;T&gt;` (`obligation.rs`) — purely ↑(negative). It's a *shifted continuation*: `Option&lt;mpsc::Sender&lt;Result&lt;T, ReplyFailed&gt;&gt;&gt;`. The `Option::take()` pattern in `.reply()` and `Drop` is a manual encoding of linear consumption.
> 
> The separation means:
> - The positive component can travel freely (the Kleisli subcategory operates on Messages without restriction)
> - The negative component is consumed exactly once (linearity, enforced at compile time via move semantics + `#[must_use]`)
> - The two have independent lifetimes: a message can outlive its associated reply obligation, or vice versa, without corrupting either
> 
> In duploid terms: BeOS tangled `A × ↑(B → T(C))` into a single object. pane separates it into `A` (passed to `receive`) and `↑(B → T(C))` (passed separately to the obligation callback). The shift is now explicit in the type system rather than implicit in the runtime protocol.
> 
> The `#[must_use]` + `Drop` compensation pattern is pane's approximation of a linear type discipline. Rust doesn't have true linear types, but `#[must_use]` catches the common case (forgotten obligations), and `Drop` handles the panic/unwind path (the obligation is discharged as failure rather than lost). This is strictly better than BeOS's runtime flags: `MESSAGE_FLAG_REPLY_REQUIRED` was advisory (forgetting to check was a bug), while `#[must_use]` is a compiler warning and `Drop` is language-guaranteed.

> ## 1. RC's Execution Model vs. psh's `exec.rs`
>

> 
> rc's execution model, as described in the paper and implemented via yacc-generated parse trees, is straightforward:
>

> 2. **Walk the tree.** The tree nodes correspond directly to grammar productions. There is no separate "statement" vs "expression" distinction in rc — everything is a command.
> 3. **For simple commands:** check if it's a function name, then check builtins, then search `$path` for an external binary. Fork, exec, wait.

> 
> &gt; *Rc does one of several things to execute a simple command. If the command name is the name of a function defined using `fn`, the function is executed. > ### The AST IS the profunctor
> 
> A traditional shell AST bolts redirections onto commands as a flat list:

> cmd.redirects = [stdout→file, dup 2→1]

> Redirect(

>         Output { fd: 1, target: File("file") }

> ```

> 
> A profunctor P is contravariant in its first argument and covariant in its second. For a command viewed as a box taking input (stdin, fd 0) and producing output (stdout, fd 1):
> 
> - `Input` (lmap): pre-composes on the input side. `&lt; file` replaces the input source before the command sees it. Contravariant — the transformation points *toward* the command.

> - `Dup` (contraction): merges two continuation channels. After `&gt;[2=1]`, fds 1 and 2 alias one resource — two consumers contracted to one.

The evaluator cannot apply redirections in the wrong order because the tree has only one traversal order. > 
> The profunctor laws hold concretely. `dimap id id = id`: a `Redirect(inner, &lt;identity-op&gt;)` — were one expressible — would evaluate `inner` unchanged. > ### The ⊕/⅋ error convention
>

> negation that swaps CBV and CBN. `set -e` (errexit) is the explicit conversion
> from ⊕ to ⅋: it takes status-based error reporting and automatically invokes
> the ERR trap continuation.
> 
> psh uses ⊕ exclusively. Every operation returns `Status`. There are no traps,
> no continuation-based error paths, no `set -e`. This is the right default for
> a system shell. The ⅋ convention's power comes with a cost: the callee drives
> control flow, which means the caller cannot reason locally about what happens

> fails should say where and why, not unwind through an invisible trap chain.
> 
> The cost is verbosity. Without ⅋, every fallible operation needs explicit

> inherently callee-driven. The ⊕/⅋ coexistence in ksh93 was not a design

> ## Formal structure
> 
> ### The duploid, made explicit
> 
> ksh93's interpreter has duploid structure that SPEC.md identifies but that the C code never names. psh names it. The three AST sorts — `Word`, `Expr`, `Statement` — are not an engineering convenience; they are the three syntactic categories of the λμμ̃-calculus [Curien-Herbelin 2000] instantiated as a shell grammar.
> 
> | psh sort | λμμ̃ analog | Evaluation | Examples |
> |---|---|---|---|
> | `Word` | Term (producer) | CBV — evaluated eagerly before the consumer runs | `Literal`, `Var`, `CommandSub`, `Concat` |
> | `Expr` | Coterm (consumer) | CBN — pipeline stages run concurrently, demand-driven | `Pipeline`, `Redirect`, `Background`, `Coprocess` |
> | `Statement` | Cut ⟨t \| e⟩ | Connects a producer to a consumer | `Assignment`, `Exec`, `If`, `For` |
> 
> In ksh93 this structure is implicit — the `Shnode_t` union carries it via `tretyp &amp; COMMSK` tags, but nothing in the type system enforces the sort boundaries. psh makes it structural: `Word` and `Expr` are different Rust types. Cross-sort composition requires an explicit embedding (`Statement::Exec` wraps an `Expr`; `Word::CommandSub` wraps `Vec&lt;Statement&gt;`). You cannot accidentally pass a pipeline where a word is expected.
> 
> The CBV/CBN split follows the duploid's two subcategories [Mangel-Melliès-Munch-Maccagnoni 2026, §2.1]. Word expansion is Kleisli composition: each stage (`$x` lookup, concatenation, command substitution) takes a partial value and produces an expanded value with possible side effects. `eval_word` in exec.rs recurses through `Word` nodes before the command that consumes them has started. Pipeline execution is co-Kleisli: `run_pipeline` forks all stages concurrently, and data flows on demand through `pipe(2)` endpoints. `yes | head -1` does not evaluate `yes` to completion. The pipe's blocking read is the demand.
> 
> Cross-polarity composition — a pipeline stage that expands a variable (CBV) and writes to a pipe (CBN) — is non-associative in the duploid sense. Of four associativity equations, the (+,−) equation fails [Munch-Maccagnoni 2014, Table 1]: `(h ○ g) • f ≠ h ○ (g • f)` when • composes through a positive intermediary and ○ through a negative one. psh's sequential evaluation within each process serializes the crossing. Word expansion completes before `execvp` runs; the fork boundary separates the two polarities. There is exactly one bracketing per process, so the non-associativity never produces distinct reduction orders. This is the same resolution that ksh93 achieves by accident and that SPEC.md documents as the single-process serialization constraint. psh achieves it by construction: the type system requires `Statement::Exec(Expr)` to mediate, and `run_expr` is the only function that evaluates an `Expr`.
> 
> ### The shift operators
> 
> Three concrete constructs in psh are polarity shifts — operations that cross the value/computation boundary.
> 
> **Command substitution** `` `{ cmd } `` is ↓→↑: force a computation into value position. In exec.rs, `eval_word` handles `Word::CommandSub` by forking a child, piping its stdout back, running the statements in the child (↓ — enter computation), collecting the output, and returning a `Val` (↑ — return to value). This is the same structural operation as ksh93's `comsubst()` in macro.c, but in ksh93 the save/restore of `Mac_t` state is manual. In psh, the fork boundary makes the shift total: the child's entire environment is discarded on exit. No state leaks back except the pipe's bytes.
> 
> **Assignment** `x = value` is μ̃-binding. `Statement::Assignment` evaluates the right-hand side (CBV — `eval_value` forces the `Value` to a `Val`), then stores the result in the environment. This is let-binding: the value is fully computed before the name is bound. The evaluation context continues with the binding in scope. In λμμ̃ terms, `x = val; rest` is ⟨val | μ̃x.rest⟩. The discipline function hook fires at the binding site — the `.set` discipline is computation injected at a value boundary — but the reentrancy guard (`active_disciplines`) prevents recursive re-entry, serializing the crossing.
> 
> **Discipline functions** are polarity boundary crossings. A `.get` discipline fires when `$x` is expanded — computation (the discipline body) intrudes into value context (word expansion). A `.set` discipline fires on assignment — computation intrudes at a μ̃-binding site. These are the exact sites where ksh93's `sh.prefix` bugs occur (SPEC.md §"The critical pair"). psh avoids the critical pair structurally: there is no global `sh.prefix` to corrupt. Each assignment is a local `eval_value` → `set_value` sequence; the discipline body runs in its own scope (`push_scope`/`pop_scope` in `.get`, implicit in `.set`). The reentrancy guard is the runtime residue — the one place where the type system cannot prevent the crossing, and a dynamic check is required.
> 
> ### par and the duploid: distinct layers
> 
> The duploid structure described above governs evaluation order *within the interpreter* — how words become values, how pipelines compose, where the polarity boundary falls. `par` (pane-session's session-typed channel library) governs *communication protocols* — the typed exchange sequences between psh and a pane server, between coprocess endpoints, or between job handles and the shell.
> 
> These are different concerns at different layers.
> 
> The duploid is the metatheory of psh's evaluation model. It classifies `Word`, `Expr`, `Statement` and determines that `eval_word` is CBV and `run_pipeline` is CBN. It has nothing to say about what messages flow over a connection to a pane server.
> 
> `par` is the protocol discipline for those connections. When `get /pane/editor/attrs/cursor` runs, the interpreter evaluates the path argument (CBV — a `Word::Literal`), dispatches to `builtin_get` (the cut), and then — inside the builtin — opens a pane-session connection and executes a par session: `Send&lt;AttrQuery, Recv&lt;AttrValue&gt;&gt;`. The session type constrains the wire exchange. The duploid constrains how we got there.
> 
> For coprocesses (`cmd |&amp;`), `par` will type the bidirectional pipe endpoints. The shell writes to the coprocess (positive, push — `Send`) and reads from it (negative, demand — `Recv`). The coprocess's `par` session type describes the expected request/response pattern. This is not the same as the pipeline's co-Kleisli composition: a pipeline stage's stdout is an untyped byte stream with no protocol structure. A coprocess channel can carry a typed exchange. The `par` layer adds protocol fidelity that the duploid layer does not provide.
> 
> The boundary: the duploid determines *when* a `par` session begins and ends (it's always inside a cut — a `Statement::Exec` running a builtin or coprocess). `par` determines *what happens* during the session. They compose without interference because the duploid layer is complete before the `par` layer activates: word expansion finishes, the builtin dispatches, and only then does the session-typed exchange begin.
> 
> ### What ksh93's continuation stack teaches psh
> 
> SPEC.md §"Continuations and classical control" documents ksh93's `sigjmp_buf`/`struct checkpt` mechanism — a linked stack of continuation frames implementing the ⅋ error convention. Traps, `break`, `continue`, `return`, and error propagation all work by `siglongjmp` to a captured `checkpt`. The 13 jump modes (`SH_JMPNONE` through `SH_JMPSCRIPT`) form a severity hierarchy, with `SH_JMP_PROPAGATE` as the boundary between local recovery (⊕) and stack-unwinding propagation (⅋).
> 
> The continuation stack is the source of SPEC.md's category-4 boundary violation: continuation misfires where `siglongjmp` unwinds to the wrong `checkpt` because the push/pop discipline was violated. Bugs 003a and 003b are category-2 (stale context), but the mechanism is the same: saved state restored over mutations that should have persisted.
> 
> psh uses ⊕ only. `Status` is a string (rc convention). Every operation returns a `Status`. There is no `siglongjmp`, no `checkpt` stack, no continuation capture. Error propagation is the caller's responsibility: `run_stmt` returns `Status`, and compound statements (`If`, `For`, `Switch`) inspect it.
> 
> This eliminates continuation-stack corruption by absence. The formal argument: ksh93's ⅋ convention introduces classical control — the ability to jump to a captured continuation, which is the μ-binder in the λμμ̃ calculus. Classical control makes the calculus non-confluent (Wadler [2005, §2]: the critical pair arises from μ/μ̃ interaction). The continuation stack is the runtime representation of the μ-variable context (Θ in the sequent Γ ⊢ Θ). Every `siglongjmp` is a goto to a named continuation. When the continuation context is corrupted (wrong frame, stale frame), the jump lands in a state the program never constructed.
> 
> psh's ⊕-only discipline restricts to intuitionistic control: no μ-binders, no continuation capture, no non-local jumps. The sequent degenerates to Γ ⊢ A (single conclusion). This is Levy's CBPV without control operators [Levy 2004, §2.2] — every computation returns exactly once to its immediate caller. The resulting system is confluent by construction: there is no critical pair because the μ-binder doesn't exist.
> 
> The cost is expressiveness. ksh93's `trap 'handler' ERR` automatically fires on any error — the callee drives control, and the caller doesn't need to check. psh requires the caller to check `$status` or use `&amp;&amp;`/`||`. This is the ⊕/⅋ tradeoff, not a bug. Discipline functions provide a narrow form of callback (the `.set`/`.get` hooks), but they are synchronous, non-escaping, and reentrancy-guarded. They do not capture continuations.
> 
> ### Thunkability and centrality
> 
> The Filinski-Hasegawa criterion [Mangel-Melliès-Munch-Maccagnoni 2026, §3.2]: in a dialogue duploid (one with involutive negation, i.e., classical control), thunkable maps are exactly the central maps — purity and commutativity coincide. psh is not a dialogue duploid (it has no involutive negation — no μ-binder, ⊕ only), so the full theorem does not apply. But the underlying question — which operations are thunkable? — applies to any duploid.
> 
> A map is thunkable when it can be packaged as a value without changing the program's behavior [Munch-Maccagnoni 2014, Definition 2.3]. Concretely: an operation is thunkable if wrapping it in `` `{ ... } `` (command substitution — the thunk/force shift) is semantically equivalent to running it inline, modulo the fork boundary's effect isolation.
> 
> **Thunkable operations in psh:**
> - Pure word expansion: `$x`, `$#x`, literal concatenation. Packaging `echo $x` as `` `{ echo $x } `` produces the same string. These are value-level operations that don't mutate the environment.
> - External commands with no shell-visible side effects: `date`, `cat file`, `grep pattern`. Their output is the same whether run inline or in a command substitution.
> 
> **Non-thunkable operations:**
> - Assignment: `x = val` mutates the environment. `` `{ x = val } `` runs in a forked child; the assignment is lost. The fork boundary — psh's shift implementation — isolates the side effect, making the thunked version semantically different.
> - Discipline-triggering reads: `$x` where `x.get` mutates state. The `.get` body is a side effect; thunking it isolates the mutation.
> - `cd`, `set`, any builtin that modifies the shell's own state.
> 
> The fork boundary in command substitution acts as a purity test. Operations that survive the fork unchanged are thunkable; operations whose effects are lost in the fork are not. This is not an accident — it is the operational content of thunkability. The shell's process model provides what a type system would otherwise need to enforce: an isolation boundary that separates pure values from effectful computations.
> 
> In a dialogue duploid, centrality (commutativity with all other maps) would additionally follow from thunkability. psh doesn't have classical control, so centrality is a weaker property here — it means only that the operation's position in a sequence doesn't affect the result. The thunkable operations listed above are also central in this sense: reordering pure expansions doesn't change behavior. Non-thunkable operations are not central: `x = 1; echo $x` differs from `echo $x; x = 1`.
