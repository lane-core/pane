> **Q2 — Dis VM namespace isolation:** Each process group (`Pgrp`) owned a mount hash table. Fork was full copy, not COW — copying 10-30 mount entries (~1-2KB of pointer metadata). Underlying channels were refcounted and shared. Cheap in practice because mount tables are small. No COW was deliberate: the semantics of shared-then-modified mount tables are subtle, and the memory savings are negligible.
> 
> **Q3 — Styx as enabler:** Because everything in Inferno spoke Styx (in-kernel devices, local Limbo servers, remote machines), the mount table was just a routing table — "walk requests into `/net` go down this channel." The namespace layer was a multiplexer, not an interpreter. This works when the things being named all speak a uniform protocol. pane's session-typed protocol is the equivalent, but it's not file-oriented, so pane's namespace is a *projection layer* over server state rather than a *routing table* to protocol endpoints.
> 
> **Q4 — pane's adaptation:** Per-connection `ConnectionNamespace` in pane-server — a visibility predicate over pane attributes, configured by `.plan` for remote connections, defaulting to "all" for local. Non-visible panes return ENOENT, not EACCES (Inferno semantics: absent, not forbidden). FUSE layer uses uid-based filtering from `fuse_req_ctx` — coarser but sufficient for the scripting tier. Don't replicate bind/mount operations; pane's computed view directories are the equivalent mechanism. The key insight: pane-server is the runtime that mediates name resolution, playing the same role emu played for Inferno.

> 
> - **`msg: P::Message`** -- Protocol-scoped, not `impl Serialize`. This is the key change from the architecture spec's `impl Serialize + Send + 'static`. Correct per decision #3: compile-time protocol agreement within a compilation unit.

> ## 1. Duploids and the two-phase protocol
> 
> The handshake phase (Send/Recv, linear, finite) lives in the positive subcategory P of a duploid. It is the Kleisli category of the monad T = R ∘ L: composition is associative within it, effects (network IO) are sequenced left-to-right. The active phase (runtime-dispatched, infinite message stream) lives in the negative subcategory N -- the co-Kleisli category of the comonad K = L ∘ R: composition is associative within it, but evaluation is demand-driven (the looper waits for messages).
> 
> The transition from handshake to active phase is a *transverse map* in the collage category E = E(L,R) -- a map from a positive object to a negative object. By the duploid construction (Mangel et al. 2025, Section 1.2, equation (3)), composition across a polarity boundary is defined by *factoring through the adjunction*: either via g^{rhd} (when the intermediate is positive) or f^{lhd} (when the intermediate is negative). The two recipes are not interchangeable. This is why the protocol transition is structurally a one-way gate. pane already enforces this: the bridge consumes the typed channel and returns an untyped connection. The duploid says this consumption is *the shift operator* omega_X : X -&gt; Downarrow X (Definition in Section 2), which is thunkable-epi but *not an isomorphism* in the non-associative sense. You cannot go back.
>

The fact that the handshake channel is consumed and cannot be recovered is the non-invertibility of the positive shift in the duploid. > ## 2. The server as a cut
> 
> The server reads from connection A and writes to connection B. In a dialogue duploid, this is composition of two transverse maps: f : A_0 -&gt; Y_1 and g : Y_0 -&gt; B_1 through an intermediate object Y. The composition g ∘ f depends on Y's polarity. >

> exactly what the optics design brief already says, now grounded in the duploid structure.

> Q4: Should the shift operator be a type?
> 
> Yes, ActivePhase is the right encoding, and it should carry negotiated state.
> 
> The shift ω_X : Handshake → ActivePhase is one-way (thunkable-epi, as your session-type consultant identified).
> It should be a newtype, not just a marker, because it carries state that the handshake produced:
> 
> pub struct ActivePhase<T> {
> inner: T,
> max_message_size: u32, // from Welcome
> peer_auth: PeerAuth, // from transport verification
> known_services: ServiceSet, // from Welcome.services

> 
> 1. The shift carries information. ω_X is not just a type coercion — the handshake negotiated parameters that
>    constrain the active phase. max_message_size limits what can be sent. known_services determines which service
>    IDs are valid. These are the "positive residue" of the dialogue — data extracted from the finite dialogue that
>    constrains the infinite active phase.
> 2. The shift is not invertible. You cannot go from ActivePhase back to a handshake channel. The par channel was
>    consumed. ActivePhase<T> should not implement any method that yields a handshake-typed value.
> 3. The shift prevents phase confusion. Without ActivePhase, nothing stops code from calling
>    bridge_client_handshake() on a connection that's already in the active phase. The newtype makes this a compile
>    error.
> 
> The inner: T should be the reader half of the connection (mpsc::Receiver or equivalent). The writer half is
> held separately by the bridge thread. This separation is itself a polarity decomposition: the reader (negative,
> demand-driven) lives in the handler; the writer (positive, push-based) lives in the bridge.

> Q5: Filinski-Hasegawa for batch optimization
> 
> (b) The criterion applies locally within sub-interactions that have dialogue structure, plus you can use a
> weaker criterion for the non-dialogue active phase.
> 
> The Filinski-Hasegawa theorem (MMM25, Theorems 11.1 and 11.2) states: in a dialogue duploid, central =
> thunkable. This requires dialogue structure (involutive negation). The active phase lacks this.
> 
> But you don't need the full FH theorem. You need the weaker result that holds in any duploid (not just dialogue
> duploids):
> 
> Thunkable ⟹ central holds in every duploid (MM14b, Proposition 6). Thunkability means: f composed with any
> Kleisli composition behaves associatively. Centrality means: f commutes with all effectful compositions.
> Thunkable implies central always. The converse (central ⟹ thunkable) requires dialogue structure.
> 
> For batch optimization, you want: "which handlers can I safely reorder?" The answer:
> 
> - Thunkable handlers can always be reordered. A pure handler (no effects, output depends only on input) is
>   thunkable. By Proposition 6, it's central — it commutes with everything. AttrReader::view-based operations are
>   thunkable (B5). You can freely reorder, batch, or parallelize them.
> - Non-thunkable handlers might still be reorderable with respect to specific operations. Two effectful handlers
>   might commute with each other even though neither is thunkable. But you'd need to prove this case-by-case. The
>   duploid doesn't give you a general criterion for this.
> - The conservative rule is sufficient: reorder only thunkable operations. This is the one-directional
>   implication from Proposition 6, which holds without dialogue structure.
> 
> So: use thunkability as your criterion. It's sound (never produces incorrect reorderings) and complete enough
> for practical purposes (the operations you most want to reorder — reads, pure transformations — are exactly the
> thunkable ones).
>

> Q6: Does the duploid framework suggest changes to ServiceFrame?
> 
> ServiceFrame's current structure is correct. Organizing by polarity would clarify the dispatch logic but is not
> structurally required.
> 
> The polarity classification:
> 
> ServiceFrame::Request — positive (a value being offered to the receiver)
> ServiceFrame::Reply — positive (a value satisfying a prior demand)
> ServiceFrame::Failed — positive (a value reporting failure)
> ServiceFrame::Notification — positive (a fire-and-forget value)
> 
> All four variants are positive. This is not a bug — it's correct. ServiceFrame is a wire type. Everything on
> the wire is positive (serialized, transmitted, deserialized). The negative types (handlers, obligation
> endpoints) live at the endpoints, not in the frame.
> 
> The cross-polarity interaction happens at dispatch, not at framing:
> 
> 1. A positive ServiceFrame::Request arrives at the server.
> 2. The server dispatches it to a negative handler (Handles::receive).
> 3. The handler produces effects and (optionally) a positive Reply.
> 4. The Reply is framed and sent back.
> 
> Steps 1→2 and 3→4 are the polarity crossings. The duploid's composition rules apply at the dispatch boundary,
> not at the frame type.
> 
> What WOULD prevent future deadlocks: Not restructuring ServiceFrame, but ensuring that dispatch never holds
> cross-polarity locks. The deadlock occurred because reader threads (negative) held routing state while
> performing writes (positive). The fix is architectural (actor model), not type-structural (frame variants).
> ServiceFrame's variants don't need polarity annotations because the polarity discipline is enforced at the
> dispatch layer (LooperCore + I6), not at the framing layer.
> 
> That said, a comment in the ServiceFrame type documenting that all variants are positive wire values, and that
> polarity crossings happen at dispatch, would prevent future confusion.
>

> Q7: Plan 9 in this framework
> 
> 9P is a collage category, which is a degenerate duploid where all cross-polarity morphisms factor through the
> transverse maps.
> 
> Let me be precise.
> 
> 9P's T/R structure. A T-message is a co-Kleisli morphism: the client applies a demand to a comonadic server
> resource (identified by fid). An R-message is the positive result. The T/R pair is not a cut — it's the
> evaluation of a co-Kleisli map (demand → supply). This is MM14b's collage category construction (Section 3.2,
> before Definition 1): given categories C₁ (server-side, negative) and C₂ (client-side, positive), the collage
> has objects from both and morphisms that include the transverse maps between them.
> 
> Per-fid serialization is the Plan 9 analog of the actor model: each fid's operations are serialized, so the
> non-associative (positive, negative, positive) bracket never arises on a single fid. Multiple fids can be in
> flight concurrently, but their operations are independent (no shared intermediate object). The multiplexing tag
> system provides request/response correlation without cross-fid locking.
> 
> Namespace composition. bind and mount are monoidal operations on the namespace context Γ. The monoidal
> structure is the ordered product (because union directories are ordered — exchange fails). This is a
> non-commutative monoid, which places it outside standard linear logic (which assumes exchange) and into
> non-commutative linear logic (Lambek calculus and its descendants). The duploid framework doesn't directly
> model non-commutativity — MM14b's duploids assume exchange. Extending the framework to non-commutative settings
> would require Lambek duploids or ordered duploids, which are not (to my knowledge) in the literature.
> 
> The filesystem-as-API vs typed-protocol distinction. This is the deepest architectural question. Here is the
> categorical reading:
> 
> - Plan 9 (filesystem-as-API): All operations are mediated by a single protocol (9P) with a fixed set of
>   operations (walk, open, read, write, stat, clunk, ...). The protocol is uniform — every resource exposes the
>   same interface. Type information is in the data, not the protocol. This is a single co-Kleisli category with a
>   universal comonad (the filesystem interface).
> - pane (typed protocol): Operations are protocol-specific (Handles has a different receive signature for each
>   P). The protocol is indexed — each service has its own message types. Type information is in the protocol, not
>   (only) the data. This is an indexed family of Kleisli categories, one per protocol, unified by the dispatch
>   mechanism.
> 
> Can pane-fs and the protocol layer share algebraic structure? Partially. The MonadicLens is the bridge:
> 
> - Protocol-side: handler receives typed messages (positive), processes them, mutates state.
> - Filesystem-side: FUSE reads the same state through AttrReader (negative-to-positive shift).
> 
> Both sides observe the same state through different interfaces. The algebraic structure they share is the
> duploid itself — the state S is the common negative object, the protocol operates on it via Kleisli maps (set),
> the filesystem observes it via co-Kleisli maps (view). They share the object but use different subcategories.
> 
> What they cannot share: The protocol's typed dispatch (Handles with protocol-specific Message types) and the
> filesystem's untyped dispatch (ctl command strings parsed by AttrWriter) are fundamentally different. The
> protocol is a family of Kleisli categories indexed by ServiceId. The filesystem is a single co-Kleisli category
> with text as the universal type. These are related by the MonadicLens's parse function (fn(&str) -> Result<A,
> String>), which is the adapter between the untyped filesystem world and the typed protocol world. This adapter
> is not an algebraic morphism — it's a partial function that can fail (parse errors). The two layers cannot be
> unified algebraically without losing the filesystem's universality or the protocol's type safety.
> 
> Recommendation: Keep them separate. The MonadicLens is the bridge, and its two components (view for filesystem,
> set for protocol) live in different subcategories by design. The duploid framework confirms that this
> separation is structurally correct: they are the two faces of a cross-polarity morphism, not two versions of
> the same morphism.
