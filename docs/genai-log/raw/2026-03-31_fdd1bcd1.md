> ## Phase 2 Timer Migration: Session-Type Analysis
> 
> ### Preliminary: calloop Timer API shape
> 
> calloop 0.14's `Timer` is a one-shot deadline source. Periodic behavior is achieved by returning `TimeoutAction::ToDuration(d)` from the callback, which reschedules in-place. Cancellation uses `LoopHandle::remove(RegistrationToken)`. The critical constraints:
> 
> - `LoopHandle<'l, Data>` contains `Rc<LoopInner>` → **`!Send`, `!Sync`**
> - `RegistrationToken` is `Clone + Copy + Debug + PartialEq + Eq` — a bare index into a slab
> - Timer callbacks receive `&mut Data` (the shared `LooperState`)
> 
> ### Question 1: TimerToken linearity and calloop's RegistrationToken
> 
> **Verdict: calloop's RegistrationToken is strictly worse for linearity than the current Arc<AtomicBool> design. Do not expose it.**
> 
> The current `TimerToken`:
> ```rust
> #[derive(Debug, Clone)]
> pub struct TimerToken {
>     id: u64,
>     cancelled: Arc<AtomicBool>,
> }
> ```
> 
> This is `Clone + Send + Sync`. The clone is an affine gap — multiple handles can observe/set the cancellation flag, which means the protocol `create → (fire)* → cancel` has no enforcement of single-cancellation. But `AtomicBool::store(true)` is idempotent, so multi-cancel is harmless. The real gap is: **drop without cancel leaves the timer running**. This is the standard affine-vs-linear problem — Rust can't enforce "you must call cancel."
> 
> calloop's `RegistrationToken` is `Copy` (a raw slab index). It's even worse:
> 1. **No cancellation semantics at all** — calling `handle.remove(stale_token)` after the slab slot is reused removes the *wrong source*. This is a use-after-free analogue at the event-source level.
> 2. **`LoopHandle` is `!Send`** — the token can only be used on the looper thread. Worker threads can't cancel timers directly.
> 3. **Copy, not affine** — no ownership tracking whatsoever.
> 
> The session-type reading of timer lifecycle is a recursive protocol:
> 
> ```
> TimerSession = μX. (Tick.X ⊕ Cancel.End)
> ```
> 
> This is an external choice (the timer holder decides to continue or cancel). In TLL+C terms (Fu/Xi/Das, §3.2), this is a positive type — the provider (looper) offers repeated service, the client (timer holder) chooses when to stop. The `Arc<AtomicBool>` encodes the `Cancel` branch as a side-effect (store `true`), which collapses the choice into an unconditional continuation with early termination. This loses the typestate progression but preserves safety because:
> 
> - Cancellation is idempotent (AtomicBool can be set to true multiple times)
> - The looper checks the flag on each iteration, so cancelled timers never generate spurious events after one iteration delay
> - Drop-without-cancel means the timer runs forever, which is a liveness issue, not a safety issue
> 
> **Recommendation:** Keep `TimerToken` with `Arc<AtomicBool>` as the public API. Internally, the looper can use `RegistrationToken` to manage calloop sources, but this should be an implementation detail stored in `LooperState`, keyed by the timer's `u64` id. The `Arc<AtomicBool>` flag triggers removal of the calloop source on the next dispatch cycle.
> 
> Could you get closer to linear? Yes — make `TimerToken` non-`Clone` and add `impl Drop for TimerToken { fn drop(&mut self) { self.cancel(); } }`. This gives cancel-on-drop (linear-ish: the protocol always terminates). But this breaks the current `set_pulse_rate` pattern, which stores the token in `Arc<Mutex<Option<TimerToken>>>` — that requires the token to be moveable into the mutex, not dropped on scope exit. You'd need to restructure to `token.take()` + explicit cancel, which is actually what `set_pulse_rate` already does. So cancel-on-drop is compatible if `Clone` is removed.
> 
> **Affine gap analysis for cancel-on-drop TimerToken:**
> - **Static guarantee:** Every TimerToken is either explicitly cancelled or cancelled on drop. Protocol `create → (fire)* → cancel` is enforced.
> - **What's lost vs linear:** Nothing meaningful — the "must use" property is satisfied by Drop.
> - **What's lost vs current:** `Clone` on TimerToken goes away. Any code that clones a token to cancel from multiple sites needs restructuring (use `Arc<TimerToken>` if needed, though this re-introduces the affine gap at the Arc level).
> 
> ### Question 2: Factory closures and calloop Timer callbacks
> 
> **Verdict: The factory pattern composes cleanly with calloop's callback model. The migration is a refactoring, not a redesign.**
> 
> calloop's timer callback signature (from `EventSource` for `Timer`):
> ```
> FnMut(Instant, &mut (), &mut Data) -> TimeoutAction
> ```
>

> 
> ```rust

> - rio(4) `wctl`: "A subsequent read will block until the window changes size, location, or state."
> - proc(3) `wait`: "If the process has no extant children... a read of wait will block."
> - rio(4) `mouse`: "Reading the mouse file blocks until the mouse moves or a button changes."
> 
> This requires one thread per observable per observer. For scripting, it's perfect. For a reactive UI framework, it doesn't scale.
> 
> **The plumber provides proto-observer semantics** but via a different mechanism: multicast message dispatch. From `reference/plan9/man/4/plumber`: "A copy of each message is sent to each client that has the corresponding port open." This is worth preserving in pane's routing — when a routing rule matches and the destination port has multiple listeners, all should receive the message.
> 
> **The `click` attribute** from `reference/plan9/man/6/plumb` is a pattern pane hasn't adopted yet. It carries cursor position within text, allowing the plumber's regex rules to narrow a selection to the semantically relevant portion. "The longest leftmost match to the regular expression that contains or abuts the textual location identified by the click." This should inform pane's routing rule design for content refinement.
> 
> ### 4. pane-shell (Terminal Emulator)
> 
> The 8½ paper (`reference/plan9/papers/8½/8½.ms`) describes the core architecture: 8½ is a file server that multiplexes `/dev/cons`, `/dev/mouse`, etc. Each window gets its own instance via per-process namespaces. The window creation sequence is 33 lines of C: fork, duplicate namespace, mount 8½'s pipe onto `/dev` with MBEFORE, open `/dev/cons` three times, exec shell. "The environment 8½ provides its clients is exactly the environment under which it is implemented" — hence recursive operation.
> 
> From `reference/plan9/man/4/rio`, the per-window file hierarchy:
> - `cons` — virtual terminal
> - `consctl` — mode control. **"Closing the file makes the window revert to default state (raw off, hold off)."** This is a lease pattern — holding the fd open holds the mode. Rust's RAII makes this natural for pane-shell: a `RawModeGuard` struct that reverts on drop.
> - `label` — window label (read/write), used as tag when hidden
> - `snarf` — clipboard
> - `text` — read-only full window contents
> - `wctl` — geometry + state, blocks on read until change
> - `wdir` — writable working directory, app maintains it, used for plumb context
> - `winid` — unique unchangeable window ID
> - `wsys` — directory of all windows by ID
> 
> pane-fs should mirror this hierarchy per pane: `body` (= text), `ctl` (= wctl), `attrs/title` (= label), `attrs/cwd` (= wdir).
> 
> ### 5. Authentication/Identity
> 
> From `reference/plan9/man/4/factotum` and `reference/plan9/papers/auth.ms`:
> 
> factotum is a per-user file server with files: `rpc` (private auth channel per open), `proto` (available protocols), `confirm` (interactive consent), `needkey` (credential prompting), `log`, `ctl` (key management). The RPC flow is: `start` (select protocol + role + key template) then `read`/`write` shuttle until `done`.
> 
> Three patterns worth adopting:
> 
> 1. **The `confirm` pattern:** When a key has the `confirm` attribute, factotum refuses to use it without interactive user confirmation via the `confirm` file. A GUI reads the confirm file, prompts the user, writes back the answer. For pane: when a remote agent tries to access local resources that `.plan` allows but are marked sensitive, an interactive confirmation could be required.
> 
> 2. **The `needkey` pattern:** When a required credential is missing, factotum asks an external program to provide it rather than failing. For pane: when a TLS certificate is missing for a remote connection, prompt the user rather than returning an opaque error.
> 
> 3. **Key auto-disable:** factotum marks failed keys with `disabled=by.factotum` — quarantine, don't retry. For pane: temporary blacklisting of PeerIdentity entries that fail authentication, to prevent brute-force retry.
> 
> The auth(5) message flow confirms that authentication is pluggable — 9P provides the channel (afid), the protocol over it is between factotums. pane already achieves this separation via the Transport trait.
> 
> ### 6. Clipboard (/dev/snarf)
> 
> From `reference/plan9/man/4/rio`: `snarf` returns the current buffer contents on read, sets them on write. No locking, no notification, last writer wins. Per-rio-session (all windows share one), with recursive rio delegating to parent's snarf.
> 
> The divergence between Plan 9's snarf and pane's clipboard design is large and justified. Plan 9's snarf was text-only, no MIME types, no metadata, no concurrency control. pane's `ClipboardWriteLock` typestate, `ClipboardMetadata` (sensitivity, TTL, locality), and named clipboards are all necessary for a modern system.
> 
> One insight to preserve: rio's recursive snarf delegation maps to pane's `Locality::Federated`. The inner instance uses the parent's buffer. pane's clipboard federation should make this automatic for nested/remote sessions, with `.plan` controlling sharing policy.

> ## Task 4: Patterns We Should Adopt But Haven't Considered
> 
> Seven concrete patterns from the primary sources that are not in the current roadmap:
> 
> ### 1. The `iostats` Transparent Proxy (names paper)
> 
> The names paper describes `iostats` — a command that wraps a process in a local namespace, monitoring all 9P requests. This maps directly to a `ProxyTransport` that wraps any `Transport` impl, logging all messages between app and compositor. The Transport trait makes this trivial to implement. Use case: debugging distributed sessions, performance profiling, compliance auditing.
> 
> **Recommendation:** Add a `--trace` flag to pane-headless that wraps all client connections in a logging transport. Low cost, high diagnostic value.
> 
> ### 2. The `aan(8)` Session Resilience Filter (import(4))
> 
> import(4)'s `-p` flag pushes `aan` onto the connection — a transparent filter that buffers messages during temporary disconnection and replays on reconnection. This is critical for mobile/WiFi scenarios where TCP connections drop frequently.
> 
> **Recommendation:** Implement a `ReconnectingTransport` wrapper with configurable timeout. Buffer outgoing messages during disconnection, replay on reconnection. After timeout, give up and emit `SessionError::Disconnected`. This sits between the application and the TLS transport, transparent to both.
> 
> ### 3. The `consctl` Lease Pattern (rio(4))
> 
> "Closing the file makes the window revert to default state." Holding an fd open is holding a lease on a mode. This maps perfectly to Rust's RAII — a `RawModeGuard` that reverts terminal mode on drop.
> 
> **Recommendation:** For pane-shell, and potentially for any pane mode that should revert (exclusive access, input capture), use ownership handles where drop = revert. This is already the pattern for `ClipboardWriteLock` — generalize it.
> 
> ### 4. The Plumber `click` Context Refinement (plumb(6))
> 
> The `click` attribute carries cursor position within text. The plumber uses it to narrow regex matches to "the longest leftmost match that contains or abuts the textual location identified by the click." This allows clicking in the middle of `rmstar.c:32:` and having the plumber correctly extract the filename and line number.
> 
> **Recommendation:** pane's routing rules should accept a cursor-context annotation. When a user middle-clicks text in a pane, the pane sends a routing message with the surrounding text *and* the click offset. The routing engine uses the offset to refine regex matches. This is a significant UX improvement for minimal implementation cost.
> 
> ### 5. The `wctl` Blocking-Read Observer for Scripting (rio(4))
> 
> pane-fs should expose per-pane files that block on read until state changes. This gives shell scripts free observability — `while read event < /pane/<id>/event; do handle $event; done`. No subscription setup, no teardown, no library needed. This coexists with the protocol-level push observer.
> 
> **Recommendation:** Priority for pane-fs: implement blocking-read event files alongside the static attr files. This is the cheapest way to make pane scriptable from day one.
> 
> ### 6. The `cpu` Reverse Export Pattern (names paper, drawterm(8))
> 
> Plan 9's `cpu` command exports the terminal's local devices (display, keyboard) *to* the CPU server, which imports them and runs programs against them. The terminal is a server for its own devices. drawterm implements this for non-Plan-9 machines.
> 
> **Recommendation:** pane already supports `App::connect_remote()` (app connects to compositor), but consider the inverse: a first-class operation where the local compositor *exports* its display/input services to a remote headless instance, so that remote apps can use local I/O without each app individually connecting. This would be pane's `cpu` — "run my computation on the cloud server, display it on my screen."
> 
> ### 7. Protocol Tracing for Headless Server (exportfs(4))
> 
> exportfs(4) has `-d -f dbgfile` to log all 9P traffic to a file. pane-headless should have equivalent functionality — a flag to dump all protocol messages in a human-readable format for debugging distributed sessions.
> 
> **Recommendation:** Add `--protocol-trace <file>` to pane-headless. Log all handshake and active-phase messages in a timestamped, human-readable format. This is invaluable for debugging federation issues and should be available from day one.
>

> **1. Pane count race: `fetch_sub` + separate `load` in early-exit path**

> }

> if pane_count.fetch_sub(1, Ordering::Release) == 1 {

>     done_signal.1.notify_all();
> }

> ## Additional Findings the Reviewers Missed
> 
> ### A. Handler trait: missing `frame_moved` equivalent
> 
> Haiku's `BWindow::FrameMoved` (Window.h:121) notifies when the window position changes. pane has `Message::Resize` for geometry changes but no equivalent for position-only changes. This is probably fine for the tiling model (the compositor controls position, and panes shouldn't care where they are), but it means pane has no way to notify a handler about its screen position — which matters for:
> 
> - Multi-monitor awareness (which monitor am I on?)
> - Coordinate translation for drag-and-drop
> - Popup positioning relative to parent
> 
> If pane ever needs position awareness, a `Moved` variant or extending `PaneGeometry` with position data would be the way to go. Not blocking, but worth noting as a deliberate gap.
> 
> ### B. Handler trait: missing `screen_changed` equivalent
> 
> Haiku has `BWindow::ScreenChanged` (Window.h:131-132) — notified when the screen resolution or color depth changes. In a Wayland compositor, output configuration changes are real events. If a pane is moved between monitors with different DPI, it needs to know. This should eventually be a `Message` variant or folded into a `Resize` delivery with DPI info. Currently absent.
> 
> ### C. Missing `B_READY_TO_RUN` parallel
> 
> BeOS's `B_READY_TO_RUN` (AppDefs.h:55) was sent to BApplication after all initial refs/args were delivered, signaling "initialization complete, start your main work." pane has `Message::Ready` for panes but nothing for `App` itself. In the current design where `App` is a factory (not a looper), this isn't needed — but if App ever grows lifecycle hooks, this gap will surface.
> 
> ### D. Filter chain: no `message_delivery` / `message_source` discrimination
> 
> Haiku's `BMessageFilter` (MessageFilter.h:30-40) could filter by delivery method (`B_DROPPED_DELIVERY` vs `B_PROGRAMMED_DELIVERY`) and by source (`B_REMOTE_SOURCE` vs `B_LOCAL_SOURCE`). pane's `MessageFilter::matches` takes `&Message` and can only discriminate by event type.
> 
> This matters for security: a filter that should only apply to network-originated events can't distinguish a `Key` from the local compositor versus a `Key` injected via `Messenger::send_message` from a worker thread. If pane's headless/remote model matures, filters may need provenance metadata. Not blocking now, but the `matches` signature should be designed to accommodate this without breaking changes — adding an `EventContext` parameter later would break all existing filters.
> 
> ### E. No `B_MESSAGE_NOT_UNDERSTOOD` / `B_NO_REPLY` protocol reply
> 
> BeOS had `B_MESSAGE_NOT_UNDERSTOOD` (AppDefs.h:111) as a standard reply when a handler couldn't process a scripting message. pane has `ScriptError::PropertyNotFound` which maps to this, and `ReplyPort::drop` sends `ReplyFailed` which maps to `B_NO_REPLY`. These are correctly mapped. Just confirming the reviewers didn't flag these as missing when they aren't.
> 
> ### F. Messenger::send_and_wait looper detection is thread-local only
> 
> The `is_looper_thread()` check at `proxy.rs:157` uses a `thread_local!` flag. This correctly prevents self-deadlock (calling `send_and_wait` from your own looper thread). But it doesn't prevent mutual deadlock between two panes: pane A's looper calls `send_and_wait` targeting pane B, while pane B's looper simultaneously calls `send_and_wait` targeting pane A. Both pass the thread-local check (neither is sending to itself) but both block forever.
> 
> BeOS had this exact same bug — mutual synchronous `SendMessage` between two loopers was a known deadlock. The BeBook warned against it, and Haiku's `BMessenger::SendMessage` documentation still says "never call this from a BHandler's MessageReceived if the reply might come from another looper."
> 
> pane's mitigation is better (the `WouldDeadlock` error, the async `send_request` path), but the mutual case still exists. Options:
> 
> 1. **Runtime detection:** Track in-flight `send_and_wait` targets globally. If A is waiting on B and B tries to wait on A, return `WouldDeadlock`. Cost: a global mutex on every `send_and_wait` entry/exit.
> 2. **Documentation:** Declare mutual `send_and_wait` as undefined behavior and document the `send_request` async path as the deadlock-free alternative.
> 3. **Timeout-based recovery:** `send_and_wait` already takes a timeout, so it won't hang forever — but 5 seconds of freeze is still a bug, not a fix.
> 
> I'd go with (2) for now and revisit if it bites anyone. The real answer is that looper-to-looper communication should be async, and `send_and_wait` should only be used from non-looper threads (CLI tools, worker threads, tests). The thread-local guard already pushes in this direction.
> 
> ### G. Coalescing skips Request messages in closure handler
> 
> In `looper.rs:352-359`, when a `Request` arrives in the closure handler path, the handler sees the message but the `reply_port` is dropped (sending `ReplyFailed`):
> 
> ```rust
> Unwrapped::Request(msg, reply_port) => {
>     let keep_going = handler(&proxy, msg)?;
>     drop(reply_port);
> ```
> 
> This is documented in the comment, but it means any pane using `Pane::run` (closure form) silently fails all incoming requests. If another pane does `send_request` or `send_and_wait` targeting a closure-based pane, the requestor gets `ReplyFailed` with no indication of why. The `handler` closure sees the message, may act on it (thinking it succeeded), but the requestor doesn't get a response.
> 
> In BeOS, every looper could receive and reply to messages regardless of how the handler was set up. This asymmetry between `run` and `run_with` is a divergence worth documenting prominently — either in `Pane::run`'s doc comment or as a `tracing::warn!` when a `Request` arrives on a closure handler.

> We encourage applicants to trust their own skills and write their own proposals. That being said, applicants may use GenAI tools in preparing applications, but any such use must be disclosed. This includes drafting, translation, or summarisation. It applies both to written proposals and to materials provided during interactive evaluation. Disclosure allows evaluators to understand how the proposal was produced and ensures fairness.

>

> Use of GenAI in project development
> 
>     Grantees must ensure that all submitted work can be legally published under a FLOS licence. This includes verifying that GenAI-assisted outputs do not reproduce copyrighted or incompatible material.
>         Example: when using a code assistant, check the assistant’s terms of use, and ensure that outputs are not reconstructed from copyrighted sources.
>         Example: Under EU law, purely AI-generated outputs without substantial human intellectual contribution are not eligible for copyright protection [Note]. In any case, outcomes purely generated by AI are not allowed to be submitted as work eligible for payment (as part) of the grant.
> 
>     Grantees must not present AI-generated content as if it were their own human-authored work.
>         Explanation: When we provide a grant to a person to develop a project, we expect that person to do the work. They should not outsource the work to another person while pretending they did it themselves. Similarly, grantees should not deliver GenAI outcomes and pretend it was their own human effort. Human contributors remain accountable for accuracy, originality, and integration of GenAI-supported work.
> 
>     Use of GenAI must not reduce the quality, clarity, reliability, or reproducibility of the work.
>         Explanation: Tools may assist, but human responsibility for quality remains. Human contributors are are expected to understand and be able to explain design and code decisions.
> 
>     It is allowed to work on the topic of GenAI itself within the scope of a grant, but only if this is explicitly part of approved work.
>         Explanation: We are not against GenAI, so applicants should not hesitate to submit proposals that involve work on GenAI. However, work on GenAI only falls within scope if it is part of the approved project plan.
> 
> Transparency & logging for project development
> 
> Use of GenAI should be disclosed and transparent. For any substantive use of GenAI that materially affects outputs, public disclosure is required, making it available to both users and contributors.
> 
>     The general stance toward the use of GenAI within a project should be disclosed and transparent for the public by providing a broad description.
>         Example: A codebase declares, typically in its ‘readme’, broadly how GenAI is used, e.g. for logic/boilerplate/tests/documentation/…. This can help set expectations for both users and contributors.
>         Example: A project publishes its own policy for contributors, outlining its do's and don't with regards to the use of GenAI.
> 
>     Generated content should be marked as such. When adding (partially) generated code, make sure the provenance is clear for each such contribution. Specify which model was used, (including version), and how it was used. Provide the used prompts/interactions and resulting output, or a summary thereof.
>         Example: When using git, distinguish commits that add generated code and include the used model and prompts in the commit message. Consider choosing a development tool that helps you create commits, and auto-fills the relevant information. See [Note 2] for example commits.
>         Make sure to provide the information in a logical place where it can easily be found. Avoid hosting it on third-party platforms that require a log-in or may disappear over time.
> 
>     If GenAI is not used for generating code but only for tasks like testing or creating documentation, it suffices to provide a general description of the use in the ‘readme’. More detailed logging on a per commit basis is preferred but not required.
> 
> Alternative methods for logging
> 
> The goal of disclosure is to inform NLnet, users and contributors about the extend to which GenAI was used to generate project results. If you prefer to use different methods for logging with equivalent results, this can be acceptable too. Use common sense to determine such equivalence and make sure you are able to answer questions about the use of GenAI from our team.
> Exceptions for grantees with active projects during the publication of the policy
> 
> We ask everyone to make an effort to uphold our request for transparency and logging. However, we also understand that for grantees with ongoing projects, this is a new requirement while work was already underway. Grantees of ongoing projects who feel that none of disclosure options offered above will work for them, can propose a personalised plan for transparency to their contact person at NLnet.
> An ongoing project in this context, is a project for which the Memorandum of Understanding was signed before December 8, 2025, the date when this policy came into force. Note that this is project-based and does not apply to returning grantees with projects starting after this date.
