> ## TYPE DUPLICATION AUDIT REPORT

> Based on my thorough exploration of the pane codebase across all five crates (pane-proto, pane-session, pane-app, pane-fs, pane-hello), I've identified several patterns of type duplication and polymorphism opportunities. Here are my findings:

> ### **FINDING 1: Obligation Handle Pattern (ReplyPort / CompletionReplyPort / CancelHandle)**
> 
> **Category:** (a) should unify now — but understand unified semantics first
> 
> **Location:**
> - `/Users/lane/src/lin/pane/crates/pane-proto/src/obligation.rs:55-82` — ReplyPort&lt;T&gt;
> - `/Users/lane/src/lin/pane/crates/pane-proto/src/obligation.rs:93-120` — CompletionReplyPort
> - `/Users/lane/src/lin/pane/crates/pane-proto/src/obligation.rs:131-156` — CancelHandle
> 
> **Issue:** Three obligation handles with nearly identical structure but different semantics:
> - **ReplyPort&lt;T&gt;**: carries success value T, sends `Result&lt;T, ReplyFailed&gt;` on `.reply()` or drop
> - **CompletionReplyPort**: no success value (unit type), sends `Result&lt;(), CompletionFailed&gt;` on `.complete()` or drop  
> - **CancelHandle**: no obligation—drop is a no-op, only `.cancel()` fires the continuation
> 
> All three follow the pattern:
> ```rust
> struct X {
>     tx: Option&lt;mpsc::Sender&lt;...&gt;&gt;,  // obligation carrier
> }
> // Methods: consume self (no Clone), success path consumes, Drop fires compensation
> ```
> 
> **Why they're separate:** The semantic difference is intentional:
> - ReplyPort &amp; CompletionReplyPort both have **mandatory** obligations (must reply or drop fires)
> - CancelHandle has **optional** obligation (drop is safe, `.cancel()` is voluntary)
> - Success paths carry different payloads (T vs () vs no payload)
> 
> **Unification option:** A generic `ObligationHandle&lt;T, E&gt;` parameterized by success type T and error type E. But this obscures the semantic intent (mandatory vs optional). **Current separation is justified — do NOT unify.** The three-way distinction is a feature, not a bug.
>
