> # Brainstorming Ideas Into Designs
> 
> Help turn ideas into fully formed designs and specs through natural collaborative dialogue.
> 
> Start by understanding the current project context, then ask questions one at a time to refine the idea. Once you understand what you're building, present the design and get user approval.
> 
> <HARD-GATE>
> Do NOT invoke any implementation skill, write any code, scaffold any project, or take any implementation action until you have presented a design and the user has approved it. This applies to EVERY project regardless of perceived simplicity.
> </HARD-GATE>

> ## Anti-Pattern: "This Is Too Simple To Need A Design"
> 
> Every project goes through this process. A todo list, a single-function utility, a config change — all of them. "Simple" projects are where unexamined assumptions cause the most wasted work. The design can be short (a few sentences for truly simple projects), but you MUST present it and get approval.
>

> ## The Process
> 
> **Understanding the idea:**
> 
> - Check out the current project state first (files, docs, recent commits)
> - Before asking detailed questions, assess scope: if the request describes multiple independent subsystems (e.g., "build a platform with chat, file storage, billing, and analytics"), flag this immediately. Don't spend questions refining details of a project that needs to be decomposed first.
> - If the project is too large for a single spec, help the user decompose into sub-projects: what are the independent pieces, how do they relate, what order should they be built? Then brainstorm the first sub-project through the normal design flow. Each sub-project gets its own spec → plan → implementation cycle.
> - For appropriately-scoped projects, ask questions one at a time to refine the idea
> - Prefer multiple choice questions when possible, but open-ended is fine too
> - Only one question per message - if a topic needs more exploration, break it into multiple questions
> - Focus on understanding: purpose, constraints, success criteria
> 
> **Exploring approaches:**
> 
> - Propose 2-3 different approaches with trade-offs
> - Present options conversationally with your recommendation and reasoning
> - Lead with your recommended option and explain why
> 
> **Presenting the design:**
> 
> - Once you believe you understand what you're building, present the design
> - Scale each section to its complexity: a few sentences if straightforward, up to 200-300 words if nuanced
> - Ask after each section whether it looks right so far
> - Cover: architecture, components, data flow, error handling, testing
> - Be ready to go back and clarify if something doesn't make sense
> 
> **Design for isolation and clarity:**
> 
> - Break the system into smaller units that each have one clear purpose, communicate through well-defined interfaces, and can be understood and tested independently
> - For each unit, you should be able to answer: what does it do, how do you use it, and what does it depend on?
> - Can someone understand what a unit does without reading its internals? Can you change the internals without breaking consumers? If not, the boundaries need work.
> - Smaller, well-bounded units are also easier for you to work with - you reason better about code you can hold in context at once, and your edits are more reliable when files are focused. When a file grows large, that's often a signal that it's doing too much.
> 
> **Working in existing codebases:**
> 
> - Explore the current structure before proposing changes. Follow existing patterns.
> - Where existing code has problems that affect the work (e.g., a file that's grown too large, unclear boundaries, tangled responsibilities), include targeted improvements as part of the design - the way a good developer improves code they're working in.
> - Don't propose unrelated refactoring. Stay focused on what serves the current goal.
> 
> ## After the Design
> 
> **Documentation:**
> 
> - Write the validated design (spec) to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
>   - (User preferences for spec location override this default)
> - Use elements-of-style:writing-clearly-and-concisely skill if available
> - Commit the design document to git
> 
> **Spec Self-Review:**
> After writing the spec document, look at it with fresh eyes:
> 
> 1. **Placeholder scan:** Any "TBD", "TODO", incomplete sections, or vague requirements? Fix them.
> 2. **Internal consistency:** Do any sections contradict each other? Does the architecture match the feature descriptions?
> 3. **Scope check:** Is this focused enough for a single implementation plan, or does it need decomposition?
> 4. **Ambiguity check:** Could any requirement be interpreted two different ways? If so, pick one and make it explicit.
> 
> Fix any issues inline. No need to re-review — just fix and move on.
> 
> **User Review Gate:**
> After the spec review loop passes, ask the user to review the written spec before proceeding:
> 
> > "Spec written and committed to `<path>`. Please review it and let me know if you want to make any changes before we start writing out the implementation plan."
> 
> Wait for the user's response. If they request changes, make them and re-run the spec review loop. Only proceed once the user approves.
> 
> **Implementation:**
> 
> - Invoke the writing-plans skill to create a detailed implementation plan
> - Do NOT invoke any other skill. writing-plans is the next step.

> ## Key Principles
> 
> - **One question at a time** - Don't overwhelm with multiple questions
> - **Multiple choice preferred** - Easier to answer than open-ended when possible
> - **YAGNI ruthlessly** - Remove unnecessary features from all designs
> - **Explore alternatives** - Always propose 2-3 approaches before settling
> - **Incremental validation** - Present design, get approval before moving on
> - **Be flexible** - Go back and clarify when something doesn't make sense

>

>

>

> ## 2. How BMessage Serialized
> 
> BMessage had **three distinct wire formats** across its history, identified by a magic number in the first 4 bytes:
> 
> **R5 format** (`'FOB1'`):
> ```
> [magic: u32][checksum: u32][flattened_size: i32][what: i32][flags: u8]
>   ... then conditional sections based on flags (target, reply info, field data)
> ```
> 
> **Haiku format** (`'1FMH'`), from `MessagePrivate.h`:
> ```c
> struct message_header {
>     uint32  format;          // magic: '1FMH'
>     uint32  what;            // message type code
>     uint32  flags;           // MESSAGE_FLAG_VALID, _REPLY_REQUIRED, etc.
>     int32   target;          // handler token
>     int32   current_specifier;
>     area_id message_area;    // -1 or area ID for large messages
>     port_id reply_port;      // where to send reply
>     int32   reply_target;    // handler token for reply
>     team_id reply_team;      // team owning reply port
>     uint32  data_size;       // total field data bytes
>     uint32  field_count;
>     uint32  hash_table_size;
>     int32   hash_table[5];   // field name hash → field index
> };
> ```
> 
> The flattened size is `sizeof(message_header) + field_count * sizeof(field_header) + data_size`. The `what` field **is** the message type discriminant — it's an int32 four-char-code like `'_QRQ'` (B_QUIT_REQUESTED) or `'PSTE'` (B_PASTE).
> 
> The receiver knows the message type because `what` is always at a fixed offset in the header. The magic number tells the unflatten code which format variant to parse.
> 
> ## 3. How BeOS Multiplexed on One Port
> 
> There were actually **two multiplexing layers**, and this is the architecturally interesting part.
> 
> **Layer 1: Port-level `code` field.** When BMessage sends through a port, it uses a *fixed* port code: `kPortMessageCode = 'pjpp'`. All BMessages sent through normal messaging use the same code. The code field distinguishes BMessages from the app_server link protocol (`kLinkCode = '_PTL'`). So the port code is a *protocol discriminant*, not a message-type discriminant.
> 
> **Layer 2: `what` inside the BMessage payload.** The `what` field is the message-type discriminant. BLooper's dispatch loop (`task_looper` at `/Users/lane/src/haiku/src/kits/app/Looper.cpp:1162`) calls `DispatchMessage(message, handler)`, which does a `switch(message-&gt;what)` for system messages (`_QUIT_`, `B_QUIT_REQUESTED`) and falls through to `handler-&gt;MessageReceived(message)` for everything else. The handler then typically switches on `what` in its own `MessageReceived` override.
> 
> There's also a **handler token** in the message header (`target`). When BLooper receives a message, it looks up the token to route to the right BHandler *before* the `what` dispatch happens. So the routing is: port → looper → handler (by token) → message type (by `what`).
> 
> **The app_server link protocol** (in `link_message.h`) is a separate protocol that multiplexes *differently*. It batches multiple small messages into a single port write, with its own framing:
> ```c
> struct message_header {
>     int32  size;   // includes this header
>     uint32 code;   // server protocol command
>     uint32 flags;  // kNeedsReply, etc.
> };
> ```
> 
> Multiple `message_header + payload` sequences are concatenated into one buffer and flushed as a single `write_port(port, kLinkCode, buffer, totalSize)`. The `LinkReceiver` parses them out sequentially. This was a performance optimization — reducing system calls for the high-frequency app_server communication.
> 
> ## 4. Message Size Limits
> 
> `PORT_MAX_MESSAGE_SIZE` is 256 KB. `write_port` returns `B_BAD_VALUE` if you exceed it.
> 
> BMessage handles large messages via a clever area-based mechanism (the comment at line 1103 of `Message.cpp` explains the design). When `fHeader-&gt;data_size &gt; B_PAGE_SIZE * 10` (40 KB), instead of flattening the whole message through the port:
> 
> 1. Copy message data into a shared memory area
> 2. Transfer the area to the target team via `_kern_transfer_area`
> 3. Send only the `message_header` (no field/data payload) through the port, with `message_area` set to the transferred area ID
> 4. The receiver builds a BMessage that references the area data directly (read-only)
> 5. Copy-on-write if the receiver needs to modify the message
> 
> This means the 256 KB port limit only applies to the header, not the message payload. Practical message size was limited only by available memory.
> 
> The link protocol has its own limit: `kMaxBufferSize = 65536` per port write, with a comment: "anything beyond that should be sent with a different mechanism."
> 
> ## 5. Failure Model
> 
> **Port deleted while writer is blocked waiting for queue space:**
> The writer re-locks and checks `is_port_closed(portRef)` — returns `B_BAD_PORT_ID`. Explicit, synchronous error to the sender. No silent drops.
> 
> **Port deleted while reader is blocked:**
> Same pattern — `read_port_etc` returns `B_BAD_PORT_ID` when the port disappears.
> 
> **Port closed (but not deleted):**
> `close_port` sets capacity to 0 and wakes all waiters with `B_BAD_PORT_ID`. Pending messages in the queue are still readable until `delete_port`. This is close/drain semantics.
> 
> **Messages already in the queue when port is deleted:**
> They're freed as part of physical deletion. The senders already returned `B_OK` — there's no way to notify them. This is fire-and-forget for enqueued messages.
> 
> **Team death:**
> `delete_owned_ports()` logically deletes all ports, then physically removes them. Pending writers to those ports get `B_BAD_PORT_ID`.
> 
> **Same-team optimization:**
> When sender and receiver are in the same team, BMessage bypasses the port entirely (line 2144 of `Message.cpp`). It acquires a direct reference to the target handler and enqueues a message copy directly into its BMessageQueue. This avoids the port size limit, avoids kernel transitions, and avoids the deadlock that can happen when the port queue is full and both threads are in the same team.

> ## 1. 9P's Message Framing
> 
> The 9P2000 header is:
> 
> ```
> size[4]  type[1]  tag[2]  ...parameters...
> ```
> 
> All fields are unsigned, little-endian.
> 
> **Size:** Four bytes. From intro(5): "Each 9P message begins with a four-byte size field specifying the length in bytes of the complete message **including the four bytes of the size field itself**." So Rflush — the smallest complete message (size + type + tag, no parameters) — has size = 7.
> 
> The canonical read loop (exportfs's `localread9pmsg`, line 410-442) demonstrates:
> 
> ```c
> // Read the 4-byte length
> m = readn(fd, buf, BIT32SZ);
> // Extract it
> len = GBIT32(buf);
> // Sanity: must be &gt; 4 (has to contain at least type+tag)
> if(len &lt;= BIT32SZ || len &gt; n) return -1;
> // Read the rest
> len -= BIT32SZ;
> m = readn(fd, buf+BIT32SZ, len);
> ```
> 
> **Type:** One byte, used for dispatch. The T/R types are interleaved in an enum (Tversion=100, Rversion=101, etc. in the Plan 9 4th edition). R-type = T-type + 1 by convention, with Rerror as a wildcard R-type that substitutes for any expected reply. The srv.c library dispatches with a switch on type (line 710), and exportfs uses a function-pointer table indexed by type (line 21-36, dispatched at line 400).
> 
> **Tag:** Two bytes, client-chosen correlation ID. The client must ensure uniqueness among outstanding requests. NOTAG (0xFFFF) is reserved for Tversion. The tag space is 65534 concurrent requests — never a practical limitation.
> 
> The type field is the demux mechanism. It separates Twalk from Tread from Tflush. There's no further multiplexing — 9P serves one file tree per connection. The tag field correlates requests with replies, enabling pipelining: "A client can send multiple T-messages without waiting for the corresponding R-messages, but all outstanding T-messages must specify different tags."
> 
> ## 2. 9P's Framing Error Handling
> 
> Three tiers:
> 
> **Tier 1 — Framing integrity violations: connection-fatal.**
> - Size &gt; msize: kernel mount driver (devmnt.c line 862) discards the queue and returns -1. exportfs (line 433) returns -1 with "bad length in 9P2000 message header". Connection dead.
> - Size &lt;= BIT32SZ (4 bytes): same path. A message that can't hold type+tag is corrupt.
> - convM2S returns 0 (parse failure on well-sized data): srv.c's getreq returns nil, exits service loop. exportfs calls `fatal()`. Connection dead.
> 
> The reasoning is correct and fundamental: once framing is lost, you can't find the next message boundary. Recovery is impossible without an external resynchronization mechanism, and 9P doesn't have one.
> 
> **Tier 2 — Semantic errors within well-framed messages: Rerror reply, connection continues.**
> - Unknown type: srv.c line 711-712, `respond(r, "unknown message")`. The message was parsed; the type just isn't recognized. Connection lives.
> - Duplicate tag: srv.c line 76-89, `r-&gt;error = Eduptag`. Error reply, connection lives.
> - Bad fid, wrong state, permission errors: all Rerror, connection lives.
> 
> **Tier 3 — Protocol-level session reset: Tversion.**
> A Tversion mid-session aborts all outstanding I/O and clunks all fids. It's the "nuke from orbit" option for recovering from a confused state. Both sides go back to square one. This isn't an error path exactly — it's a designed reset mechanism.
> 
> **What pane should take from this:** The two-tier split (framing error → connection-fatal, semantic error → per-service error reply) is the right architecture. Your spec already has this: "A frame exceeding the negotiated max_message_size is a connection-level error" and "I12: Unknown service discriminant → connection-level error." I'd actually question I12: in 9P, an unknown type gets an Rerror, not a connection kill. The reasoning is that if the frame parsed correctly, you can skip it safely. An unknown service discriminant in a well-framed pane message means you got length + service + payload, the length was valid, you just don't know what service 0x37 is. You could reply with an error on service 0 (Control) rather than tearing down the connection. The question is whether service IDs are negotiated (in which case an unknown one signals a serious protocol violation) or could legitimately appear due to version skew. Since pane assigns them dynamically via DeclareInterest, an unknown discriminant means one side's state is wrong — that's closer to "framing is broken" than "I just don't understand this message." So I12 being connection-fatal is defensible, though I'd document the reasoning.
> 
> ## 3. ProtocolAbort vs. 9P
> 
> 9P had nothing like ProtocolAbort. Transport death was detected by read/write failures — `read9pmsg` returns 0 on EOF, -1 on error, "it does not return partial messages." That was sufficient because:
> 
> - In-kernel, connections were pipes or device channels with reliable EOF semantics.
> - Network connections went through the kernel's network stack, which delivered errors promptly.
> - The mount driver handled cleanup on read failure — all pending Mntrpc structures were unwound.
> 
> Pane needs ProtocolAbort because it operates in user space over TCP, where:
> 
> - TCP half-close, keepalive timeouts, and NAT state expiry can leave the other side waiting for data that will never come.
> - A crashed process might not cleanly close its socket, leaving the peer blocked in read.
> 
> **The two-byte sentinel has a framing problem.** Your framing is `[length: u32][service: u8][payload]`. The reader reads a u32 length first. If you send `[0xFF][0xFF]` (two bytes), the reader is blocked waiting for 4 bytes of length. If the sender then closes the connection, the reader gets a short read (2 bytes instead of 4). That short read is itself an error condition — you don't need the sentinel to detect it. The sentinel is only useful if it's the same width as the length field, so the reader reads a complete u32 and then checks "is this the abort pattern?"
> 
> Options:
> 1. Make the sentinel 4 bytes: `[0xFF, 0xFF, 0xFF, 0xFF]` — a u32 length of 0xFFFFFFFF (4 GB), which is guaranteed to exceed any sane max_message_size. The reader reads 4 bytes, sees a length that exceeds max_message_size, and can then check "is it specifically the abort sentinel?" before treating it as a generic oversized-frame error.
> 2. Make the sentinel 5 bytes: `[0x00, 0x00, 0x00, 0x00, 0xFF]` — a zero-length frame (which is invalid, since length doesn't include itself but payload must have at least the service byte), followed by 0xFF as a service-level marker. The reader reads the u32 (0), recognizes it as invalid, reads one more byte to distinguish abort from corruption.
> 3. Abandon the sentinel entirely and rely on transport-level EOF/error, as 9P did. This is simpler and honestly might be sufficient if you set TCP keepalive and socket timeouts appropriately.
> 
> I'd lean toward option 1 or option 3. The current two-byte sentinel is the wrong width for the framing format.
> 
> ## 4. What Pane Should Learn
> 
> **Is `[length: u32][service: u8][payload]` the right shape?**
> 
> Yes — it's structurally isomorphic to 9P's `[size[4] type[1] ...]` with one deliberate omission: there's no tag field. 9P needed tags because it multiplexed many requests and replies on one connection, with out-of-order responses. Pane's architecture is different: the service discriminant routes to separate protocol handlers, and within each handler the par session types enforce protocol ordering. If the session types guarantee that at most one message is outstanding per service at a time, you don't need tags. If a service protocol allows pipelining (multiple requests before a reply), you'll need request correlation — either tags per service or some other mechanism. Check whether any of your service protocols allow this.
> 
> **Should length include itself?**
> 
> Your spec (line 1097-1098) says: "The `length` field counts the service byte plus payload. The length field does NOT include itself."
> 
> 9P included the size field in itself. Both conventions work. The trade-offs:
> 
> - **Inclusive (9P style):** The value you read from the wire is the total number of bytes to allocate for the complete message buffer. Simple for allocation. But you have to subtract BIT32SZ to know how many more bytes to read after the length, and a minimum valid length is 5 (4 for size + 1 for at least a type byte), so the lower bound check is `len &lt;= BIT32SZ`.
> - **Exclusive (pane style):** The value you read is exactly how many more bytes to read from the wire. Simpler read loop. But you have to add 4 to get the total allocation size, and the minimum valid length is 1 (at least the service byte). The lower bound check is `len &lt; 1`.
> 
> Either is fine. Exclusive is slightly cleaner for the read loop, which is the hot path. 9P's inclusive convention was partly historical — Plan 9's GBIT32/PBIT32 macros and the serialization library were designed around "total message size in the first 4 bytes," and that's how the kernel's block-buffer management worked. Pane doesn't inherit that machinery.
> 
> My recommendation: keep exclusive. It's what you have, it's marginally simpler for reading, and changing it now would be pure churn. Just be explicit in the spec about which convention you use — the 9P world assumed inclusive, so anyone coming from that background will assume the same.
> 
> **Is 256 discriminants (u8) enough?**
> 
> 9P's type field was also a u8, and it defined about 26 values (13 T/R pairs). It was never close to exhaustion. Pane uses the service discriminant differently — it's a session-local routing slot, assigned dynamically per DeclareInterest. The question is: will a single pane connection ever need more than 255 simultaneously active services (service 0 is Control, leaving 1-255)?
> 
> For a windowing framework: almost certainly yes, 255 is enough. Consider what services look like — Display, Clipboard, an AI inference service, a settings service, maybe a dozen domain-specific ones. A pane connecting to hundreds of services simultaneously is an architectural smell, not a reasonable use case.
> 
> If you ever hit it, the solution isn't widening the discriminant. It's opening a second connection to the same server (which your Open Question 3 already contemplates). Multiple connections with independent discriminant spaces scale indefinitely.
> 
> Keep the u8.
> 
> **Partial reads (TCP delivers a partial frame):**
> 
> 9P handled this with `readn` — a wrapper that loops `read()` until it has exactly n bytes or hits EOF/error. The exportfs code (line 424, 438) shows the pattern:
> 
> ```c
> m = readn(fd, buf, BIT32SZ);       // read exactly 4 bytes
> if(m != BIT32SZ) return error;
> len = GBIT32(buf);
> // validate len
> m = readn(fd, buf+BIT32SZ, len-BIT32SZ);  // read exactly the rest
> if(m &lt; len-BIT32SZ) return 0;       // short = EOF
> ```
> 
> The pattern is: read in two phases: (1) read the fixed-size header (length field), (2) read exactly that many more bytes. Each phase loops until complete or errors.
> 
> For pane in Rust, this maps cleanly to `AsyncReadExt::read_exact()` or the equivalent in your I/O layer. The read loop should be:
> 
> ```
> 1. read_exact(4 bytes) → length: u32
> 2. validate: length &gt;= 1, length &lt;= max_message_size
> 3. read_exact(length bytes) → service + payload
> 4. demux on service byte
> ```
> 
> If read_exact returns early (connection closed mid-frame), that's a connection error. No partial frame processing, no buffering across frames. This is identical to 9P's approach and it's the right one. Do not attempt to recover from mid-frame connection drops.

> ## 5. msize Negotiation (Tversion/Rversion)
> 
> From version(5): "The client suggests a maximum message size, msize. [...] The server responds with its own maximum, msize, which must be less than or equal to the client's value. Thenceforth, both sides of the connection must honor this limit."
> 
> The negotiation is **not** "both propose, take the minimum." It's: **client proposes a ceiling, server responds with an equal or lower value.** The server can never increase the client's msize. This is enforced (devmnt.c line 195-196):

> ```c
> if(f.msize &gt; msize)
>     error("server tries to increase msize in fversion");
> ```
> 
> The kernel also enforced sanity bounds (devmnt.c line 197):
> 
> ```c
> if(f.msize&lt;256 || f.msize&gt;1024*1024)
>     error("nonsense value of msize in fversion");
> ```
> 
> In practice, the typical msize was 8192+IOHDRSZ (about 8KB + header overhead). IOHDRSZ was defined as the maximum 9P header size, so the payload data for a Tread/Rread was at most msize-IOHDRSZ bytes. The MAXRPC constant in the kernel was `IOHDRSZ+8192` (devmnt.c line 19).

>     messagesize = t-&gt;work.msize;       // update effective msize

> 
> `min(client_msize, server_msize)` — though expressed as "server clamps, never increases."
> 
> **How this maps to pane's Hello/Welcome:**
> 
> Your spec says: "Both sides send their maximum; the effective limit is the minimum. Default: 16MB."
> 
> That's functionally equivalent to 9P's model. The only difference is whether you express it as "take the minimum symmetrically" or "server clamps to client's value." In Welcome, the server sends `min(client_max, server_max)`. Done.
> 
> One thing 9P enforced that pane should too: a minimum. The kernel rejected msize &lt; 256. For pane, I'd set a floor that's at least large enough to encode any control message. A max_message_size of 64 bytes is either a bug or an attack.

> 1. An actor is **idle** with state `b`
> 2. It has a stored handler for role `s[p]` paired with a failure callback `a`

> ## Test 2: `e_suspend_e_react_end_to_end` — **PASS, with one weakness**
> 
> ### EAct correspondence
> 
> The test claims to validate the full E-Suspend followed by E-React cycle. The relevant rules are:
> 
> **E-Suspend** (Fowler/Hu, Figure 4, line 2007; extended in Figure 12, line 3250): An actor currently evaluating in session context `s[p]` evaluates `suspend(c, a, b)`, which:
> - Installs handler `c` (paired with failure callback `b` in the `EAct_!` extension) for role `s[p]`
> - Returns the actor to `idle(a)` state
> 
> **E-React** (Fowler/Hu, Figure 4, line 1986; extended in Figure 12, line 3221): An idle actor with a stored handler for `s[p]`, and a matching message in the session queue, transitions to:
> - Dequeue the message
> - Evaluate the handler body with the message payload and current state
> - Remove the stored handler from `H`
> 
> The test captures:
> 
> - **E-Suspend: entry installation.** `dispatch.insert(conn, entry)` installs a one-shot handler keyed by `(ConnectionId, Token)`. The assertion `dispatch.len() == 1` verifies the handler is stored. This corresponds to the `H[s[p] -&gt; (c, b)]` update in E-Suspend.
> 
> - **Idle return.** After `insert`, control flow returns to the test body (the caller). In the real looper, this means the looper returns to its event loop — the actor is idle. The test's sequential control flow models this: between `insert` and `fire_reply`, the actor is conceptually idle, waiting for a message.
> 
> - **E-React: message arrival and handler invocation.** `fire_reply(conn, token, &amp;mut handler, &amp;messenger, payload)` dequeues the stored handler and evaluates it with the payload. The assertion `events.contains("on_reply:hello")` verifies the handler body was evaluated with the message payload and had access to mutable handler state (`&amp;mut H`).
> 
> - **One-shot consumption.** `dispatch.len() == 0` after `fire_reply`. This matches E-React removing the stored handler from `H`.
> 
> - **Failure callback does not fire.** `!events.contains("on_failed_BUG")` verifies that the success path (E-React) and failure path (E-CancelH) are exclusive. This is correct: in the paper, E-React and E-CancelH have incompatible preconditions (E-React requires a matching message; E-CancelH requires a cancelled peer and an empty queue).
> 
> ### The key property: idle between suspend and react
> 
> The test comment claims to capture "the actor returns to idle between suspend and react." This is the critical property from Corollary 4.10 (line 3119): every thread-terminating actor eventually returns to idle via E-Suspend or E-Return.
> 
> The test captures this **structurally but not observationally**. The sequential control flow between `insert` (E-Suspend) and `fire_reply` (E-React) means the actor's state *is* conceptually idle — no handler callback is executing, the dispatch table holds the pending entry, and the next action is driven by an external event (the reply arriving). But the test doesn't assert any observable property of the idle state. There's no "the handler was not touched between suspend and react" assertion.
> 
> This is a real weakness, but a minor one. In the real looper, the idle property is enforced by the calloop event loop: after `insert`, control returns to `calloop::EventLoop::dispatch`, which blocks until the next event. The test can't model this without calloop. What the test *does* model is that `fire_reply` works correctly when called after `insert` with no intervening dispatch — i.e., the idle invariant is respected by construction.
> 
> ### The borrow split
> 
> The test comment mentions "the real looper splits the borrow between &amp;mut Dispatch and &amp;mut H when calling fire_reply." The test demonstrates this by accessing `core.dispatch` and `core.handler` separately. This is not an EAct property per se, but it's load-bearing for the Rust implementation: the looper can't call `fire_reply` through `LooperCore::dispatch()` because that takes `&amp;mut self` and passes a callback receiving `&amp;mut H`, but `fire_reply` needs `&amp;mut Dispatch&lt;H&gt;` and `&amp;mut H` simultaneously. The test validates that the split-borrow pattern works.
> 
> ### What the test does NOT capture
> 
> 1. **State threading.** In E-React, the handler body is evaluated with both the message payload *and* the current actor state: `M{a/x, b/st}`. The test's `on_reply` callback receives `&amp;mut H` (the full handler, not a state value) and the payload. The state threading is implicit in the `&amp;mut H` borrow — the callback can read and write handler state. This is correct but loses the explicit state-passing semantics of the calculus. Not a soundness issue.
> 
> 2. **Handler body returning a new suspend.** In the paper, the handler body `M` typically evaluates to another `suspend(c', a', b')`, re-entering E-Suspend and continuing the protocol. The test's `on_reply` returns `Flow::Continue`, which tells the looper to return to idle without installing a new handler. This models the protocol-terminating case (the handler doesn't need to continue the session). A stronger test would have `on_reply` install a new dispatch entry and verify the cycle repeats.
> 
> 3. **catch_unwind integration.** The real looper wraps every callback in `catch_unwind`. This test bypasses `LooperCore::dispatch()` and calls `fire_reply` directly on the dispatch table. A panic in `on_reply` would propagate uncaught. This is intentional — the test isolates E-Suspend/E-React from the panic boundary — but it means the test doesn't validate that E-React composes correctly with the panic=unwind invariant (I1).
> 
> ### Missing assertions that would strengthen EAct correspondence
> 
> One assertion would meaningfully strengthen this test:
> 
> **Assert the handler state was actually mutated by on_reply.** The test checks that a log message was pushed, but the log is in an `Arc&lt;Mutex&lt;Vec&lt;String&gt;&gt;&gt;` shared outside the handler. A stronger assertion would verify that a field on `TestHandler` itself was modified by the callback, confirming that `&amp;mut H` in the callback has the same identity as `core.handler`. This would directly validate the E-React property that the handler body runs "in the context of" the actor's state.
> 
> Currently the test does this indirectly — the log is stored as `Arc&lt;Mutex&lt;...&gt;&gt;` inside `TestHandler`, and the callback accesses it through `h.log`. But a direct field mutation (e.g., `h.replies.push(...)` then `assert!(!core.handler.replies.is_empty())`) would make the state-threading property explicit. However, after `fire_reply` the borrow on `core.handler` is released, so `core.handler.log` *could* be checked. The test checks `log.lock().unwrap()` instead, which goes through the shared Arc — equivalent but less direct.
