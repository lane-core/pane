> ## 1. compiler.ms — "Plan 9 C Compilers"
> 
> **Author:** Ken Thompson (ken@plan9.bell-labs.com)
> 
> ### Exemplary sentences (writing voice)
> 
> 1. "The compiler is a single program that produces an object file."
>    -- The canonical Plan 9 opening move: one sentence, declares the entire architecture. No hedge, no motivation, no history. The thing is what it does.
> 
> 2. "The keyword `register` is recognized syntactically but is semantically ignored."
>    -- Precise distinction between two levels of meaning in six words. Notice the parallel construction: "recognized syntactically / ignored semantically."
> 
> 3. "Structures are now almost first-class citizens of the language."
>    -- Rare moment of personality. "Almost" does real work here -- it's a precise qualification, not a hedge. "Citizens" is the only metaphor in the entire paper.
> 
> 4. "Since this code generator is so small (less than 500 lines of C) and easy, it hardly seems worth the effort."
>    -- Dismissal of an entire subfield of compiler research in one sentence. The parenthetical acts as evidence; "hardly seems worth the effort" is the verdict. The tone is not arrogant -- it's just settled.
> 
> 5. "The new compilers compile quickly, load slowly, and produce medium quality object code."
>    -- Three-part verdict on their own work. No inflation. "Medium quality" is brutal self-honesty. The rhythm is tricolon with descending confidence.
> 
> ### Key technical insights
> 
> - **Single-program compiler, binary object files.** The compiler combines preprocessor, lexer, parser, codegen, local optimizer, and half the assembler into one program. The loader does the other half plus global optimization. This is the opposite of the Unix pipeline-of-passes model. Rationale: simplicity and speed.
> 
> - **Parallel compilation bottleneck.** On multiprocessors, all compilations must finish before the single-threaded loader can begin. "Any shift of work from compile to load results in a significant increase in real time." This is the fundamental tension in any compile-then-link model.
> 
> - **Cross-compilation is the default, not a special mode.** All code is machine-independent and can be cross-compiled. Object files are architecture-independent in format (known byte orders). There is no "native" vs "cross" distinction.
> 
> - **Addressability as a porting knob.** The compiler's machine-dependent layer is captured in a small table (11-18 rules) describing what expressions are legal as instruction operands. When porting, you start with "only leaves are addressable" -- bad code, but it works. Then you refine the table. This is an excellent model for incremental backend development.
> 
> - **Caller-saves convention chosen deliberately.** "The convincing argument is that with caller-saves, the decision to registerize a variable can include the cost of saving the register across calls." System calls look like procedure calls, so the kernel entry path is fast.

> ## 2. mk.ms — "Maintaining Files on Plan 9 with Mk"
> 
> **Authors:** Andrew G. Hume (andrew@research.att.com), Bob Flandrena (bobf@plan9.bell-labs.com)
> 
> ### Exemplary sentences (writing voice)
> 
> 1. "Unlike `make`, `mk` has no built-in rules."
>    -- Design decision stated as a fact, in contrast to the predecessor. The "unlike" construction is a Plan 9 staple: define yourself against what you replaced.
> 
> 2. "A partially built file is doubly dangerous: it is not only wrong, but is also considered to be up to date so a subsequent `mk` will not rebuild it."
>    -- Explains a subtle failure mode in two clauses. "Doubly dangerous" is the strongest emotional language in the paper; the colon delivers the proof.
> 
> 3. "Since `mk` does not interpret a recipe, escape conventions are exactly those of `rc`."
>    -- A non-feature stated as a feature. The benefit is that there's one less layer of interpretation to understand. This is a recurring Plan 9 rhetorical move: absence as design.
> 
> 4. "Plan 9 presents a difficult maintenance environment with its heterogeneous architectures and languages."
>    -- Rare acknowledgment of difficulty. It frames the tool as a response to a real constraint, not an improvement for its own sake.
> 
> 5. "Metarule patterns should be as restrictive as possible to prevent conflicts with other rules."
>    -- Prescriptive guidance stated as a principle, not a warning. No "you should" or "we recommend" -- just a declarative rule.
> 
> ### Key technical insights
> 
> - **No built-in rules; everything via include.** mk has zero implicit knowledge about C, assembly, or any language. All rules come from included prototype mkfiles (e.g., `&lt;/$objtype/mkfile`). This is the opposite of GNU make's extensive implicit rule database. The benefit is that the tool never surprises you with behavior you didn't ask for.
> 
> - **Environment-variable-driven cross-compilation.** `$objtype` selects the target architecture. The same mkfile builds for any architecture by changing one variable. This is the build-system corollary to the compiler's cross-compilation-by-default.
> 
> - **Recipes are passed to rc without interpretation.** mk does not parse, substitute, or execute recipes itself. It hands the entire recipe to `rc -e` (shell with exit-on-error). This eliminates an entire class of quoting and escaping bugs. One consequence: variables in recipes have their *last* assigned value, not the value at the point of definition.
> 
> - **Parallel execution by default.** `$NPROC` controls parallelism; when there is no dependency between targets, mk runs recipes concurrently. The paper warns explicitly that shared resources (archives, databases) need serialization via dependency specification, not ad-hoc locking.
> 
> - **Graph-first, then execute.** mk constructs the entire dependency graph (including transitive closure of metarules) before executing any recipe. make builds targets on demand, allowing systematic use of side effects. mk forbids this. Cyclic dependencies are rejected.

>

>    -- The actual rebuttal: simplicity beats micro-optimization. "This general way" is a quiet flex -- it's general, not special-cased, and that's the point.

> 5. "In practice, Standard I/O is not used much in Plan 9."
>    -- Entire library ecosystem dismissed in one sentence. No argument, just observation. The reader is left to infer the reasons (bio is better, stdio is bloated, the design philosophy diverges).

> ### Key technical insights
> 
> - **One compiler per architecture, all cross-compile.** `vc` is always the MIPS compiler regardless of what machine runs it. `/bin/vc` is bound via the namespace to the correct binary for the host architecture. There is no `configure` step, no `--target` flag. The namespace does the work.
> 
> - **Header files correspond to libraries, not categories.** Plan 9 rejects the ANSI C model of grouping functions by kind (string.h, memory.h, stdio.h). Instead, each header corresponds to a library: if you link the library, include the header. Headers are not self-protecting against multiple inclusion and contain no nested includes. The programmer sorts out the dependency order, which in practice is trivial.
> 
> - **Canonical byte order over byte swapping.** Programs read and write data in a defined byte order, never swapping. The `getlong()` example reads bytes in MSB-first order regardless of host architecture. "Note that this code does not 'swap' the bytes; instead it just reads them in the correct order." > 
> - **UTF-8 as the native text encoding.** All text in Plan 9 is UTF-8 (they just call it UTF). ASCII is a proper subset. `Rune` is the 21-bit character type. The paper is careful to explain that `strchr` is wrong for multi-byte characters and `utfrune` is the correct replacement -- even when you "know" the character is single-byte, because assuming nothing about the encoding is safer.

> - **The loader rearranges data.** Variable order in the binary is unrelated to declaration order in source. Most programs don't care, but those that assume adjacency will break. This is stated as a fact, not apologized for -- it's the price of optimization, and the price is low.

> 
> Across all four papers, the Plan 9 voice has these consistent characteristics:
> 
> **Sentence structure:**
> - Lead with the fact, follow with the consequence. Never the reverse.
> - Semicolons over periods when the second clause depends on the first.
> - Parentheticals carry real information (sizes, exceptions, caveats), never filler.
> - Tricolon for emphasis, usually with ascending severity or descending sympathy.
> 
> **Voice:**
> - First person plural ("we") only for concrete actions the authors took. Never editorial "we."
> - Direct address ("you") is rare and reserved for practical advice.

> **The multiplexer-as-reproducer pattern.** 8½ runs in the same environment it provides to clients. Pane's calloop-based dispatch is the modern answer to this — a single event loop that multiplexes without requiring threads or async, exactly the compromise Pike wanted but couldn't get in C.

> ### Gap 1: app_server Wire Protocol Architecture
> 
> **What's missing:** The actual structure of the app_server link protocol — how `LinkSender`/`LinkReceiver` work, the message batching/flushing model, the `StartMessage/Attach/EndMessage/Flush` pattern, and why this matters for pane-session design.
> 
> **Content to add:**
> 
> The app_server link protocol is a buffered binary protocol over kernel ports (not BMessage-based). Key facts:
> - `LinkSender` buffers multiple messages before flushing. `StartMessage(code)` begins a message, `Attach(data, size)` appends typed fields, `EndMessage()` terminates, `Flush()` writes the batch to the port. Multiple messages can be batched before a single `Flush()`.
> - `LinkReceiver` reads from the port with `GetNextMessage(code)`, then pulls fields with `Read&lt;Type&gt;()`. No self-describing format — both sides must agree on field order.
> - `ServerLink` composes a `LinkSender` + `LinkReceiver` into a bidirectional channel. `FlushWithReply()` flushes then blocks for a response — this is the sync call path.
> - The protocol enum is in `ServerProtocol.h` (~370 opcodes covering window management, drawing, cursor, font, screen, view state). Each opcode implies a specific sequence of Attach calls.
> - `PortLink` and `AppServerLink` are thin wrappers providing default port configuration. `AppServerLink` connects to the app_server's well-known port.
> - Batching was critical for performance. George Hoffman (Newsletter Issue 2-36): async calls are batched and sent in chunks; sync calls force a flush + round-trip, making them "much slower."
> 
> **Pane mapping:** pane-session's postcard-serialized framing is the equivalent of this protocol. The key lesson is that the wire protocol between client and server should NOT be self-describing BMessage format — it should be a compact binary protocol where both sides agree on the schema (which is exactly what postcard + Rust types give you). The batching insight maps to pane's ability to coalesce multiple messages in a single calloop wakeup.
> 
> ### Gap 2: BLooper Lock Contention — The Historical Problem
> 
> **What's missing:** Why BLooper's `Lock()`/`Unlock()` model was both essential and problematic, and how pane's `&amp;mut self` avoids it.
> 
> **Content to add:**
> 
> BLooper (and its subclass BWindow) had a benaphore-based lock. Any thread that wanted to access the looper's handlers or state had to `Lock()` it first. This was necessary because:
> 1. The looper's own thread dispatches messages to handlers
> 2. Other threads (spawned workers, other windows) might need to access the same data
> 3. The app_server sends update/input messages that the looper thread processes
> 
> The problems:
> - **Deadlock potential:** Be Commandment #2 ("Thou shalt not lock the same objects in differing orders") was the rule, but enforcing it was the developer's responsibility. Lock ordering bugs were one of the most common BeOS application bugs.
> - **Priority inversion:** A high-priority thread (e.g., the window thread needing to process updates) could block on a lock held by a low-priority worker thread.
> - **Aliasing after delete:** Pavel Cisler (Newsletter Issue 3-33) documented a subtle bug where a deleted looper could be replaced by a new one at the same address, making pointer-based locks unreliable. BMessenger-based locking was the safer alternative because it checked identity, not just address validity.
> - **Lock scope confusion:** Developers often held locks too long, blocking the window thread and causing the app_server to discard update messages (leading to rendering glitches).
> 
> Pane's solution is structural: the handler has `&amp;mut self` access during dispatch, which the Rust borrow checker enforces at compile time. No locks needed because the handler state is only accessed from the looper's thread. Cross-thread communication goes through `Messenger::send_message()` (which enqueues to the calloop channel), never through direct state access. This eliminates all four problems above by construction.
> 
> ### Gap 3: BMessage Internals — Flattening and the Reply Mechanism
> 
> **What's missing:** How BMessage was structured internally (hash table fields, flattening format) and how the reply chain worked.
> 
> **Content to add:**
> 
> BMessage internals (from `MessagePrivate.h` and `Message.cpp`):
> - `message_header`: 32-bit `format`, `what` code, `flags` bitfield, `target` token, `current_specifier` index, `message_area` for shared memory, reply chain (`reply_port`, `reply_target`, `reply_team`), body info (`data_size`, `field_count`, `hash_table_size`, `hash_table[5]`)
> - `field_header`: `flags`, `name_length`, `type_code`, `count`, `data_size`, `offset`, `next_field` (linked list for hash collision)
> - Fields are stored in a flat buffer with a small hash table (5 buckets) for lookup by name. Field names are strings; values are typed by `type_code` (B_STRING_TYPE, B_INT32_TYPE, etc.).
> - Flags include: REPLY_REQUIRED, REPLY_DONE, IS_REPLY, WAS_DELIVERED, HAS_SPECIFIERS, WAS_DROPPED, PASS_BY_AREA, REPLY_AS_KMESSAGE.
> 
> The reply mechanism:
> - Every BMessage carries `reply_port`/`reply_target`/`reply_team` — the address where a reply should be sent.
> - When a message is delivered to a BLooper, the delivery code sets these fields to point back to the sender's looper.
> - `BMessage::SendReply()` uses these fields to send the reply. The `REPLY_REQUIRED` flag enforces that a reply must be sent; `REPLY_DONE` tracks whether it has been.
> - `BMessage::SendMessage(msg, &amp;reply)` (synchronous) creates a temporary reply port, sets REPLY_REQUIRED, sends, then blocks waiting on the reply port.
> 
> **Pane mapping:** pane separates these concerns. The `Message` trait carries data only (no reply addressing). `ReplyPort` is a separate obligation handle that is consumed by `.reply()`. The reply address is in the `ReplyPort`, not in the message itself. `CancelHandle` is the sender's side. This separation means messages are `Clone`-safe (EAct KP2) — something BMessage could never be because it carried mutable reply state.
> 
> ### Gap 4: BClipboard — The Locking Model
> 
> **What's missing:** How BClipboard worked and why it was simpler than X11 selections.
> 
> **Content to add:**
> 
> BClipboard was a centralized, lock-based clipboard:
> - The clipboard data lived in the registrar (not in the app that copied it).
> - To read: `Lock()` the clipboard, get the BMessage containing the data, `Unlock()`.
> - To write: `Lock()`, `Clear()`, add data to the BMessage, `Commit()`, `Unlock()`.
> - The lock was system-wide (only one process could hold it at a time).
> - The data was a BMessage, so it could carry multiple representations (text, styled text, bitmap, etc.) simultaneously.
> - `StartWatching()`/`StopWatching()` notifications when the clipboard changed.
> 
> Why it was simpler than X11:
> - X11 selections are pull-based: the paste target requests data from the copy source, which must still be running. If the source exits, the clipboard is lost.
> - BClipboard was push-based: the copy operation immediately uploaded data to the registrar. The source could exit and the data persisted.
> - X11 requires MIME type negotiation between source and target per selection request. BClipboard stored all representations at copy time.
> - X11 has three separate selection buffers (PRIMARY, SECONDARY, CLIPBOARD) with different conventions. BeOS had one clipboard (with named clipboard support added later).
> 
> Server-side implementation (from `src/servers/registrar/Clipboard.cpp` + `ClipboardHandler.cpp`): surprisingly small — 111 + 225 LOC. The registrar stores a BMessage per named clipboard and manages a watcher list for change notifications.
> 
> **Pane mapping:** pane models clipboard as a protocol (`Handles&lt;Clipboard&gt;`) with `ClipboardWriteLock` as a typestate handle. The `Lock→Clear→Write→Commit→Unlock` sequence becomes acquire `ClipboardWriteLock` → set data → `.commit()`. The typestate handle is consumed by commit, making the "forgot to commit" bug impossible. The pull-vs-push decision maps to Wayland's clipboard model (which is pull-based like X11, but with MIME negotiation happening through the compositor).
> 
> ### Gap 5: BRoster and Service Discovery
> 
> **What's missing:** How the roster worked — registration, app launching, app discovery — and how this maps to pane's ServiceId/DeclareInterest model.
> 
> **Content to add:**
> 
> BRoster was the system service registry:
> - Every BApplication registered with the registrar at startup (via `B_REG_ADD_APP` → `B_REG_COMPLETE_REGISTRATION`), providing its signature string, team_id, port_id, and entry_ref.
> - `BRoster::Launch(signature)` asked the registrar to find and launch an application by signature.
> - `BRoster::GetAppInfo(signature, &amp;info)` retrieved info about a running app.
> - `BRoster::GetAppList(signature, &amp;teams)` listed all running instances of a signature.
> - `BRoster::StartWatching(target, events)` / `StopWatching()` — get notified when apps launch/quit/activate.
> - The registrar maintained the authoritative roster (`TRoster` — 2068 LOC, the largest file in the registrar).
> - App signatures were MIME-style strings (`application/x-vnd.Be-MAIL`), stored as file attributes on the application binary.
> 
> Problems with this model:
> - String-based identity: signatures were just strings, manually assigned. No collision prevention, no structure.
> - The registrar was a single point of failure. If it crashed, the system was in trouble.
> - Launch logic was complex: pre-registration (before the app's BApplication constructor ran), flags (single/multiple launch, exclusive launch, background), finding the right binary for a signature.
> - Haiku added `launch_daemon` alongside the registrar (`B_REG_GET_CLIPBOARD_MESSENGER` still goes through registrar; service management moved to launch_daemon). This split created two overlapping service managers.
> 
> **Pane mapping:** pane replaces string-based signatures with `ServiceId` (UUID + reverse-DNS name). UUID is deterministic (derived from the name), preventing collisions. Service discovery uses `DeclareInterest` during session handshake — the compositor knows about all connected panes and their declared services. No separate registrar process needed because the compositor IS the registry. The `pane/functoriality_principle` memory explains why ServiceId has structure from day one rather than starting as bare strings.
> 
> ### Gap 6: Error Handling — status_t and InitCheck()
> 
> **What's missing:** BeOS's error handling patterns and what pane improves.
> 
> **Content to add:**
> 
> BeOS used two patterns:
> 1. **status_t returns:** Most functions returned `status_t` (an int32 typedef). `B_OK` (0) for success, negative values for errors (defined in `Errors.h`). Error codes were flat — no hierarchy, no domain separation.
> 2. **InitCheck() pattern:** Constructors couldn't fail in C++ (no exceptions in BeOS), so objects were constructed in a potentially-invalid state and you called `InitCheck()` to verify. `BFile file(path, B_READ_ONLY); if (file.InitCheck() != B_OK) ...`
> 
> What worked:
> - `status_t` was universal — every function used it, so error handling was consistent.
> - Error codes were well-defined and well-documented.
> - The convention was strong enough that developers checked return values (unlike Unix errno, which is easy to forget).
> 
> What didn't work:
> - `InitCheck()` was easy to forget. If you didn't call it, you had an invalid object that would silently misbehave.
> - Two-phase construction (construct then check) meant objects existed in an invalid state between construction and InitCheck.
> - Error codes were flat integers — no type safety, easy to accidentally compare error codes from different domains.
> - No error context — `B_ERROR` told you something failed but not what or why.
> 
> **Pane mapping:** Translation rule 6 in the existing `beapi_translation_rules` memory covers this: `Result&lt;T, E&gt;` replaces status_t/InitCheck(), failing at construction. But the deeper design lesson is that pane's three error channels (from the architecture spec) — `Result` for operation errors, `Drop` for obligation compensation, `Dispatch` for conversation-level failure — are a principled taxonomy that BeOS never had. BeOS had status_t for everything, which conflated "this operation failed" with "this peer crashed" with "this resource is gone."
> 
> ### Gap 7: The Two Kinds of Threading — Window Threads vs. Worker Threads
> 
> **What's missing:** The distinction between the system-managed threads (window threads, looper threads) and developer-spawned worker threads, and how pane collapses the former while preserving the need for the latter.
> 
> **Content to add:**
> 
> BeOS had two distinct categories of threads:
> 1. **System-managed looper threads:** Every BLooper spawned a thread that ran `_task0()` — the message loop. BWindow (a BLooper subclass) always had its own thread. BApplication had its own thread. These threads existed to process messages and were managed by the framework.
> 2. **Developer-spawned worker threads:** For heavy computation, network I/O, or any blocking operation. These communicated with looper threads via messaging (BMessenger::SendMessage).
> 
> The contract: looper threads must stay responsive (process messages quickly, never block). Worker threads do the heavy lifting and post results back to loopers.
> 
> The memory costs (Pierre Raynaud-Richard, Newsletter 4-46): a looper thread cost ~20.75 KB (thread stack + kernel structures + port + message queue). A window cost ~56 KB (two threads — client + server). This was the price of responsiveness.
> 
> **Pane mapping:** pane collapses the system-managed threads to one event loop per pane (calloop), achieving the same responsiveness guarantee without the per-looper thread overhead. The contract is the same — don't block the event loop — but enforced by session types (Handler methods are `&amp;mut self`, non-blocking). Worker threads are still the developer's responsibility for CPU-bound work, communicating back via `Messenger::send_message()` (which sends through the calloop channel). The key insight is that you don't need N threads for N-way responsiveness — you need N event sources feeding one event loop, which is exactly what calloop provides.

> 
> **Exploring approaches:**
> 
> - Propose 2-3 different approaches with trade-offs

> - Lead with your recommended option and explain why

> 
> - Once you believe you understand what you're building, present the design
> - Scale each section to its complexity: a few sentences if straightforward, up to 200-300 words if nuanced

> - Cover: architecture, components, data flow, error handling, testing

> 
> **Design for isolation and clarity:**
> 
> - Break the system into smaller units that each have one clear purpose, communicate through well-defined interfaces, and can be understood and tested independently
> - For each unit, you should be able to answer: what does it do, how do you use it, and what does it depend on?
> - Can someone understand what a unit does without reading its internals? Can you change the internals without breaking consumers? If not, the boundaries need work.
> - Smaller, well-bounded units are also easier for you to work with - you reason better about code you can hold in context at once, and your edits are more reliable when files are focused. When a file grows large, that's often a signal that it's doing too much.
> 
> **Working in existing codebases:**
> 
> - Explore the current structure before proposing changes. Follow existing patterns.
> - Where existing code has problems that affect the work (e.g., a file that's grown too large, unclear boundaries, tangled responsibilities), include targeted improvements as part of the design - the way a good developer improves code they're working in.
> - Don't propose unrelated refactoring. Stay focused on what serves the current goal.
