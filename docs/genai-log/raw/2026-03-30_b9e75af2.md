>

> ### 4. No Transactional Guarantees
> 
> The design doc mentions this briefly (Section 3, "partial mutation"). Be's scripting had no transactional semantics. A multi-specifier chain that failed midway left the system in a partially-modified state. There was no rollback.

> ## A. How does the DLfActRiS connectivity graph discipline affect the optic layer design?

> ### Inter-pane `DynOptic::set` and ownership transfer
> 
> The DLfActRiS review correctly identifies that when a `set` crosses actor boundaries, there's an ownership transfer event. But I want to challenge the framing slightly: **the optic layer itself should not carry the ownership transfer semantics. The message transport does.**
> 
> Here's why. Look at what actually happens in the current code when `send_and_wait` executes a scripting set:
> 
> 1. The caller constructs a `ScriptQuery` with `ScriptOp::Set(value)` â€” the value bytes are *moved* into the query.
> 2. The query is sent to the target's looper via `target_tx.send(LooperMessage::Request(msg, reply_port))`.
> 3. The target's handler receives it via `request_received`, resolves the chain, calls `optic.set(state, value)`.
> 4. The reply goes back through the `ReplyPort`.
> 
> The ownership transfer happens at step 2 â€” when the message crosses the channel. The value leaves the caller's ownership and enters the target's. > Caller <--[reply_rx]---- (waits)

> ## B. BeOS's scripting model and resource transfer semantics
> 
> ### ResolveSpecifier ownership story
> 
> Look at `Looper.cpp:1428-1466` again. The loop is:
> 
> ```cpp
> do {
>     target = newTarget;
>     newTarget = target->ResolveSpecifier(message, index, &specifier, form, property);
>     // check that new target is owned by looper

> 
> **For pane:** The resolve_chain function in the optics design doc already captures this correctly. The chain walk is ordinary `&mut self` method dispatch within the handler. No channel operations, no cross-thread dependencies. The connectivity graph is irrelevant here because there's nothing to connect.
> 
> ### Did hey create deadlockable relationships?
> 
> No, and understanding why is important.
> 
> The `hey` tool was an external process. It sent a single `BMessage` to the target application's port. The message carried the entire specifier chain (all specifiers packed into the message via `AddSpecifier`). The target application's looper received it, locked, ran `resolve_specifier`, dispatched to the resolved handler's `MessageReceived`, sent the reply back to `hey`, and unlocked.
> 
> From `hey`'s perspective: one synchronous `SendMessage(msg, &reply)`. One send, one receive. No intermediate forwarding. No temporary channels. The reply came back on a temporary port that Be created for the synchronous send (look at `Message.cpp:931` â€” `fHeader->reply_port` is a port created by the sender and embedded in the message header).
> 
> That temporary reply port is a one-shot. It exists only for the duration of the synchronous send. There is zero possibility of deadlock because:
> 1. `hey` is single-threaded â€” it sends and blocks.
> 2. The target looper processes the message to completion before reading the next one.
> 3. The reply goes back on the temporary port, which only `hey` reads.
> 
> The only way `hey` could deadlock is if the target application's handler, while processing the scripting message, tried to synchronously send back to `hey`. But `hey` isn't a BApplication â€” it doesn't have a looper or a port to receive on. So there's no channel for a callback.
> 
> **For pane:** `script_get`/`script_set` (wrappers around `send_and_wait`) have the same topology. The client blocks on a one-shot channel. The handler processes the query synchronously within its looper. The reply goes back on the one-shot. No cycles possible. The `WouldDeadlock` guard prevents the only bad case: a looper trying to `send_and_wait` to a target that might need to call back.
> 
> ### How BPropertyInfo avoided connectivity problems
> 
> `BPropertyInfo` is static data â€” it's a table of `property_info` structs populated at compile time (or object construction time). It doesn't create channels, doesn't send messages, doesn't establish any runtime relationships. It's the equivalent of a vtable for the scripting interface.
> 
> When `propertyInfo.FindMatch(message, index, specifier, what, property)` is called (Handler.cpp:474), it's a pure function: table lookup against the incoming specifier. No side effects, no I/O, no channel operations. The connectivity graph doesn't even apply.
> 
> The design insight here is that **property discovery is separated from property access**. `BPropertyInfo::FindMatch` tells you whether this handler handles this property. The actual access happens later in `MessageReceived`. By the time any state mutation occurs, the routing decision has already been made â€” purely from local data.
> 
> **For pane:** `ScriptableHandler::resolve_specifier` should have the same property: a pure lookup that returns `Resolution::Resolved(optic)` or `Resolution::NotFound`. No message sends, no channel operations, no side effects. The optic is a capability token â€” "this handler can access this property" â€” not an executor. Execution happens separately when the caller invokes `optic.get()` or `optic.set()`.
> 
> ## C. Concrete design implications for the optic layer
> 
> ### Should DynOptic distinguish observation from authority transfer?
> 
> No. I argued this above, but let me make it concrete with the typing.
> 
> The current `DynOptic` trait has:
> - `get(&self, state: &dyn Any) -> Result<AttrValue, ScriptError>` â€” shared reference
> - `set(&self, state: &mut dyn Any, value: AttrValue) -> Result<(), ScriptError>` â€” exclusive reference
> - `is_writable(&self) -> bool` â€” static capability query
> 
> The `&` vs `&mut` distinction on `state` already encodes the observation/mutation split at the Rust type level. A `get` call cannot mutate. A `set` call has exclusive access. This is *exactly* the resource transfer semantics that LinearActris would annotate with ownership assertions â€” but Rust enforces it at compile time rather than in a separation logic.
> 
> Adding a runtime "observation vs authority transfer" flag would be redundant with what the type system already guarantees. The only information that `DynOptic` could usefully carry beyond what Rust's type system provides is:
> 
> 1. **Side-effect scope:** Does this `set` only modify the focused field, or does it have broader effects? (E.g., setting `title` might trigger a compositor notification.) This is useful for optimizing change notification but has nothing to do with ownership transfer. I'd track it as a `ChangeScope` enum (`Local`, `Notifies(&[&str])`) rather than an ownership concept.
> 
> 2. **Atomicity:** Can this `set` fail partway through, leaving the state inconsistent? For simple field assignments, no. For traversal modifications (updating all items in a list), yes. This matters for the partial-mutation risk the design doc identifies. But again, not an ownership concern.
> 
> ### Should resolve_chain maintain a connectivity check?
> 
> No. Here's why.
> 
> The resolve_chain function operates within a single `&mut dyn ScriptableHandler`. It's not crossing any thread or process boundary. There are no channels involved. The connectivity graph concept doesn't apply to single-threaded method dispatch.
> 
> What resolve_chain *should* check is:
> 
> 1. **Chain length bound.** The specifier chain comes from the wire. A malicious or buggy client could send a specifier chain with 10,000 entries. The resolution loop should enforce a max depth (BeOS had no explicit limit, but the specifier stack in a BMessage was practically bounded by message size). Recommend a const `MAX_SPECIFIER_DEPTH = 16` â€” more than enough for any real query, prevents unbounded recursion.
> 
> 2. **No handler cycles.** If handler A's `resolve_specifier` returns a sub-handler that routes back to A, the loop is infinite. BeOS prevented this with the termination condition `newTarget != target` â€” if the resolved handler is the same as the current handler, the loop stops (that means the current handler claims ownership). Pane should do the same: if `resolve_specifier` returns `self` (by identity or by some tag), the chain is terminated, and the current step's optic is used for the final operation.
> 
> 3. **Same-pane constraint.** Like BeOS's `IndexOf(newTarget) < 0` check, pane should verify that sub-handlers don't escape the current pane. Since pane uses `&mut dyn ScriptableHandler` rather than raw pointers, this is naturally enforced â€” you can't return a reference to state you don't own. But if sub-handler navigation ever involves lookups (e.g., "child pane by name"), validate that the result is local.
> 
> None of these are connectivity checks in the LinearActris sense. They're standard loop-safety checks for a recursive resolution algorithm.
> 
> ### How does this affect CompletionReplyPort?
> 
> The `u64` token in the current design is fine, and I'd argue against replacing it with a typed `CompletionReplyPort` *for the optic layer*. Here's my reasoning.
> 
> The current `ReplyPort` (from `reply.rs`) already has the right session-type semantics:
> 
> ```rust
> pub struct ReplyPort {
>     token: u64,
>     reply_fn: Option<Box<dyn FnOnce(Message) + Send>>,
> }
> ```
> 
> The `Option<Box<dyn FnOnce>>` is the linear resource: it's consumed by `reply()` (which `take()`s it) or consumed by `Drop` (which sends `ReplyFailed`). Exactly one of these executes. Rust's ownership enforces this.
> 
> For scripting, the reply goes through the same mechanism. A scripting query arrives as `LooperMessage::Request(msg, reply_port)`. The handler resolves it via the optic chain, gets a result (or error), and calls `reply_port.reply(result)`. Done.
> 
> The `ScriptReplyToken` in the current `scripting.rs` stub duplicates `ReplyPort` unnecessarily. It should be replaced *by* `ReplyPort`, not by something new. The scripting query should arrive through `Handler::request_received` with a standard `ReplyPort`. The handler recognizes it as a scripting query (by its message variant), resolves it, and replies. No special-purpose reply type needed.
> 
> **Where a typed reply handle *would* help:** If the scripting response has a specific schema that differs from the general `Box<dyn Any + Send>`. For instance, if `script_get` always returns `AttrValue` and you want the type system to enforce that the handler replies with an `AttrValue`, not an arbitrary type. You could define:
> 
> ```rust
> pub struct ScriptReply(ReplyPort);
> 
> impl ScriptReply {
>     pub fn ok(self, value: AttrValue) { self.0.reply(ScriptResponse::Value(value)) }
>     pub fn error(self, err: ScriptError) { self.0.reply(ScriptResponse::Error(err)) }
> }
> ```
> 
> This is a newtype wrapper that restricts the reply to valid scripting responses. It composes with `ReplyPort` rather than replacing it. The Drop impl on `ReplyPort` still fires if `ScriptReply` is dropped without calling `ok()` or `error()`.
> 
> **The connectivity graph doesn't change this story.** Whether the reply is `ReplyPort` or `ScriptReply`, the ownership transfer is the same: the reply closure captures the response channel, the handler calls it once, the caller receives the response. One send, one receive, one-shot. Acyclic by construction.
> 
> ### Summary of concrete recommendations
> 
> 1. **Do not add ownership/authority annotations to `DynOptic`.** The `&` vs `&mut` distinction on state is sufficient. The observation/mutation split is already encoded in Rust's type system.
> 
> 2. **Do not add connectivity checks to `resolve_chain`.** Add chain-length bounds, cycle detection (same-handler check), and same-pane verification instead. These are safety checks, not connectivity checks.
> 
> 3. **Delete `ScriptReplyToken`.** Use `ReplyPort` directly for scripting replies. If you want schema enforcement, wrap it in a `ScriptReply` newtype that restricts the payload to `ScriptResponse` variants.
> 
> 4. **Keep the filesystem-mediated observer pattern.** It gives you decoupled connectivity for free. If you ever add direct (non-filesystem) watches, mandate async-only delivery (via `send_message`, never `send_and_wait`).
> 
> 5. **The debug-mode connectivity graph from the DLfActRiS review (recommendation #1) is worth building** â€” but as a diagnostic tool for inter-pane messaging, not for the optic layer. The optic layer is single-threaded; the connectivity concerns live at the transport layer. Track which panes have open `send_and_wait` calls, which panes have registered watchers, and check for blocking cycles. This is cheap (a small adjacency map updated on `send_and_wait` entry/exit) and would catch the deadlocks that `WouldDeadlock` can't (e.g., A's worker thread blocking on B while B's worker thread blocks on A).
> 
> 6. **The one LinearActris insight that genuinely changes the optic layer design:** The paper's channel fulfillment property validates pane's `ReplyPort::Drop` sending `ReplyFailed`. Generalize this to `ScriptReply`: if a handler drops a `ScriptReply` without responding, the caller gets a `ScriptResponse::Error(ScriptError::HandlerDroppedReply)`. This is the affine-to-linear upgrade for the scripting sub-protocol, and it's already the pattern in `reply.rs`. Just make sure it carries through to the scripting-specific wrapper.</result>
