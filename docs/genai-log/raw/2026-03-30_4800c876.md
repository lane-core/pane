> ## Scope Check
> 
> If the spec covers multiple independent subsystems, it should have been broken into sub-project specs during brainstorming. If it wasn't, suggest breaking this into separate plans — one per subsystem. Each plan should produce working, testable software on its own.
> 
> ## File Structure
> 
> Before defining tasks, map out which files will be created or modified and what each one is responsible for. This is where decomposition decisions get locked in.
> 
> - Design units with clear boundaries and well-defined interfaces. Each file should have one clear responsibility.
> - You reason best about code you can hold in context at once, and your edits are more reliable when files are focused. Prefer smaller, focused files over large ones that do too much.
> - Files that change together should live together. Split by responsibility, not by technical layer.
> - In existing codebases, follow established patterns. If the codebase uses large files, don't unilaterally restructure - but if a file you're modifying has grown unwieldy, including a split in the plan is reasonable.
> 
> This structure informs the task decomposition. Each task should produce self-contained changes that make sense independently.
>

> ## No Placeholders
> 
> Every step must contain the actual content an engineer needs. These are **plan failures** — never write them:

> - "Write tests for the above" (without actual test code)
> - "Similar to Task N" (repeat the code — the engineer may be reading tasks out of order)
> - Steps that describe what to do without showing how (code blocks required for code steps)
> - References to types, functions, or methods not defined in any task
> 
> ## Remember

> - Complete code in every step — if a step changes code, show the code
> - Exact commands with expected output
> - DRY, YAGNI, TDD, frequent commits

> ## 2. Key Design Contracts
> 
> ### BWindow
> 
> **Chrome is not the window's business.** BWindow knows it *has* a decorator (via window_look), but never draws chrome itself. The relationship is:
> - **Frame()** returns the content area in screen coordinates (where your BViews live).
> - **DecoratorFrame()** returns the total bounding rectangle including chrome. This is Haiku-era (not original BeOS R5 API).
> - **Bounds()** returns the content area in the window's own coordinate system (origin at top-left of content).
> 
> The contract: the window owns the content rect, the decorator owns everything else. App_server draws the decorator; the client draws within Bounds(). This three-rect model (decorator frame > frame > bounds) is exactly what your `DecorGeometry { frame, tag, content }` already captures.
> 
> **Look/Feel separation.** `window_look` governs chrome appearance (B_TITLED_WINDOW_LOOK, B_FLOATING_WINDOW_LOOK, B_NO_BORDER_WINDOW_LOOK, etc.). `window_feel` governs stacking behavior (normal, modal-subset, modal-app, modal-all, floating-subset, floating-app, floating-all). These are orthogonal and can be set independently via `SetLook()`/`SetFeel()`. This maps well to Wayland's xdg_toplevel + compositor-side policy.
> 
> **Subset windows.** The subset system (AddToSubset/RemoveFromSubset) creates window groups where floating/modal windows track visibility of their parent. This is a non-trivial compositor-side concept -- Wayland has no direct analog. If pane needs modal panes, this needs design.
> 
> **Update batching.** Three mechanisms with different semantics:
> 1. **DisableUpdates()/EnableUpdates()** -- suppresses all drawing. "If you want the results of several drawing operations to appear in the window all at once."
> 2. **BeginViewTransaction()/EndViewTransaction()** (Haiku-era) -- messages are sent but not processed. More fine-grained than DisableUpdates.
> 3. **Flush()/Sync()** -- Flush sends buffered commands; Sync sends AND waits for completion.
> 
> **UpdateIfNeeded()** -- Forces immediate synchronous redraw of all invalid views. The docs explicitly say "This method is ignored unless it is called from within the message loop of the thread that the BWindow is running in." This is the thread-safety contract: only the window's own thread can trigger draw.
> 
> ### BView
> 
> **B_WILL_DRAW is the opt-in.** A view without this flag gets no Draw() calls. This is a significant optimization -- the view hierarchy can be deep, but only views that declare they draw get update processing.
> 
> **B_FULL_UPDATE_ON_RESIZE** causes the entire view to be invalidated on resize, not just the exposed strip. Without it, only the newly exposed area triggers Draw(). This matters for pane's cell grid -- when a pane resizes, do we redraw the entire cell grid or just the new cells?
> 
> **Draw(updateRect) is called by the system, never by you.** To trigger a redraw, call Invalidate(). The system coalesces invalidations and calls Draw() with the union updateRect. The docs explicitly say: "This is a hook method called by the Interface Kit, you don't have to call it yourself. If you need to forcefully redraw the view consider calling Invalidate() instead."
> 
> **DrawAfterChildren(updateRect)** draws *on top of* child views. Requires B_DRAW_ON_CHILDREN flag. This is for overlay effects -- borders, watermarks. In pane's context, this would be analogous to the compositor drawing chrome over client content.
> 
> **Graphics state is a stack.** PushState/PopState save/restore a complete graphics context: origins, scales, clipping regions, drawing mode, pen, font, colors, line cap/join, miter limit, stipple pattern. The new state after PushState starts with local scale=0, local origin=(0,0), no clipping. This stack model influenced how Haiku's app_server manages per-view rendering state.
> 
> **DelayedInvalidate** coalesces: "Repeated calls to this method may be merged, but the view is guaranteed to be redrawn after the delay given in the first call." This is the damage-coalescing pattern that pane already implements via dirty flags.
> 
> **BeginLayer/EndLayer** creates compositing layers with per-layer opacity. Added for WebKit support. Nests. This is relevant if pane ever needs semi-transparent overlays on client content.
> 
> ### BLooper (threading foundation)
> 
> **One thread per looper, one looper per window.** BWindow inherits BLooper. Run() spawns a thread. Every window message -- keyboard, mouse, draw, script -- goes through this one thread's message loop. The lock (Lock/Unlock) is recursive and protects all internal data.
> 
> **The lock is the synchronization primitive.** The docs repeatedly warn: "Because a looper usually is used in multiple threads, you should make sure you Lock() and Unlock() it during most operations." Every hook method is called with the lock held. This is why BeOS windows feel responsive -- the lock is held only during message processing, not during I/O waits.
> 
> **Port capacity matters.** Loopers use kernel ports. If the port is full, incoming messages are DROPPED, not queued. The default capacity (B_LOOPER_PORT_DEFAULT_CAPACITY) is adequate for most cases, but high-message-rate scenarios need tuning. This is analogous to pane-session's channel capacity.
>

> ## 5. Pitfalls the Docs Warn About
> 
> 1. **Bounds() in draw loops.** Issue 1-12 newsletter (Benoit Schillings) explicitly warns: "A common mistake is to use the Bounds() method of BView inside a drawing loop each time you need the bounds of the view. This is a perfect case for caching, since the bounds won't change while drawing." For pane: cache DecorGeometry per frame, don't recompute per-pane-per-cell.
> 
> 2. **Synchronous calls kill throughput.** The newsletter lists synchronous calls (StringWidth, Bounds, GetMouse, DrawBitmap, GetClippingRegion, ConvertToScreen). In pane's model this isn't a concern because there's no draw protocol -- clients send finished content. But it reinforces why SetContent is the right approach: sending drawing commands would create the same synchronous-call bottleneck Be had.
> 
> 3. **Many views = slow.** Issue 1-12: "Many views cause poor performance; fewer views containing simple user views are often faster." The clipping region computation for deep view hierarchies is expensive. Pane avoids this entirely -- there are no compositor-side view trees. Each pane is one flat surface.
> 
> 4. **Show/Hide are cumulative.** "Calls to Hide() and Show() are cumulative." Two Hide() calls require two Show() calls. This is a reference-counting pattern. Pane should decide if hide/show is boolean or refcounted.
> 
> 5. **Looper must be created on the heap.** "loopers should always be created on the heap (with new), and never on the stack" because Quit() deletes self. Irrelevant to pane (Rust ownership handles this), but the self-deletion pattern is worth noting -- pane's PaneState cleanup on client disconnect is the analog.
> 
> 6. **First Show() starts the message loop.** The BWindow message loop doesn't start until Show() is called. Before that, the window exists but is inert. This is why Show() also unlocks the looper. Pane's analog: PaneCreated sends geometry, but the pane isn't rendered until the compositor's next frame. No explicit "show" message is needed because pane rendering is compositor-driven.
> 
> 7. **B_TRANSPARENT_COLOR for view color.** "If you set the view color to B_TRANSPARENT_COLOR then the Application Server won't erase the clipping region of the view before updating, this should only be used if the view erases itself by drawing on every pixel." This is the opt-out of background clearing. Pane doesn't have background clearing -- clients send complete content buffers.
>

> ## 6. What Pane Should Borrow vs. Adapt
> 
> ### Borrow directly
> 
> - **Three-rect model.** DecoratorFrame > Frame > Bounds maps perfectly to pane's `DecorGeometry { frame, tag, content }`. The plan already has this right.
> 
> - **Look/Feel separation.** window_look (chrome appearance) and window_feel (stacking behavior) should remain orthogonal. The Decorator trait handles look; the compositor's stacking/focus policy handles feel. Don't conflate them.
> 
> - **Decorator as geometry owner.** The decorator computes the footprint; content is what's left. The plan's `Decorator::geometry()` method correctly captures this.
> 
> - **Hit testing as decorator responsibility.** `RegionAt(BPoint)` in Haiku's Decorator maps to `hit_test()` in the plan. The decorator knows where the close button is, where the borders are.
> 
> - **Dirty flag coalescing.** DelayedInvalidate's merge behavior. PaneState's `title_dirty` / `content_dirty` flags are the right analog.
> 
> ### Adapt
> 
> - **No server-side view tree.** BeOS has a full view hierarchy replicated in app_server. Pane has none. Client content is a flat buffer (SetContent), not a tree of drawing commands. This is a fundamental simplification -- and the right one for a compositor. The view tree was necessary in BeOS because the server did the drawing; in pane, clients do their own rendering and hand off finished content.
> 
> - **No drawing protocol.** BView's Draw() model sends drawing commands (StrokeLine, FillRect, DrawString, etc.) from client to app_server. Pane replaces this with content snapshots. This eliminates the buffered-vs-synchronous tension that Benoit wrote about in Issue 1-12. The tradeoff: clients must render their own content (no server-side font rendering, no server-side clipping). This is the Wayland model and it's correct.
> 
> - **Decorator renders chrome, not content.** In Haiku, the decorator draws chrome and then the server triggers client content redraw. In pane, the compositor renders chrome (via DefaultDecorator geometry + texture upload) AND composites client content (from SetContent). There's no "trigger client to redraw" step -- the client pushes content asynchronously.
> 
> - **No PushState/PopState in the compositor.** The graphics state stack is a client-side drawing concept. Pane's compositor renders chrome with a simple immediate-mode approach (fill rects, blit text textures). No state stack needed.
> 
> - **Flush/Sync replaced by damage tracking.** Instead of "flush my buffered commands" / "wait for server to finish", pane uses dirty flags + frame-based rendering. Content arrives when it arrives; the compositor renders what it has at each frame. This is fundamentally different from BeOS's synchronous update model but achieves the same user-visible result: content appears promptly.
> 
> - **Window subset model needs design work.** BeOS's AddToSubset/RemoveFromSubset for modal/floating window groups has no Wayland equivalent. If pane needs modal panes (e.g., a dialog that blocks interaction with a specific other pane), this needs explicit protocol design. The Wayland xdg_popup and grab mechanisms are the closest analog but much more limited.
> 
> ### Consciously omit
> 
> - **B_OUTLINE_RESIZE.** Draws only the outline during resize. This was a performance hack for the era. Modern compositing makes it unnecessary.
> 
> - **Window alignment (SetWindowAlignment).** Byte-alignment and pixel-alignment for frame buffer offsets. This was for direct framebuffer access. Irrelevant under Wayland.
> 
> - **BPicture recording/playback.** Server-side display lists. Replaced by client-side rendering.
>
