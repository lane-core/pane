> ## COMPREHENSIVE ANALYSIS: PANE-APP RUNTIME ARCHITECTURE
> 
> ### Executive Summary
> 
> **Headless Capability: YES, with architectural caveats.** pane-app can theoretically run without a display server, but it fundamentally requires a running **pane compositor** process. There are NO hard dependencies on Wayland, X11, or GPU rendering within pane-app itself‚Äîthose concerns are isolated to pane-comp (the compositor). However, pane-app cannot function as a standalone application; it must communicate with a compositor via a Unix socket protocol.
> 
> ---
> 
> ### 1. ARCHITECTURE OVERVIEW
> 
> #### Core Design Philosophy
> pane-app is modeled on BeOS's Application Kit (`BApplication`, `BLooper`, `BHandler`, `BMessenger`), redesigned for Rust with **session-typed channels** for protocol safety. The kit provides:
> 
> - **App** ‚Äî Connection factory and dispatcher (not a looper itself)
> - **Pane** ‚Äî Configuration wrapper consumed by run/run_with (equivalent to BWindow)
> - **Looper** ‚Äî Per-pane event loop running on its own thread
> - **Handler** ‚Äî Trait-based event dispatch (trait methods replace single MessageReceived)
> - **Messenger** ‚Äî Cloneable thread-boundary crossing handle
> 
> #### Key Architectural Layer Separation
> 
> ```
> pane-app (application kit) ‚Üê Unix socket ‚Üí pane-proto (wire types)
>                              ‚Üê pane-session (session-typed channels)
>                                           ‚Üì
>                              pane-comp (compositor)
>                                  ‚Üì
>                              smithay (rendering, Wayland, X11)
> ```
> 
> **Critical point:** pane-app has ZERO dependencies on rendering, windowing, or display servers. All such concerns live in pane-comp.
> 
> ---
> 
> ### 2. APPLICATION LIFECYCLE & THREADING MODEL
> 
> #### App Lifecycle (from app.rs)
> 
> ```rust
> pub struct App {
>     comp_tx: mpsc::Sender<ClientToComp>,
>     pending_creates: Arc<Mutex<VecDeque<mpsc::Sender<CreateResponse>>>>,
>     pane_count: Arc<AtomicUsize>,
>     done_signal: Arc<(Mutex<()>, std::sync::Condvar)>,
>     _dispatcher: thread::JoinHandle<()>,  // Key: spawned immediately
> }
> ```
> 
> 1. **Connection Phase**: `App::connect(signature)` 
>    - Connects to Unix socket: `$XDG_RUNTIME_DIR/pane/compositor.sock`
>    - Runs session-typed handshake with compositor
>    - Reuses socket for active phase
> 
> 2. **Dispatcher Thread** (spawned in `connect_test`, line 120):
>    - Reads `CompToClient` messages from compositor
>    - Routes to correct pane's channel
>    - Pre-registers channels on `PaneCreated` to avoid races
>    - Exits when connection closes
> 
> 3. **Pane Threads** (spawned per `Pane::run()` call):
>    - Each pane gets its own message loop thread
>    - Sequential processing: one message at a time
>    - No shared mutable state between loopers (borrow checker)
> 
> 4. **App::run()** waits via condition variable until all panes close.
> 
> #### Threading Model Summary
> - **Single App** ‚Üí **N loopers** (one per pane)
> - **One dispatcher** routes compositor messages
> - **Worker threads** post results back via `Messenger::post_app_message()`
> - **No BApplication-level looper** (unlike BeOS)
>

> ### 3. MESSAGE LOOP SYSTEM (looper.rs)
> 
> The looper is the heart of the event system. Each pane runs `run_closure` or `run_handler`:
> 
> ```rust
> pub fn run_closure(
>     pane_id: PaneId,
>     receiver: mpsc::Receiver<LooperMessage>,
>     mut filters: FilterChain,
>     proxy: Messenger,
>     mut handler: impl FnMut(&Messenger, Message) -> Result<bool>,
> ) -> Result<ExitReason>
> ```
> 
> #### Message Sources (Unified Channel)
> 
> The looper receives from a **single `mpsc::Receiver<LooperMessage>`**:
> 
> ```rust
> pub enum LooperMessage {
>     FromComp(CompToClient),           // Compositor events
>     Posted(Message),                   // Self-delivered from workers
>     AddTimer { make_event, interval }, // Timer registration
>     AddOneShot { event, fire_at },     // Delayed event
>     Request(Message, ReplyPort),       // Inter-pane request
> }
> ```
> 
> #### Event Processing Pipeline
> 
> 1. **Receive with timeout** (for timer integration):
>    ```rust
>    fn recv_with_timers(receiver: &Receiver, timers: &Timers) 
>        ‚Üí Option<LooperMessage>
>    ```
>    - Returns `None` on timeout (fires timers)
>    - Returns `Some(msg)` on received message
>    - Returns `Err` on disconnect
> 
> 2. **Drain and coalesce** (batch optimization):
>    - Pulls all pending messages from channel at once
>    - **Coalesces resize/mouse-move** events (keep only latest)
>    - **Truncates batch after Close** to avoid input floods
> 
> 3. **Timer firing**:
>    - Sorted vector of timer entries
>    - Periodic timers: factory closure produces fresh event each fire
>    - One-shot timers: consumed on fire, no cloning needed
>    - Cancelled flag checked per-iteration (atomic bool)
> 
> 4. **Filter chain**:
>    - Applied per-message before dispatch
>    - Can drop or transform messages
>    - Shortcuts implemented as a filter
> 
> 5. **Dispatch**:
>    - **Closure form**: `handler(&proxy, msg) ‚Üí Result<bool>`
>    - **Handler trait form**: Dispatches to `handler.ready()`, `handler.key()`, etc.
>    - Return `Ok(false)` to exit, `Err` for error exit
> 
> #### Exit Conditions
> 
> ```rust
> pub enum ExitReason {
>     HandlerExit,    // Handler returned Ok(false) (send RequestClose)
>     CompositorClose, // CloseRequested received (don't send RequestClose)
>     Disconnected,    // Channel closed or I/O error
> }
> ```
> 
> #### Disconnect Handling
> 
> On channel close (receiver end of dispatcher thread closes):
> 1. Calls `handler.disconnected(&proxy)`
> 2. Returns `ExitReason::Disconnected`
> 3. Broadcasts `Message::PaneExited` to monitored loopers
>

> ### 5. MESSENGER ARCHITECTURE (proxy.rs)
> 
> The Messenger is the thread-boundary crossing point:
> 
> ```rust
> pub struct Messenger {
>     id: PaneId,
>     sender: mpsc::Sender<ClientToComp>,       // To compositor
>     looper_tx: Option<mpsc::SyncSender<LooperMessage>>, // To own looper
>     broadcaster: ExitBroadcaster,             // Monitor/notify
>     pulse_token: Arc<Mutex<Option<TimerToken>>>,
> }
> ```
> 
> #### Capabilities
> 
> 1. **Outbound to Compositor** (sends `ClientToComp`):
>    - `set_title()`, `set_vocabulary()`, `set_content()`
>    - `resize_to()`, `set_size_limits()`, `set_hidden()`
> 
> 2. **Self-Delivery to Looper**:
>    - `send_message(event)` ‚Äî blocks if channel full
>    - `try_send_message(event)` ‚Äî non-blocking variant
>    - `post_app_message::<T>(msg)` ‚Äî for worker thread results
> 
> 3. **Inter-Pane Communication**:
>    - `send_and_wait(target, msg, timeout)` ‚Äî **blocks looper thread** (danger!)
>    - `send_request(target, msg)` ‚Äî async, reply arrives later
>    - Deadlock detection: `if is_looper_thread() { return WouldDeadlock }`
> 
> 4. **Monitoring** (Erlang-style crash propagation):
>    - `monitor(watcher)` ‚Äî registers watcher's looper_tx with broadcaster
>    - On exit: non-blocking try_send to all watchers (advisory, not guaranteed)
> 
> 5. **Timers**:
>    - `send_delayed(event, delay)` ‚Äî one-shot via `AddOneShot` message

> ### 11. BApplication/BLooper/BHandler HEADLESS EQUIVALENCE
> 
> #### BeOS (Original)
> 
> ```
> BApplication (looper + factory)
>   ‚îú‚îÄ BLooper (main loop runs on app thread)
>   ‚îî‚îÄ BWindow (looper subclass)
>        ‚îî‚îÄ BHandler (event dispatch)
> ```
> 
> **Single looper at app level** ‚Äî all windows post to the same message queue.
> 
> #### Pane (Redesigned)
> 
> ```
> App (factory + dispatcher, NOT a looper)
>   ‚îú‚îÄ Dispatcher thread (routes compositor messages)
>   ‚îî‚îÄ Pane (per-window configuration)
>        ‚îî‚îÄ Looper (one per pane, on separate thread)
>             ‚îî‚îÄ Handler trait (event dispatch)
> ```
> 
> **Multiple loopers** ‚Äî each pane gets its own sequential loop.
> 
> #### Headless Implications
> 
> **BeOS**: If you hide the app from the display server, `BApplication` still runs. Losing the display server would cause port/messaging issues, but the looper is independent.
> 
> **Pane**: pane-app is **already more headless-friendly**:
> - No app-level looper dependency on display
> - Dispatcher thread only talks to compositor (decoupled)
> - Pane loopers are pure message loops (no graphics calls)
> 
> **Barrier**: pane-app requires a compositor process, while BeOS apps could theoretically run without a display server (though nothing useful would happen).

> ### 3. Serialization & Transport Abstraction
> 
> #### Wire Protocol
> - **Serialization:** postcard (compact binary format with variable-length integers)
> - **Framing:** 4-byte little-endian length prefix + payload
> - **Max message size:** 16 MB (prevents unbounded allocation on corrupt length prefixes)

> #### Transport Trait (from pane-session)
> ```rust
> pub trait Transport: Sized {
>     fn send_raw(&mut self, data: &[u8]) -> Result<(), SessionError>;
>     fn recv_raw(&mut self) -> Result<Vec<u8>, SessionError>;
> }
> ```
> 
> **Current Implementations:**
> 1. **MemoryTransport** (`pane-session/transport/memory.rs`)
>    - Backed by `std::sync::mpsc`
>    - For testing without sockets
>    - Fully typed, crash-safe

>    - Uses framing helpers from `pane_session::framing`
>    - Production IPC on a single machine

> **Key Design:** The `Transport` trait is minimal and completely orthogonal to the protocol. Adding TCP, TLS, SSH tunneling, or any other transport requires:
> 1. Implementing `Transport` for that mechanism
> 2. Creating `Chan<S, NewTransport>` ‚Äî the protocol logic is unchanged

> ### 4. Network Transparency Assessment
> 
> #### Current State: Highly Ready for Distribution
> 
> The protocol is **already network-transparent** in the architectural sense:
> 
> 1. **No Serialization Impedance:** Every message is already serialized (via postcard) before being passed to the transport. The framing layer is generic (`Vec<u8>` ‚Üí wire format ‚Üí `Vec<u8>`).
> 
> 2. **Transport-Agnostic Session Types:** The handshake and active-phase protocols are defined purely in terms of session type combinators (`Send`, `Recv`, `Branch`). They don't depend on how the transport delivers bytes.
> 
> 3. **No Timestamp Coupling:** `KeyEvent` and `MouseEvent` timestamps are optional (`Option<u64>` in microseconds since epoch), allowing remote clients to use local timestamps or omit them entirely.
> 
> 4. **Geometry Decouples Client/Server Rendering:** `PaneGeometry` contains logical dimensions (pixel width/height, cell cols/rows) but no screen-space coordinates, color depth, or visual state. The client and compositor can run on different machines with different hardware and render independently.
> 
> 5. **Content Opaque:** `SetContent { pane, content: Vec<u8> }` carries raw bytes ‚Äî the format is "negotiated during handshake" (current code says this but doesn't implement it yet; capability negotiation is in place for future expansion).
> 
> #### What Would Be Needed for True Remote Operation
> 
> Very little, because the architecture is already clean:
> 
> 1. **Add a Network Transport**
>    ```rust
>    pub struct TcpTransport {
>        stream: TcpStream,
>    }
>    impl Transport for TcpTransport { ... }
>    ```
> 
> 2. **Address Negotiation** (if needed)
>    - Currently: `pane-app` connects to `$XDG_RUNTIME_DIR/pane/compositor.sock`
>    - For remote: extend `ClientCaps` to declare network location if needed, or use environment variables
>    - The handshake already supports arbitrary capability negotiation
> 
> 3. **Encryption** (optional but wise)
>    - Wrap `TcpTransport` in a `TlsTransport` or `SshTransport`
>    - The session type layer neither knows nor cares
> 
> 4. **Content Format Negotiation** (future)
>    - Extend `ClientCaps` to declare supported content serialization formats
>    - Both `SetContent` and `SetVocabulary` could use different encodings per session
> 
> #### NOT Required
> - No protocol restructuring
> - No schema versioning (postcard variant handling exists but isn't needed yet due to co-deployment)
> - No centralized registry (panes are identified by `PaneId` allocated by the compositor)
> - No additional framing (length-prefixed messages already exist)

> ### 8. Design Strengths for Distribution
> 
> 1. **Explicit Session Types:**
>    - Handshake protocol is enforced by types, not convention
>    - Reduces protocol version mismatches
>    - Enables compile-time verification of correctness

>    - `Transport` trait is minimal (2 methods)
>    - New transports are straightforward
>    - Protocol logic never touches transport details

>    - `ClientCaps` already exists for future extension

>    - Input events are portable (timestamps optional)

>    - `recv()` returns `Err(SessionError::Disconnected)` not panic

> 
> 1. **Version Negotiation:**
>    - Handshake carries `version: u32` but no backward compatibility logic yet

>    - The design cleanly supports TLS wrapping but doesn't require it

> 
> ### Content Outline
> 
> **11.1 Network Transparency.** The Transport trait is the extension point. `TcpTransport` + TLS makes every session-typed interaction network-transparent. The same `Chan<S, T>` works over unix sockets and TCP -- the session type enforces protocol correctness regardless of transport.
> 
> **11.2 Headless Instances.** A pane server without a compositor. Runs the full pane protocol -- clients connect, create panes, exchange messages -- but no rendering occurs. Headless instances serve AI agent workloads, batch processing, remote services. They are full pane citizens: they participate in roster federation, host pane-fs namespaces, run applications.
> 
> **11.3 Identity Model.** Unix identity carries across the network. Remote connections declare `PeerIdentity` (username, uid, hostname) during handshake. TLS client certificates bind identity to cryptographic keys. The server maps remote identity to a local unix user for permission checks. Landlock enforcement scopes to the mapped user's capabilities. No new auth scheme -- the `.plan` file governs agent behavior regardless of whether the agent is local or remote.
> 
> **11.4 Federation Model (Phase 2 Spec).** When pane-roster exists: instances register with each other. Service discovery spans instances. A client on instance A can discover and connect to a service on instance B via pane-roster federation. The roster maintains a peer table of known instances, their capabilities, and their reachability.
> 
> **11.5 Network pane-fs Mounting (Phase 2 Spec).** When pane-fs is built: remote `/pane/` namespaces mount transparently under `/pane/remote/<hostname>/`. A script on the local machine can read `/pane/remote/cloud-1/by-sig/com.pane.agent/1/attrs/status` to check a remote agent's state. The FUSE layer translates file operations into pane protocol messages forwarded over TcpTransport.
> 
> **11.6 Network-Aware .plan Enforcement (Phase 2 Spec).** When the AI kit is built: Landlock policies specified in `.plan` apply to network-originated requests. A remote agent's `.plan` specifies `network: allow outbound 443` -- Landlock + network namespaces enforce this at the kernel level on the instance running the agent. The `.plan` is the single source of truth for agent permissions, whether the agent runs locally or remotely.

> **Risk: TLS adds complexity to the handshake.** Mitigation: TLS establishment happens before the session-typed handshake.
