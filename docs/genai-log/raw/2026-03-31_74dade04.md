> # Brainstorming Ideas Into Designs
> 
> Help turn ideas into fully formed designs and specs through natural collaborative dialogue.
> 
> Start by understanding the current project context, then ask questions one at a time to refine the idea. Once you understand what you're building, present the design and get user approval.
> 
> <HARD-GATE>
> Do NOT invoke any implementation skill, write any code, scaffold any project, or take any implementation action until you have presented a design and the user has approved it. This applies to EVERY project regardless of perceived simplicity.
> </HARD-GATE>
> 
> ## Anti-Pattern: "This Is Too Simple To Need A Design"
> 
> Every project goes through this process. A todo list, a single-function utility, a config change — all of them. "Simple" projects are where unexamined assumptions cause the most wasted work. The design can be short (a few sentences for truly simple projects), but you MUST present it and get approval.
> 
> ## Checklist
> 
> You MUST create a task for each of these items and complete them in order:
> 
> 1. **Explore project context** — check files, docs, recent commits
> 2. **Offer visual companion** (if topic will involve visual questions) — this is its own message, not combined with a clarifying question. See the Visual Companion section below.
> 3. **Ask clarifying questions** — one at a time, understand purpose/constraints/success criteria
> 4. **Propose 2-3 approaches** — with trade-offs and your recommendation
> 5. **Present design** — in sections scaled to their complexity, get user approval after each section
> 6. **Write design doc** — save to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` and commit
> 7. **Spec self-review** — quick inline check for placeholders, contradictions, ambiguity, scope (see below)
> 8. **User reviews written spec** — ask user to review the spec file before proceeding
> 9. **Transition to implementation** — invoke writing-plans skill to create implementation plan
>

> ## The Process
> 
> **Understanding the idea:**
> 
> - Check out the current project state first (files, docs, recent commits)
> - Before asking detailed questions, assess scope: if the request describes multiple independent subsystems (e.g., "build a platform with chat, file storage, billing, and analytics"), flag this immediately. Don't spend questions refining details of a project that needs to be decomposed first.
> - If the project is too large for a single spec, help the user decompose into sub-projects: what are the independent pieces, how do they relate, what order should they be built? Then brainstorm the first sub-project through the normal design flow. Each sub-project gets its own spec → plan → implementation cycle.
> - For appropriately-scoped projects, ask questions one at a time to refine the idea
> - Prefer multiple choice questions when possible, but open-ended is fine too
> - Only one question per message - if a topic needs more exploration, break it into multiple questions
> - Focus on understanding: purpose, constraints, success criteria
> 
> **Exploring approaches:**
> 
> - Propose 2-3 different approaches with trade-offs
> - Present options conversationally with your recommendation and reasoning
> - Lead with your recommended option and explain why
> 
> **Presenting the design:**
> 
> - Once you believe you understand what you're building, present the design
> - Scale each section to its complexity: a few sentences if straightforward, up to 200-300 words if nuanced
> - Ask after each section whether it looks right so far
> - Cover: architecture, components, data flow, error handling, testing
> - Be ready to go back and clarify if something doesn't make sense
> 
> **Design for isolation and clarity:**
> 
> - Break the system into smaller units that each have one clear purpose, communicate through well-defined interfaces, and can be understood and tested independently
> - For each unit, you should be able to answer: what does it do, how do you use it, and what does it depend on?
> - Can someone understand what a unit does without reading its internals? Can you change the internals without breaking consumers? If not, the boundaries need work.
> - Smaller, well-bounded units are also easier for you to work with - you reason better about code you can hold in context at once, and your edits are more reliable when files are focused. When a file grows large, that's often a signal that it's doing too much.
> 
> **Working in existing codebases:**
> 
> - Explore the current structure before proposing changes. Follow existing patterns.
> - Where existing code has problems that affect the work (e.g., a file that's grown too large, unclear boundaries, tangled responsibilities), include targeted improvements as part of the design - the way a good developer improves code they're working in.
> - Don't propose unrelated refactoring. Stay focused on what serves the current goal.
> 
> ## After the Design
> 
> **Documentation:**
> 
> - Write the validated design (spec) to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
>   - (User preferences for spec location override this default)
> - Use elements-of-style:writing-clearly-and-concisely skill if available
> - Commit the design document to git
> 
> **Spec Self-Review:**
> After writing the spec document, look at it with fresh eyes:
> 
> 1. **Placeholder scan:** Any "TBD", "TODO", incomplete sections, or vague requirements? Fix them.
> 2. **Internal consistency:** Do any sections contradict each other? Does the architecture match the feature descriptions?
> 3. **Scope check:** Is this focused enough for a single implementation plan, or does it need decomposition?
> 4. **Ambiguity check:** Could any requirement be interpreted two different ways? If so, pick one and make it explicit.
> 
> Fix any issues inline. No need to re-review — just fix and move on.
> 
> **User Review Gate:**
> After the spec review loop passes, ask the user to review the written spec before proceeding:
> 
> > "Spec written and committed to `<path>`. Please review it and let me know if you want to make any changes before we start writing out the implementation plan."
> 
> Wait for the user's response. If they request changes, make them and re-run the spec review loop. Only proceed once the user approves.
> 
> **Implementation:**
> 
> - Invoke the writing-plans skill to create a detailed implementation plan
> - Do NOT invoke any other skill. writing-plans is the next step.
> 
> ## Key Principles
> 
> - **One question at a time** - Don't overwhelm with multiple questions
> - **Multiple choice preferred** - Easier to answer than open-ended when possible
> - **YAGNI ruthlessly** - Remove unnecessary features from all designs
> - **Explore alternatives** - Always propose 2-3 approaches before settling
> - **Incremental validation** - Present design, get approval before moving on
> - **Be flexible** - Go back and clarify when something doesn't make sense
> 
> ## Visual Companion
> 
> A browser-based companion for showing mockups, diagrams, and visual options during brainstorming. Available as a tool — not a mode. Accepting the companion means it's available for questions that benefit from visual treatment; it does NOT mean every question goes through the browser.
> 
> **Offering the companion:** When you anticipate that upcoming questions will involve visual content (mockups, layouts, diagrams), offer it once for consent:
> > "Some of what we're working on might be easier to explain if I can show it to you in a web browser. I can put together mockups, diagrams, comparisons, and other visuals as we go. This feature is still new and can be token-intensive. Want to try it? (Requires opening a local URL)"
> 
> **This offer MUST be its own message.** Do not combine it with clarifying questions, context summaries, or any other content. The message should contain ONLY the offer above and nothing else. Wait for the user's response before continuing. If they decline, proceed with text-only brainstorming.
> 
> **Per-question decision:** Even after the user accepts, decide FOR EACH QUESTION whether to use the browser or the terminal. The test: **would the user understand this better by seeing it than reading it?**
> 
> - **Use the browser** for content that IS visual — mockups, wireframes, layout comparisons, architecture diagrams, side-by-side visual designs
> - **Use the terminal** for content that is text — requirements questions, conceptual choices, tradeoff lists, A/B/C/D text options, scope decisions
> 
> A question about a UI topic is not automatically a visual question. "What does personality mean in this context?" is a conceptual question — use the terminal. "Which wizard layout works better?" is a visual question — use the browser.
>

> ### Sensitive data in transit through the filesystem
>

> ## 2. Security as Protocol Concern
> 
> ### 2a. TTL and protocol expectations
> 
> **TTL does NOT violate protocol expectations, but it introduces a distinction that must be visible.**
> 
> Consider the protocol interaction:
> 1. Writer locks clipboard, writes sensitive data with TTL=30s, commits.
> 2. Reader reads clipboard at t=5s. Gets data. Fine.
> 3. Reader reads clipboard at t=35s. Gets empty. Is this a protocol violation?
> 
> No. The clipboard protocol as designed already has `Option<Data>` semantics — you can read the clipboard and find it empty (no one has written to it, or the last writer cleared it). TTL-expiry is semantically identical to "the service cleared it." The reader's protocol type is `Recv<Option<ClipboardData>, End>`, not `Recv<ClipboardData, End>`. An empty clipboard is a valid state.
> 
> **What matters:** The event file must emit a `ClipboardCleared { reason: TtlExpired }` event when TTL fires. Without this, a watcher has no way to distinguish "data was cleared by TTL" from "data was cleared by a new writer." The reason tag is metadata; the protocol event is what makes the temporal state change observable. This is the same principle as pane-notify's `AttrCause` — you can see *why* state changed, not just *that* it changed.
> 
> ### 2b. Sensitivity in the session-typed write protocol
> 
> **Sensitivity is metadata on the write, not a separate protocol step.** The protocol is:
> 
> ```
> Lock -> Write(data, metadata) -> Commit
> ```
> 
> Where `metadata` includes sensitivity, TTL, content type, etc. Making sensitivity a separate method call (`set_sensitivity()` between lock and commit) adds protocol states without adding protocol safety. The service doesn't care about the order of `write_data` vs `set_sensitivity` — both are mutations to the pending-write state that get committed atomically.
> 
> Compare with ReplyPort: the payload is a single `reply(message)` call, not a multi-step build-then-send. The clipboard write should follow the same pattern: one `commit(data, metadata)` call that carries everything.
> 
> In the typestate handle design:
> 
> ```rust
> impl ClipboardWriteLock {
>     /// Write data and commit in one step.
>     /// Consumes the lock handle (like ReplyPort::reply).
>     pub fn commit(self, data: Vec<u8>, metadata: ClipboardMetadata) { ... }
>     
>     /// Revert without writing.
>     /// Explicit revert; Drop also reverts (affine gap compensation).
>     pub fn revert(self) { ... }
> }
> ```
> 
> `ClipboardMetadata` carries sensitivity, TTL, content type, locality constraints. This keeps the typestate linear: `Lock -> Commit | Revert | Drop`.
> 
> ### 2c. Access control: Branch vs. error
> 
> **Branch is the correct modeling.** The access check is a protocol-level decision that changes the conversation's continuation type.
> 
> ```
> LockClipboard("private") -> Branch<
>     Left: Recv<ClipboardWriteLock, ...>,   // Granted
>     Right: Recv<AccessDenied, End>          // Denied
> >
> ```
> 
> This is not an error in the same sense as "malformed message" or "service unavailable." It's a protocol branch — the service decides which path to take based on the `.plan` policy, and the client must handle both cases. Using `Result<ClipboardWriteLock, AccessDenied>` at the kit API level is fine (it's the Rust surface for Branch), but the protocol-level structure is a branch, not an exception.
> 
> The distinction matters for the filesystem interface: a write to `/pane/clipboard/private/ctl` that gets denied should return EACCES, not EAGAIN. The filesystem error code maps from the protocol branch: `Left` -> success, `Right(AccessDenied)` -> EACCES. This is a clean mapping because POSIX errno codes are already a (crude) Branch type.
> 
> Compare with PaneCreateFuture, which is `Recv<Created | Refused, End>` — same pattern. The creation can be refused; the client handles both branches.

> ## 3. Undo and the Optic Recording Layer
> 
> ### 3a. Sensitive data in the undo stack
> 
> **This is a real information leak.** If `RecordingOptic::set()` captures `(old_value, new_value)` unconditionally, then:
> 
> - Clearing a password field records `(old="hunter2", new="")` in the undo stack.

> 
> **Yes, if descriptions are auto-generated from property names.** If the `RecordingOptic` generates descriptions like "Set {property_name} to {truncated_value}" or even just "Set {property_name}", the property name itself may be sensitive context. >

> ### 3c. Sensitive edit marker
> 
> **This is the right design.** Extend `DynOptic` (or the `RecordingOptic` wrapper) with a sensitivity flag:
> 
> ```rust
> pub trait DynOptic: Send + Sync {
>     // ... existing methods ...
>     
>     /// Whether edits to this property should be recorded for undo.
>     /// Default: true. Override to false for sensitive properties
>     /// (passwords, tokens, private data).
>     fn is_undoable(&self) -> bool { true }
> }
> ```
> 
> When `is_undoable()` returns false, `RecordingOptic` skips recording. The edit happens (the underlying optic's `set()` is called), but no `(old_value, new_value)` is pushed to the undo stack.
> 
> **Consequence:** undo is impossible for that edit. The undo stack has a gap. If the user does: edit A, edit B (sensitive), edit C, then undo — they get back to the state after edit B, not before it. This is correct: the alternative (storing sensitive data in the undo stack) is worse.
> 
> The description in `attrs/undo-description` should skip the sensitive edit entirely — the undo stack never knew about it.
> 
> **Alternative (if you need undo for sensitive edits):** Store the edit as a `SensitiveEdit { undo_fn: Box<dyn FnOnce(&mut State)> }` that captures a closure but not the raw values. The closure can be invoked for undo but cannot be inspected, serialized, or exposed through pane-fs. This is more complex but preserves undo capability. The trade-off: the undo stack now contains closures that hold references to state, which complicates serialization and the pane's fault domain (a panic in the undo closure kills the pane). This is probably not worth it — the simpler "sensitive edits are not undoable" is the right default.
> 
> **The UndoPolicy trait should have this as part of its interface:**
> 
> ```rust
> pub trait UndoPolicy {
>     fn should_record(&self, optic: &dyn DynOptic) -> bool {
>         optic.is_undoable()
>     }
>     // ... push, pop, group, etc.
> }
> ```
> 
> This lets policies override — e.g., an "audit everything" policy that records even sensitive edits (for forensics) could override `should_record` to always return true. But the default respects the optic's sensitivity annotation.
> 
> The optic law preservation argument from my earlier analysis (Clarke et al. Proposition 2.3 — composition preserves laws) still holds: `RecordingOptic` is still transparent to the underlying optic. > ## 4. Cross-Cutting: The Two-Interface Problem
> 
> ### 4a. View or full peer?
> 
> **The filesystem interface is a protocol client, not a peer.** This is already established by pane-fs.md: "pane-fs is a translation layer — it converts FUSE operations into pane protocol messages." The filesystem interface cannot do anything the protocol can't do — it's a thin syntax transformation.
> 
> But the filesystem interface is a *degraded* client: it cannot hold typestate handles. A protocol client holds a `ClipboardWriteLock` handle that enforces lock -> write -> commit ordering at compile time. pane-fs cannot hold typestate handles because FUSE operations are stateless — each `read()` or `write()` is independent.
> 
> This means the filesystem interface must either:
> 
> **Option A: Atomic operations only.** Each filesystem operation maps to one complete protocol interaction. Write to `ctl` with `set <data>` does lock+write+commit atomically. The service handles the multi-step protocol internally on behalf of the filesystem client. The filesystem client never holds a lock across operations.
> 
> **Option B: Session-scoped ctl files.** Opening `/pane/clipboard/system/ctl` creates a pane-fs-internal session with the clipboard service. Subsequent writes to the fd are commands within that session. Closing the fd ends the session (drop = revert, like `ClipboardWriteLock`). This gives the filesystem interface transactional semantics, but ties the protocol session to the fd lifecycle.
> 
> Option A is simpler and covers 99% of use cases. Option B is more powerful but introduces the complexity of fd-scoped sessions in pane-fs. I'd recommend Option A as the default with Option B deferred until there's a concrete use case for multi-step filesystem clipboard writes.
> 
> ### 4b. Preventing invariant violations
> 
> With the architecture as designed, the filesystem interface **cannot** violate invariants that the protocol enforces, because it IS a protocol client. Every filesystem operation becomes a protocol message. The service serializes all messages. The service enforces all invariants.
> 
> The specific invariant that the typestate protocol enforces — ordering — is enforced differently by the two interfaces:
> - Protocol client: compile-time via typestate handles (`ClipboardWriteLock` consumed by `commit(self)`)
> - Filesystem client: runtime via the service's lock state (write-to-locked-clipboard returns EBUSY)
> 
> The static guarantee is lost for the filesystem interface. But the safety property (no data corruption, no invalid state) is preserved by the service's serialization. This is the affine gap analysis applied to the two-interface problem: the filesystem drops from "protocol correctness by construction" to "protocol correctness by runtime enforcement." Compensation is sufficient for safety; liveness is weaker (a filesystem client that opens a ctl session and hangs without committing holds the lock until fd timeout).
> 
> ### 4c. Does service serialization eliminate the two-interface problem?
> 
> **Yes, for safety. No, for liveness.** The service as single source of truth means both interfaces see consistent state. No data races, no torn writes, no phantom reads. The service's looper thread processes one message at a time — this is exactly the BLooper actor guarantee.
> 
> The liveness concern: a misbehaving filesystem client can hold a clipboard lock indefinitely (if using Option B above, where the fd lifecycle controls the lock). The protocol client has the same problem (a pane that acquires a lock and never commits). The mitigation for both: **lock timeout at the service level.** After N seconds without a commit, the service forcibly reverts and releases the lock. This is standard for cooperative lock protocols and is independent of the access interface.
> 
> For the undo case: the filesystem interface to undo (`ctl` commands `undo`, `redo`) maps 1:1 to handler-local undo operations. The handler processes the ctl command's protocol message, invokes `UndoPolicy::undo()`, and the state changes. The `attrs/undo-count` file reflects the new state on next read. No two-interface problem here — undo is a single atomic operation, not a multi-step protocol.

> 
> The single port was a deliberate choice, but for reasons that don't apply to pane. >

> - `fClientReplyPort` — the server sends synchronous replies here
> - `fClientLooperPort` — the server sends events to the BLooper's port
>

> 
> For pane, Rust mpsc channels have near-zero creation cost compared to kernel ports. The economic argument for single-port doesn't apply. >

George Hoffman warned in Newsletter Issue 2-36 that if a window thread blocks, "the app_server will start to throw away update messages."
>

> 
> ### 3. Cross-Protocol Ordering: Don't Reconstruct It
> 
> Some Be apps did depend on cross-protocol ordering (e.g., `B_WINDOW_ACTIVATED` arriving before `B_KEY_DOWN`). This was an emergent property of the single port, not a documented guarantee.
>

The orderings that matter — activated before key input, ready before anything — are same-protocol (both are compositor events on the compositor channel). The orderings that don't matter — clipboard notification vs. mouse move — are cross-protocol.
>
