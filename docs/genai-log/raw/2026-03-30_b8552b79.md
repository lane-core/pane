> ## 2. Threading/Locking Considerations
> 
> The Haiku Book is explicit about several threading contracts that pane handles differently:
> 
> **A. "The looper needs to be locked"** -- The Book warns that `AddFilter()`, `RemoveFilter()`, `SetFilterList()`, `SetNextHandler()` all require the associated looper to be locked. Failure to lock causes crashes. In Haiku's implementation, this is enforced by `debugger()` calls (see `SetNextHandler` at line 316 of Handler.cpp: `debugger("handler must belong to looper before setting NextHandler")`).
> 
> Pane sidesteps this entirely because filters are owned by the looper and immutable after construction, and there is no handler chain. The `Handler` trait requires `Send + 'static` and the handler is moved into the looper thread, so there's no shared access to worry about. This is correct but **undocumented** -- the Handler doc comment should state that the handler is moved into the looper thread and cannot be accessed from other threads (only through `Messenger`).
> 
> **B. `LockLooper()` / `UnlockLooper()`** -- In Be, any time you wanted to touch a handler from outside its looper thread, you locked the looper first. Pane replaces this with `Messenger::send_message()`, which posts to the channel. The cross-thread access pattern is fundamentally different: you don't lock and mutate, you send a message. This is documented in the Messenger comments but could be stated more explicitly on Handler.
> 
> **C. `filter_hook` could retarget** -- The `BHandler** target` out-parameter in the filter hook and `Filter()` method allowed filters to redirect messages to a different handler within the same looper. Since pane has one handler per looper (per pane), this capability doesn't apply and isn't needed. But if pane ever has views-within-panes (sub-handlers), this would need to be revisited.

> ## 3. Behavioral Contracts
> 
> Several behavioral contracts from the Haiku Book that pane may diverge from:
> 
> **A. Default `MessageReceived` replies `B_MESSAGE_NOT_UNDERSTOOD`**
> 
> In Haiku (Handler.cpp lines 277-282), if no handler in the chain processes a message, *and* it was dropped or has specifiers, the base `BHandler::MessageReceived` prints a warning and sends `B_MESSAGE_NOT_UNDERSTOOD` as a reply. This is a contract: senders can expect a reply for unhandled messages.
> 
> Pane's `fallback()` returns `Ok(true)` (silently continue). There is no reply mechanism for unhandled messages. **This matters for scripting** -- when scripting arrives, unrecognized script messages need to produce error replies. For now it's fine but should be documented as a future consideration.
> 
> **B. Handler owns filters, destructor frees them**
> 
> The Book says: "Free the filters of this handler, as well as the list of observers." In pane, `FilterChain` owns its boxed filters and drops them when the looper exits. This is fine -- Rust's ownership handles it. But there's no equivalent of the warning that "This destructor does not remove the handler from the looper" because in pane the handler *is* the looper's sole handler and dies with it.
> 
> **C. Observer notification message mutation**

The implementation is correct: it keeps only the last resize and last mouse-move while preserving order of other events. This is **well-documented and faithful**.

> **E. `close_requested` return convention inversion**

> Documented in the Handler trait: Be returned `true` to accept close, pane returns `Ok(false)` to stop the loop. The inversion is noted but could trip up developers familiar with BeOS. The doc comment is clear about this.

> ## 4. The Handler Chain
> 
> This is the most significant structural departure from Be, and it deserves explicit documentation.
> 
> **What Be had:**
> - A BLooper contains a list of BHandlers.
> - Each BHandler has a `fNextHandler` pointer forming a chain.
> - When `MessageReceived` is called on a handler and it doesn't handle the message, it calls `fNextHandler->MessageReceived(message)` (after applying the next handler's filters).
> - The chain walks until someone handles it, or the end is reached.
> - BLooper itself is a BHandler and is always the last handler in the chain.
> - `BLooper::_TopLevelFilter` applies common filters first, then `_HandlerFilter` walks handler-specific filters for the target, potentially retargeting.
> 
> **What pane has:**
> - One handler per pane. No chain. No NextHandler.
> - The Looper dispatches directly to the single handler's typed methods.
> - `fallback()` is the catch-all but there's nowhere else for the message to go.
> 
> **Assessment:** This is an **intentional and correct** simplification. In Be, the handler chain existed because a BWindow contained multiple BViews, each a BHandler. The chain was the view hierarchy's message routing. Pane doesn't have views -- each pane is a flat surface. If pane ever introduces sub-pane views (a view hierarchy within a pane), the chain would need to come back, probably as a tree rather than a linked list.
> 
> **The skip is not documented.** The Handler trait's doc comment mentions it descends from BHandler but doesn't explain what was left out or why. A "BeOS divergences" section should note:
> - No handler chain (pane has one handler per pane, no view hierarchy)
> - No `NextHandler` / handler chain walking
> - No handler-to-looper association API (the relationship is implicit and 1:1)
> 
> ## 5. Observer Pattern (StartWatching / StopWatching / SendNotices)
> 
> **What Be had:**
> 
> BHandler doubles as a state machine with observer pattern. The API surface is substantial:
> - `StartWatching(BHandler*, uint32)` -- local same-looper observation
> - `StartWatching(BMessenger, uint32)` -- cross-team (cross-process) observation  
> - `StartWatchingAll(BHandler*)` / `StartWatchingAll(BMessenger)` -- watch all states
> - `StopWatching` / `StopWatchingAll` variants
> - `SendNotices(uint32, const BMessage*)` -- emit state change to observers
> - `IsWatched()` -- check if anyone is watching
> 
> The implementation (Handler.cpp) maintains an `ObserverList` with separate maps for handler-pointers (local) and messengers (remote/cross-team). When `SendNotices` fires, it sends `B_OBSERVER_NOTICE_CHANGE` messages to all registered observers for that `what` value *plus* all wildcard (`B_OBSERVER_OBSERVE_ALL`) observers.
> 
> **What pane has:**
> 
> `Messenger::monitor()` -- Erlang-style exit monitoring. One pane can monitor another and receive `Message::PaneExited { pane, reason }` when it dies. This is crash propagation, not state observation.
> 
> **Assessment:** These are **different mechanisms solving different problems**.
> 
> - Be's observer pattern is about **state change notification within a running system** -- "tell me when the network status changes," "tell me when the selection changes." It's the same concept as KVO, Qt signals, or Rx observables.
> - Pane's monitoring is about **lifecycle events** -- "tell me when this pane dies."
> 
> Pane has no equivalent of the state-change observer pattern. This is a **gap worth flagging** for the future. When pane has applications that want to coordinate state (e.g., a preferences pane notifying all windows that a setting changed), the options are:
> 1. Build an observer system on Handler, similar to Be's
> 2. Use the messaging system directly (handlers post to interested parties via Messenger)
> 3. Use a shared-state mechanism outside the message loop (filesystem attributes, like the BFS approach)
> 
> Option 2 is already possible with pane's `Messenger::send_message()`. Option 3 aligns with pane's filesystem-as-database philosophy. Option 1 is probably not needed if 2 and 3 are well-documented. But this should be a **conscious decision, not an omission by neglect**.
> 
> ## 6. Filter Specifics: `wants()` vs. Delivery/Source Enums
> 
> **What Be had:**
> 
> Three orthogonal filter criteria, checked in order (see Looper.cpp `_ApplyFilters`, line 1362-1402):
> 1. **Command match** -- `FiltersAnyCommand()` or `Command() == message->what`
> 2. **Delivery method** -- `message_delivery` enum:
>    - `B_ANY_DELIVERY` -- accept all
>    - `B_DROPPED_DELIVERY` -- only drag-and-drop
>    - `B_PROGRAMMED_DELIVERY` -- only `PostMessage()`
> 3. **Source** -- `message_source` enum:
>    - `B_ANY_SOURCE` -- accept all
>    - `B_REMOTE_SOURCE` -- from other applications
>    - `B_LOCAL_SOURCE` -- from same application
> 
> After these three pass, *then* the `filter_hook` (if set) or `Filter()` (if overridden) runs. So the built-in criteria are a **pre-filter** before the custom logic.
> 
> Additionally, the `filter_hook` and `Filter()` have a `BHandler** target` out-parameter that lets them **retarget** the message to a different handler.
> 
> **What pane has:**
> 
> `wants(&self, &Message) -> bool` -- a single predicate that replaces all three built-in criteria. It examines the event type/variant instead of delivery method and source.
> 
> `filter(&mut self, event: Message) -> FilterAction` -- returns `Pass(Message)` or `Consume`. Can transform the message but **cannot retarget**.

For pane-store's attribute indexing, knowing *which* xattr changed and *how* (created vs removed vs modified) is operationally important — without it, every `Attrib` event requires reading all xattrs and diffing against cached state.
> 
> **Recommendation:** `Attrib` should carry the attribute name and change cause when the kernel provides it. On inotify, you don't get the attr name, but on fanotify with `FAN_ATTRIB`, you at least know it happened. If pane-store needs per-attribute granularity, the fanotify backend should open-code a read of xattr names before/after, or the indexer should accept the re-read cost. Either way, the `Event` struct should have a field for it even if it's `Option<String>`.
> 
> **B_DEVICE_MOUNTED / B_DEVICE_UNMOUNTED.** These are completely absent from pane-notify. In Haiku, `B_WATCH_MOUNT` lets any listener know when volumes appear or disappear. On Linux, the equivalent would be watching `/proc/mounts` or using `fanotify` with mount notifications (Linux 5.1+). This matters for pane-store (new volumes need indexing) and for any Tracker-like file manager.
> 
> **Recommendation:** Add `DeviceMounted` and `DeviceUnmounted` variants to `EventKind`, and a `watch_mounts()` method on `Watcher`. Implementation can monitor `/proc/mounts` via inotify or poll it.
> 
> ### 2. Watch Flags
> 
> | Haiku flag | pane-notify equivalent | Status |
> |---|---|---|
> | `B_WATCH_NAME` | Implicit in `watch_path` with `Create`/`Delete`/`MovedFrom`/`MovedTo` | **Implicit** |
> | `B_WATCH_STAT` | Implicit in `watch_path` with `Modify` | **Implicit** |
> | `B_WATCH_ATTR` | Implicit in `watch_path` with `Attrib` | **Implicit** |
> | `B_WATCH_DIRECTORY` | `watch_path` on a directory | **Implicit** |
> | `B_WATCH_ALL` | Passing all `EventKind` variants | **Implicit** |
> | `B_WATCH_MOUNT` | — | **Missing** |
> | `B_WATCH_INTERIM_STAT` | — | **Missing** |
> | `B_WATCH_CHILDREN` (Haiku extension) | — | **Missing** |
> | `B_WATCH_RECURSIVELY` (BPathMonitor) | — | **Missing** |
> | `B_STOP_WATCHING` | Drop `WatchHandle` (RAII) | **Covered, better** |
> 
> The fundamental design difference: Haiku's flags are a bitmask combined in a single `watch_node()` call. pane-notify takes an `&[EventKind]` slice. This works for the basic cases but has a structural problem — **the watch flags and the event kinds are conflated**.
> 
> In Haiku, `B_WATCH_NAME` means "tell me when the *name* of a node changes" — this produces `B_ENTRY_CREATED`, `B_ENTRY_REMOVED`, and `B_ENTRY_MOVED` notifications. `B_WATCH_STAT` means "tell me when stat data changes" — producing `B_STAT_CHANGED`. `B_WATCH_ATTR` means "tell me when attributes change" — producing `B_ATTR_CHANGED`. The flags select *categories of interest*; the opcodes describe *what happened*.
> 
> pane-notify collapses these two layers: you subscribe to specific `EventKind` variants, which are both the subscription filter and the notification type. This works for simple cases but breaks down for anything more complex:
> - You can't express "watch this directory for name changes only" vs "watch this directory for stat changes only" as cleanly.
> - `B_WATCH_ALL` = `B_WATCH_NAME | B_WATCH_STAT | B_WATCH_ATTR | B_WATCH_DIRECTORY` — note that `B_WATCH_MOUNT` is explicitly *not* included. This intentional design choice is lost when flags and events are the same enum.
> 
> **Recommendation:** This is actually fine for now. The `&[EventKind]` filter model maps well enough to inotify's mask. But if you ever need watch-scope flags that don't map 1:1 to event kinds (like `B_WATCH_INTERIM_STAT`, which coalesces but doesn't change the event type, or `B_WATCH_RECURSIVELY`, which changes scope not event kind), you'll need to introduce a separate `WatchFlags` type. Consider doing it early.
> 
> ### 3. Live Queries
> 
> This is the big architectural question. In Haiku, live queries are deeply coupled with node monitoring at the kernel level. From the kernel source (`node_monitor.cpp` lines 1047-1365), there are dedicated functions: `notify_query_entry_created()`, `notify_query_entry_removed()`, `notify_query_entry_moved()`, `notify_query_attribute_changed()`. These send `B_QUERY_UPDATE` messages (a distinct message code from `B_NODE_MONITOR`) directly to the query's target messenger.
> 
> The coupling works like this:
> 1. `BQuery::SetTarget(messenger)` makes the query "live" — it tells the kernel to keep the query open.
> 2. `BQuery::Fetch()` returns the initial result set.
> 3. The filesystem layer calls `notify_query_entry_*()` when an entry enters or leaves the query's result set.
> 4. The listener gets `B_QUERY_UPDATE` messages with `B_ENTRY_CREATED` (entry now matches) or `B_ENTRY_REMOVED` (entry no longer matches) opcodes.
> 
> From the Newsletter Issue 3-29 (the detailed node monitoring article), the interaction is subtle: you typically need *both* a live query *and* node monitoring on the returned entries, because the query tells you about set membership changes but node monitoring tells you about changes to entries you're already tracking.
> 
> **pane-notify's compatibility:** The current design is compatible in the sense that it doesn't block this, but it doesn't help either. pane-notify only handles the "node monitoring" half. For live queries in pane-store, you'd need:
> - A query engine in pane-store that evaluates predicates against the xattr index
> - A mechanism for pane-store to fire "query update" events when an entry enters/leaves a query's result set
> - Those events would need their own channel or enum variant
> 
> Since pane doesn't control the filesystem, live queries can't be kernel-mediated. They'd be a pane-store feature, built on top of pane-notify's raw events. pane-store watches for `Attrib` events, re-evaluates affected queries, and pushes updates to subscribers. This is the right architecture for userspace.
> 
> **Recommendation:** Add a `QueryUpdate { entered: bool }` variant to `EventKind`, or define a separate `QueryEvent` type in pane-store. The consumer needs to distinguish "this entry's attributes changed" from "this entry now matches/no-longer-matches your query." Don't try to combine these — they're different information channels, as Haiku's separate `B_NODE_MONITOR` vs `B_QUERY_UPDATE` message codes reflect.
> 
> ### 4. Message Fields
> 
> Here's what Haiku's notification messages carry, compared to pane-notify's `Event` struct:
> 
> **B_ENTRY_CREATED / B_ENTRY_REMOVED:**
> ```
> Haiku:  opcode, device, directory (ino_t), node (ino_t), name
> pane:   kind (EventKind), path (PathBuf)
> ```
> 
> **B_ENTRY_MOVED:**
> ```
> Haiku:  opcode, device, from_directory, to_directory, node_device, node, from_name, name
> pane:   kind (MovedFrom or MovedTo), path (PathBuf)
> ```
> 
> **B_STAT_CHANGED:**
> ```
> Haiku:  opcode, device, node, fields (bitfield of what changed)
> pane:   kind (Modify), path (PathBuf)
> ```
> 
> **B_ATTR_CHANGED:**
> ```
> Haiku:  opcode, device, directory, node, attr (name), cause (created/removed/changed)
> pane:   kind (Attrib), path (PathBuf)
> ```
> 
> **B_DEVICE_MOUNTED:**
> ```
> Haiku:  opcode, new_device, device (parent), directory (parent mount point)
> pane:   (not represented)
> ```
> 
> **B_DEVICE_UNMOUNTED:**
> ```
> Haiku:  opcode, device
> pane:   (not represented)
> ```
> 
> The pattern is clear: **pane-notify's `Event` struct is radically underpopulated.** Haiku messages carry structured data about *what* changed, *where* (in inode space), and *how* (cause/fields). pane-notify has a path and a kind.
> 
> This matters practically. Three examples:
> 
> 1. **Attribute indexing (pane-store):** When `Attrib` fires, pane-store needs to know *which* attribute changed and *whether it was created, removed, or modified*. Without this, every `Attrib` event requires a full xattr re-read and diff.
> 
> 2. **File manager (Tracker equivalent):** When displaying a directory, a `B_STAT_CHANGED` with `fields = B_STAT_SIZE` means "update the size column." A `B_STAT_CHANGED` with `fields = B_STAT_MODE` means "update the permissions icon." Without the `fields` discriminant, every stat change requires a full `stat()` call and UI diff.
> 
> 3. **Move tracking:** Haiku's `B_ENTRY_MOVED` gives you both endpoints and the node identity. pane-notify's split `MovedFrom`/`MovedTo` doesn't carry the node identity, so the consumer can't even tell if two events refer to the same move without the missing cookie field.
> 
> **Recommendation:** Enrich the `Event` struct. At minimum:
> 
> ```rust
> pub struct Event {
>     pub kind: EventKind,
>     pub path: PathBuf,
>     /// For MovedFrom/MovedTo: cookie linking the pair. From inotify.
>     pub cookie: Option<u32>,
>     /// For Attrib: which attribute changed (if known).
>     pub attr_name: Option<String>,
>     /// For Attrib: created, removed, or modified (if known).
>     pub attr_cause: Option<AttrCause>,
> }
> ```
> 
> Alternatively, make `EventKind` carry its own payload:
> 
> ```rust
> pub enum EventKind {
>     Create,
>     Delete,
>     Modify,
>     Attrib { name: Option<String>, cause: Option<AttrCause> },
>     Moved { from: PathBuf, to: PathBuf },
> }
> ```
>

> ### 5. Per-Path vs Per-Node Watching
> 
> This is the deepest design question, and the Haiku docs are quite clear about the tradeoff.
> 
> **Haiku watches by `node_ref` (device + inode).** From `Node.dox`: "A BNode is concerned solely with the entry's data and attributes" — it's unaware of names. The `node_ref` struct (device, node) is a stable identity that survives renames, moves across directories on the same volume, and hard link creation. When you `watch_node(&nodeRef, ...)`, you're watching the *data*, not the *name*.
> 
> **pane-notify watches by path.** Both `watch_path()` and `watch_mount()` take paths. This maps directly to inotify's API, which also watches by path (or more precisely, by the inode *at the time of the watch registration*, resolved from the path — but the inotify watch descriptor doesn't track if the node is moved).
> 
> **What this means in practice:**
> 
> 1. **Rename resilience:** In Haiku, if you watch node N and someone renames the file from `foo` to `bar`, you still get notifications because you're watching the node, not the name. With inotify/pane-notify, if someone moves the file to a different directory, your inotify watch on the original directory won't see events on the file anymore. You'd need to re-establish the watch.
> 
> 2. **Hard links:** Haiku's node-based watching naturally handles hard links — all names for the same node are equivalent. inotify watches specific directory entries.
> 
> 3. **Mount points:** Haiku's `node_ref` includes the device ID, making cross-volume distinctions natural. pane-notify's path-based model doesn't distinguish.
> 
> 4. **Haiku knew this was a pain point.** The `BPathMonitor` class (in `headers/private/storage/PathMonitor.h` — a private API, never made public!) exists precisely because path-based watching is what applications *actually want* most of the time. It's built *on top of* node monitoring: it resolves the path to a `node_ref`, watches that, and also watches all ancestor directories for renames that would change the path. The `B_WATCH_RECURSIVELY` flag is a BPathMonitor feature, not a kernel feature.
> 
> **pane-notify's position is actually the right default.** Path-based watching is what applications want. The Haiku team implicitly admitted this by creating BPathMonitor. Node-based watching is the correct *primitive* when you control the kernel, but pane doesn't control the kernel. inotify gives you path-to-fd watching; fanotify gives you mount-wide or filesystem-wide watching. Both are path-oriented at the API level.
> 
> **The real gap** is that pane-notify doesn't handle the problems that BPathMonitor was designed to solve:
> - **Recursive watching** is not supported. The doc says "not recursive — watch subdirectories individually if needed." BPathMonitor has `B_WATCH_RECURSIVELY`.
> - **Ancestor monitoring** is not implemented. If you watch `/foo/bar/baz` and someone renames `bar` to `quux`, your watch silently goes stale. BPathMonitor re-establishes watches when ancestor directories change.
> - **Re-establishment after move** is not handled. When a watched file is moved, the consumer gets `MovedFrom` but has no mechanism to automatically re-watch at the new location.
> 
> **Recommendation:** The path-based model is correct for pane. Don't try to introduce `node_ref` — it's a kernel concept and you're in userspace. But you need to decide whether pane-notify handles the reliability problems (ancestor monitoring, re-establishment) itself, or punts to consumers. BPathMonitor's existence proves that every consumer re-implementing this logic is a bad outcome. Build it into pane-notify, at least as an opt-in `watch_path_stable()` method that monitors ancestors.
>

> ## 2. BMessage Features vs Pane's Typed Enum
> 
> This is the most fundamental architectural divergence, and it's deliberate. Let me lay out what's different and why it matters.
> 
> ### What BMessage had that the enum doesn't:
> 
> **Arbitrary typed data fields (AddData/FindData/ReplaceData):**
> BMessage was a dynamically-typed key-value container. You could attach arbitrary data to any message: `msg->AddString("name", "hello")`, `msg->AddInt32("count", 42)`, `msg->AddRect("frame", rect)`. This made messages extensible at runtime -- you could add fields without changing the type definition.
> 
> Pane's `Message` enum has typed payloads: `Message::Key(KeyEvent)`, `Message::Resize(PaneGeometry)`. The data is the variant itself. There's no mechanism to attach extra data to an event.
> 
> **Assessment:** This is the right tradeoff for compositor events. You know at compile time exactly what data a resize event carries. The typed enum eliminates the entire class of "forgot to check if the field exists" bugs that plagued BMessage handlers. However, pane currently has **no mechanism for application-defined messages**. BMessage's `what` field was a `uint32` -- applications could define their own message types and pack arbitrary data into them. In pane, `Message` is a closed enum. You can't add variants.
> 
> The `Messenger::send_message(Message)` API lets you post any `Message` variant to the looper, but you can only post existing variants. There is no `Message::Custom { what: u32, data: HashMap<String, Value> }` or equivalent.
> 
> **This is a real gap.** Applications need to post custom events to themselves. A network thread finishes a fetch and wants to tell the UI. Currently, the only options are:
> - Abuse an existing variant (bad)
> - Use `Message::CommandExecuted` with magic strings (worse)
> - Restructure to avoid self-posting (not always possible)
> 
> **Recommendation:** Add a `Message::App(Box<dyn Any + Send>)` or a `Message::App { id: u64, payload: Vec<u8> }` variant for application-defined events. This is the one place where BMessage's dynamic typing solved a real problem that the typed enum currently doesn't.
> 
> **Specifiers (HasSpecifiers, AddSpecifier, GetCurrentSpecifier, PopSpecifier):**
> These were the scripting protocol's addressing mechanism. A specifier chain like `["Window", 0, "View", "status", "Text"]` navigated a hierarchy of scriptable objects. This is the `hey` protocol.
> 
> **Assessment:** pane has designed a separate optics-based scripting mechanism (per earlier memory). The specifier system was specific to BMessage's dynamic structure. pane doesn't need specifiers *in Message* because scripting goes through a different channel. Correct architectural decision.
> 
> **Reply chain (SendReply, ReturnAddress, Previous, IsReply):**
> BMessage had a built-in request-reply correlation mechanism. You could reply to any message, and the system tracked the conversation.
> 
> **Assessment:** pane's protocol is unidirectional by event type -- the compositor sends events, the client sends commands. There's no request-reply pattern in the active phase. For cases that need it (completion requests), pane uses explicit `token` fields for correlation. This works, but if the protocol grows, you'll want a general request-reply mechanism. **Monitor this.** CompletionRequest/CompletionResponse is already an ad-hoc version of what BMessage's reply system provided generically.
> 
> **Delivery metadata (WasDelivered, IsSourceWaiting, IsSourceRemote, WasDropped, DropPoint):**
> BMessage tracked *how* a message arrived: was it posted, was it dropped (drag-and-drop), was it from another app?
> 
> **Assessment:** Most of this is N/A. pane events come from exactly two sources: compositor or self-posted. There's no drag-and-drop yet. But when drag-and-drop lands, pane will need a `Message::Drop` variant with position data. **Flag for future.**
> 
> **Flattening (Flatten/Unflatten):**
> BMessage could serialize to/from byte streams. This was used for clipboard, drag-and-drop, file storage (settings), and IPC.
> 
> **Assessment:** pane's protocol types already derive `Serialize`/`Deserialize` via serde. The `Message` enum itself doesn't, but it could trivially. The clipboard and settings use cases will need serialization, but they'll likely use different types, not `Message` itself. **Not a gap -- different mechanism, same capability when needed.**
> 
> ### What the typed enum does better:
> 
> - **Exhaustive matching.** You cannot forget to handle a message type. If a new variant is added, existing code fails to compile until it's addressed.
> - **No runtime type errors.** `FindInt32("count")` could return `B_NAME_NOT_FOUND` or `B_BAD_TYPE`. Enum variant destructuring cannot fail.
> - **Documentation is the type.** You can read `Message` and see every possible event. BMessage required documentation to know which fields to expect with which `what` codes.

> ## 5. Coalescing
> 
> ### What BWindow did:
> 
> Per the Haiku docs and the Be Newsletter articles, BWindow's `DispatchMessage()` coalesced:
> - **B_WINDOW_RESIZED:** Only the last resize in a batch was delivered.
> - **B_MOUSE_MOVED:** Only the last mouse move in a batch was delivered.
> 
> Both were coalesced to avoid flooding the handler with intermediate states when the system was under load.
> 
> ### What pane does:
> 
> Looking at `looper.rs` lines 36-85, `drain_and_coalesce()`:
> 
> ```
> fn drain_and_coalesce(
>     first: LooperMessage,
>     receiver: &mpsc::Receiver<LooperMessage>,
>     pane_id: PaneId,
> ) -> Vec<Message> {
> ```
> 
> It does a blocking `recv()` for the first message, then `try_recv()` to drain the queue, then coalesces:
> - `Message::Resize` -- keeps only the last (correct, matches Be)
> - `Message::Mouse` where kind is `MouseEventKind::Move` -- keeps only the last (correct, matches Be)
> - Everything else -- delivered in order (correct)
> 
> **Assessment: Coalescing is correctly implemented per the Haiku docs.** Two things to verify:
> 
> 1. **Mouse move coalescing preserves ordering relative to other events.** In pane's implementation, the last mouse move replaces the first one *at the first one's position in the sequence*. This means if you get `[Move, Key, Move]`, you'll see `[Move(latest), Key]` -- the key event comes after the coalesced move. This is slightly different from Be's behavior where the move would be at its last position in the sequence. **Minor difference, unlikely to matter in practice.**
> 
> 2. **Missing coalescing: B_WINDOW_MOVED.** Be coalesced position changes too. pane doesn't have position events (Wayland doesn't report them), so this is N/A.
> 
> 3. **Not coalesced but should be considered:** Focus/Blur pairs. If the queue contains `[Focus, Blur, Focus]`, should pane coalesce to just `[Focus]`? Be didn't explicitly coalesce these, and pane doesn't either. Current behavior is correct.
