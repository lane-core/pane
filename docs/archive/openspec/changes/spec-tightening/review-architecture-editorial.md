# Architecture Spec: Editorial Review — Tensions and Commitment Boundaries

Reviewed: `openspec/specs/architecture/spec.md`
Cross-referenced: `openspec/specs/foundations/spec.md`
Date: 2026-03-20

Focus: (1) unnamed tensions between engineering commitments, (2) implementation details stated as commitments.

---

## Critical

Issues where the spec's own commitments create constraints it doesn't acknowledge, or where a mislabeled implementation detail could force unnecessary rewrites of a document that should be stable.

### C1. The two-thread-per-pane commitment vs. the "threads are cheap" assumption

The spec commits to the BeOS two-thread pattern — one client-side looper, one server-side thread per pane-native connection (§3, pane-comp). It also commits to multiple panes per connection (§3: "multiple panes per connection are sub-sessions"). These two commitments are in tension.

If one connection carries multiple panes (sub-sessions), does the compositor still spawn one server-side thread per connection, or one per pane? If per-connection, the two-thread model becomes "two threads per *connection*," not per pane — a weaker guarantee with different responsiveness properties (a slow pane blocks its siblings). If per-pane, the threading cost scales with pane count, not connection count, and the "~20KB per thread" budget (§6, citing Raynaud-Richard) matters more than the spec acknowledges.

The thread-cost table in §6 says "each pane-native connection (server-side) = dedicated thread," which implies per-connection. But §3 describes the pattern as analogous to "one ServerWindow thread per client," which in BeOS was per-window. The spec needs to state explicitly which grain it commits to and what the responsiveness tradeoff is.

### C2. Filesystem-as-interface commitment vs. the performance of FUSE

The foundations spec commits to the filesystem as universal FFI (§2, §8). The architecture spec commits to FUSE at `/srv/pane/` (§3, pane-fs) and simultaneously commits to "the typed protocol is the verified channel" (§3). The spec doesn't acknowledge the tension: FUSE introduces kernel-user round trips on every operation. For the canonical use case (shell scripts reading pane state), this is fine. But the spec also positions the filesystem as a *composition mechanism* — "anything exposed at `/srv/pane/` is scriptable" — which invites high-frequency programmatic access patterns where FUSE latency would be a real problem.

The tension: the stronger you make the filesystem commitment (more state exposed, richer operations), the more performance-sensitive access patterns you attract, and the worse FUSE performs for them. The spec should acknowledge this boundary — where does the filesystem interface stop being appropriate and the typed protocol take over? — rather than presenting both as equally viable for all consumers.

### C3. "No theme engine" vs. the democratic orientation

The architecture spec commits to "one opinionated look" and "no theme engine" (§10). The foundations spec commits to a "democratic orientation" where "the best user experiences are not generated by imposing a particular view" and users "compose their own" (§1). These are held in productive tension — the foundations spec (§9) explicitly names this tension — but the architecture spec's "no theme engine" is stated as flat commitment without the nuance.

The foundations spec says "the stock aesthetic — one opinionated look — empowers rather than constrains: users can build alternatives through the kit infrastructure" and "the mutable specs must find where the stock aesthetic retains its identity while customization remains accessible." The architecture spec doesn't do this work. It says "no theme engine" and moves on. This is where the architecture spec should name the mechanism: *how* does a user who wants a different look exercise the democratic principle? Rebuild the kit? Override specific rendering functions? The answer determines whether "no theme engine" means "themes are impossible" or "themes require more effort than dropping in a CSS file."

### C4. Session type enforcement vs. the transport bridge pragmatics

The spec commits to session types as "the single source of truth for every protocol in the system" (§4, pane-proto) and also commits to Phase 1 transport: "a hand-written state machine that mirrors the session type" (§7). The tension: a hand-written state machine that *mirrors* a session type is exactly where protocol drift happens. The spec's own mitigation — "the transport code stops compiling because it references the same enums and structs" — is a shared-types argument, not a session-type argument. Shared Rust enums give you the right message shapes; they don't verify ordering, branching, or completion.

The spec acknowledges this in §13 (Open Questions, "session type transport bridge") as needing a prototype, which is honest. But the body of the spec (§7) presents Phase 1 as if it delivers the session type guarantee, when it actually delivers shared-type-checking with session types as documentation. The commitment boundary here matters: is the commitment "session types enforce protocol correctness" or "session types specify protocol correctness and we verify what we can at each phase"? The spec should say which.

## Important

Tensions and boundary issues that affect design clarity but don't risk structural confusion.

### I1. calloop as "scoped exception" vs. par's async model everywhere

The spec is careful to say calloop is scoped to the compositor only (§3, §6, §11). But it also says par is async-native and the compositor drives par futures within calloop's executor (§6, §7). This creates a second async runtime (calloop's futures executor) that the "no async runtime" commitment (§6: "no async runtime, no system-wide executor") doesn't acknowledge. calloop's futures executor *is* an async executor scoped to the compositor. The spec would be more honest to say "no *system-wide* async runtime; the compositor uses a local executor for par integration" rather than implying there's no executor at all.

### I2. Kit as library (can't crash independently) vs. kit complexity

§3 argues routing belongs in the kit because "the kit is a library linked into every pane-native process — it cannot 'crash' independently of the process that uses it." This is true of the crash semantics, but the kit is doing significant work: rule evaluation, content transformation, registry queries, reconnection management, send buffering (§4, pane-app). A bug in the kit *does* crash the host process. The more functionality moves into the kit, the larger the blast radius of a kit bug — every pane-native process is affected. This is the classic shared-library risk. The spec should acknowledge that "can't crash independently" is not the same as "can't cause crashes" and note the testing/stability requirements that follow.

### I3. "Agents as Unix users" vs. namespace isolation

§4 (pane-ai) commits to agents as actual system users with their own accounts, then immediately says their filesystem view is "scoped via Linux user namespaces." But user namespaces change the meaning of "actual system user" — a namespaced user may not appear in `who` output from the host namespace, and its uid mapping may differ. The spec's own examples (`who` shows agents, `finger agent.reviewer` shows its spec) assume host-namespace visibility. The spec should state whether agents are host-namespace users (visible everywhere, limited by traditional Unix permissions) or namespace-isolated users (invisible from host, limited by namespace boundaries). These produce very different security and discoverability properties.

### I4. Heartbeat intervals as commitment

§7 (Heartbeat) specifies "interval: 2s, threshold: 3 misses = 6s detection." §3 (pane-watchdog) repeats the same numbers. These are tuning parameters that should be determined empirically — 6 seconds may be too slow for a crashed compositor (the user sees a frozen screen) or too aggressive for a loaded system (false positives). Stating them as spec-level commitments means the spec needs revision when profiling suggests different values. These belong in configuration, not in the architecture document.

### I5. The xattr size limit decision is a commitment; the filesystem names are implementation details

§3 (pane-store) commits to btrfs (primary) or XFS (supported alternative) and rules out ext4. This is a genuine architectural commitment — it constrains installation targets. Good. But the specific numbers ("btrfs supports ~16KB per xattr value"; "XFS supports 64KB per value with three-tier storage") are filesystem implementation details that could change with kernel versions. They belong in a rationale document, not in the architecture spec. What the spec should commit to is the *requirement*: "the target filesystem must support xattr values of at least N KB with no restrictive per-inode total limit."

Similarly, bcachefs speculation ("2027-2028") will age badly. A timeline for a third-party project's maturity doesn't belong in an architecture commitment.

### I6. The `.app` directory + Nix flake installation model is underspecified relative to its ambition

§13 introduces `.app` directories as "inspired by macOS's application bundles, built on Nix flakes." This is a significant design commitment (application packaging model) buried in an open question section and described in two paragraphs. The tension: the spec commits to progressive integration (legacy Wayland apps gain pane features incrementally), which requires the `.app` directory model to be flexible enough to wrap arbitrary applications. But the spec also commits to Nix flakes as the installation mechanism, which requires the application to be expressible as a Nix derivation. Many applications that users would want to wrap (AppImages, proprietary binaries, Flatpaks) resist Nix packaging. The spec doesn't acknowledge this gap.

## Minor

Lines where a specific library, version, or mechanism is stated as commitment when it's really a current best guess. If a better option appeared tomorrow, the spec shouldn't need to change.

### M1. Specific library names in the technology table (§11)

The following entries are implementation choices, not architectural commitments. If better alternatives emerged, pane should be able to adopt them without revising the architecture spec:

- **femtovg** — "2D vector graphics on OpenGL via glow." The commitment is GPU-accelerated 2D widget rendering. femtovg is the current choice.
- **taffy** — "Flexbox/grid layout engine." The commitment is declarative layout (flexbox/grid model). taffy is the current choice.
- **fuser** — "FUSE at `/srv/pane/`." The commitment is a FUSE filesystem. fuser is the current choice.
- **postcard** — "Serde-based, varint-encoded, compact binary." The commitment is compact binary serialization compatible with serde. postcard is the current choice.
- **zbus** — "Rust D-Bus implementation." The commitment is D-Bus bridging. zbus is the current choice.
- **proptest** — "Property-based tests for protocol correctness." The commitment is property-based testing. proptest is the current choice.

Contrast with entries that *are* architectural commitments: smithay (defines the compositor's structure), calloop (required by smithay), par (defines the session type model), s6 (defines the boot and supervision model), Nix (defines the build and distribution model), PipeWire (defines the media architecture).

The table could separate these two categories: architectural commitments that define pane's structure vs. current implementation choices that serve those commitments.

### M2. Specific Wayland protocol extensions listed in §3

§3 lists specific protocol extensions: "xdg-shell, layer-shell, linux-dmabuf, wl_shm, wl_seat, xdg-decoration, fractional-scale, viewporter, presentation-time, input method protocols." Some of these are core Wayland (wl_shm, wl_seat) and effectively permanent. Others (layer-shell is wlr-layer-shell, not yet a Wayland standard; xdg-decoration may be superseded) are current best options. The list will need maintenance as the Wayland ecosystem evolves. Consider separating core Wayland protocol from compositor-specific extensions.

### M3. Specific s6 commands and file formats in §9

§9 includes specific execline scripts, notification-fd files, and dependency directory layouts. These are useful documentation but they're implementation details that will change as service definitions evolve. The commitment is "s6-rc service definitions are Nix derivation outputs with readiness notification and dependency ordering." The specific scripts are examples that belong in implementation docs.

### M4. "~120,000 packages" is a point-in-time number

§1 and §9 cite "120,000+" and "~120,000" nixpkgs packages. This number grows. The commitment is "pane inherits the full nixpkgs ecosystem without forking it." The specific count is color commentary.

### M5. GPU glyph atlas as commitment vs. technique

§11 lists "GPU glyph atlas" with "instanced rendering" as a technology choice. This is a rendering technique, not a technology selection. The commitment is fast text rendering shared across pane-native clients. The specific technique (glyph atlas vs. SDF rendering vs. some future approach) is an implementation detail.

### M6. "Rounded corners (3-4px radius)" in §10

Pixel-level aesthetic specifications will change with testing, HiDPI considerations, and design refinement. The commitment is "rounded corners on controls." The specific radius is a design token, not an architecture decision.

## Alignment notes (architecture vs. foundations)

A few observations on spec alignment, for context. The accuracy review (review-architecture-accuracy.md) covers contradictions in detail; these are structural alignment observations.

**Good alignment:** The architecture spec successfully concretizes the foundations spec's commitments on protocol discipline (§3/§7), the pane as universal object (§2), infrastructure-first design (§8), and the kit ecosystem (§4/§10). The removal of the central router (§3) is well-argued and consistent with the foundations spec's "coordination emerges from the protocol, not from a global coordinator" (foundations §3).

**Structural gap:** The architecture spec's §7 (Error Composition) was added to address C1 from the accuracy review, but it reads as an addendum rather than integrated design. The monadic error composition principle should thread through the server and kit descriptions — each server's crash behavior, each kit's error propagation — rather than being stated once in the protocol section.

**Scope creep into the architecture spec:** §13 (Open Questions) contains material that straddles architecture and implementation. The `.app` directory model, the PipeWire screen capture integration, and the selection-first vs. verb-first question are implementation-level decisions that don't need to live in the architecture spec. The architecture spec's open questions should be "things that affect the architecture if answered differently" — the transport bridge, the affine/linear gap, the dynamic optic composition. Whether widgets use selection-first or verb-first doesn't change the architecture.
