# Pane — Theoretical Foundations

This document is the canonical source of truth for the principles that inform pane's design. All other specifications — the architecture spec, component specs, implementation plans — reference and depend on this document. They may change freely; this document is static until serious redesign is explicitly required.

We are confident in our design, but uncompromising also means interrogating our own decisions at every stage. These principles are commitments, not dogma. They are held because they have been examined — and they will be examined again if the evidence demands it.

---

## 1. What Pane Is

A pane is a transparent object. Its purpose is to let you see what it presents clearly.

Pane is a desktop environment and distribution for Linux built from a small, principled core — typed protocols, filesystem interfaces, composable servers — from which the entire experience is derived by first principles. The degree of integration between pane's user experience layer and the underlying system is deep and opinionated — closer to what NeXTSTEP was to Mach/BSD than to what a conventional desktop environment is to its host distribution. You don't think of macOS as "Aqua running on Darwin." It's one thing. Pane aspires to the same: a complete system identity built over the Linux kernel, where the boundary between the desktop and the OS is invisible to the user. The core is to be compact, efficient, and bulletproof. Every other part of the system can be rewritten as needs evolve — a ship of Theseus that retains its identity through its architecture, not through its specific components.

Thus Pane can be more properly characterized as the protocol and contract that sets explicit, expressive, yet reliable terms by which all components and users of the resulting system interact. The goal is for users to have the greatest possible creativity of curating their own experience without compromising on stability, while providing a reference stock UX that is compelling on its own account, setting a high bar yet encouraging its users to climb over it and giving them a variety of powerful tools to do so. This departs from the democratic orientation of our design philosphy: the best user experiences are not generated by imposing a particular view of what computing should be, but by providing a powerful and flexible foundation that users can build on to invent their own experiences. The stock UX is just one presentation of a variety of possible experiences enabled by the underlying system architecture.

The system should be approachable on first contact and reveal depth as the user grows. This is not achieved through tutorials, wizards, or help dialogs that users dismiss without reading. They can learn by experiment and on their own initiative page the help when things go wrong (literally).

We imagine this sort of scenario: a new user's first encounter with pane is a resident guide agent — a system participant (§10) whose `.plan` file (living in their home directory) is to help users understand pane's interfaces by demonstration. The guide teaches pane _using pane_. Every explanation can be accompanied by a live demonstration of the system's own capabilities, and catered to the user's specific interests or knowledge gaps. The ideal is that the user doesn't need to study abstract principles to start to understand how their system works — they can learn by watching the system work from the inside, taught by a responsive assistant who can inform them in minute detail how their system is configured, who can also walk them through how to alter it step by step to suit their own needs. If the user wants to know where their agent learned these things from, they can show them exactly where in the manual to look and where to find related information. This agent is not an overseer of user activity, it is a new colleague eager to work with them showing them the ropes.

The user absorbs the system's patterns by watching someone (something) use them naturally. The system absorbs the user's patterns by being programmed for responsiveness to their areas of concern. When they outgrow the guide, their understanding of the system was guided by a source able to meet them where they were at, who encouraged inquiry and supported the user's initiative and experiment. To facilitate the smooth functioning of such endeavors, not only is a resident agent trained with the system's own documentation, its actions are executed by means of the protocol the system establishes, with faculties for sanity checking and safety guarantees provided by the system architecture and fundamental to its design. Not just through well thought out security and design principles, but also by contracts enforced via typechecked protocols governing each context under which system interaction takes place, by human users or otherwise.

All of this is achieved through principled systems design infrastructure whose compositional approach extends to its relationship to the host system, the Linux base and the variety of tools already populating the ecosystem. When the user asks "how do I customize this?" the guide modifies a config file and the user sees the change take effect live. When they ask "what are my agents doing?" the guide shows them `who` and `finger` — the same tools they'll use on their own when engaging with the user and other agents. In other words, the guide uses the same tools the user will eventually use directly to interact with the system.

This last example with `who` and `finger` is instructive as to a resourcefulness that permeates our design philosophy. Designed in the days when Unix systems were exclusively the province of mainframes located within large firms, government or educational institutions, `who` and `finger` are relics of an interface built to facilitate clean and efficient interaction on multi-user systems, which fell into obscurity when commodity hardware enabled single-user computing as the norm. We might be led to ask: could it not be the case that some of the most brilliant systems engineers of all time already solved in principle major design challenges presenting themselves to agentic workflow developers? These were workflows developed to coordinate activities between dozens of the most adversarial actors of all: human beings; they must certainly be sufficient to manage agents. And they were developed to facilitate communication using the same command line tools which are already the bread and butter of agents and users alike, which is already becoming the primary way they communicate in various settings.

Our aim is not to reinvent yet-another-suite of yet-another-\* tools, to have our own music or photo apps. Our ambition is to realize a powerful environment enabling startling new uses for existing tools. And who knows, maybe something amazing happens once you imagine each computer as a collective endeavor no different in principle from any other enterprise, an assemblage of interactions between human beings or AI agents alike within or across networks.

The ideas that inform pane were not born in a vacuum. BeOS proved in the 1990s that message-passing discipline, per-component threading, and infrastructure-first design produce stable, responsive, integrated desktop systems. Plan 9 proved that protocol uniformity and per-process composition enable emergent capabilities no designer anticipated. These ideas were pushed to obscurity not because they failed but because the platforms that carried them lost the market to entrenched ecosystems. Pane picks up the thread — applying a unified OS design philosophy over a contemporary Linux base, drawing on what those systems proved while leveraging untapped potentials of what computing makes possible in 2026.

Although we are obliged to ship a compelling stock UX experience, our goal is not to say the final word about what the experience of using pane is, it is to demonstrate by proof of concept what a pane system is, one vision among many. The architecture described above may sound utopian in its dream of a unicorn able to satisfy power users and normal users alike. It certainly cannot proceed without carefully considering the system down to its most fundamental objects and interfaces. It also would be foolish to pass up any opportunities to stand on the shoulders of giants, whose strength and weaknesses we must learn from. The remainder of this document presents the hypothesis of how we'll get there.

## 2. The Pane as Universal Object

The pane is the universal object of the system. Every interface — to the user, to the system, to other components, to external tools — is a pane. This is not a UI convention but a structural commitment: the system's power derives from the uniformity of this fundamental component, the same way that unix's philosophy that "everything is a file" enabled a compositional design sensibility producing emergent use cases that no designer anticipated. The beauty is that they didn't need to anticipate them. The user was given everything they already needed.

A pane is one thing with many views. The same pane presents itself differently to different consumers: as a visual display to the user, as a protocol endpoint to other components, as a filesystem node to scripts and tools, as a semantic object to accessibility infrastructure. These are not separate systems bolted together — they are projections of the same underlying state. When state changes through one view, the other views reflect it consistently. The projections are to be governed by the discipline of optics (§4): each view satisfies correctness criteria that guarantee consistency between internal state and external representation.

Consider notifications. In conventional desktops, a notification is a transient popup — it appears, it disappears, and if you missed it, it's gone. In pane, a notification is a pane. It can be displayed as a floating element at a configurable position. But because it is a pane, it also has a filesystem projection — its attributes (source, timestamp, content, priority) are queryable. A user who wants to review missed notifications can query them. A user who doesn't care to keep them (or who is security minded) can set a retention policy that deletes them after a timeout, the way modern clipboards expire their contents. A user who wants notifications routed to a log can write a routing rule. None of this requires special notification infrastructure — it follows from the notification being a pane, and panes have filesystem views and participating in pane's routing infrastructure.

Panes compose. Two panes viewed together — tiled, tabbed, nested — form a compound structure. We: might say, they are tensored up to a notion of abstraction whose concrete presentation depends upon observational context. This juxtaposition might be spatial arrangement on a screen, where the user can see two panes displaying distinct sorts of data related to the an encapsulating pane, `foo`; on the filesystem, these panes are juxtaposed inhabitants of the `/srv/pane/<...>/foo` directory. Thus there is a sense in which the interface ought to be characterized by the faithful translation of program semantics across different choices of repesentation. The interactions that transform pane state are the two axes of composition. Those familiar with contemporary will recognize that we are attempt to think of panes as objects in a category, with interactions as morphisms and spatial composition as a monoidal product — but the principle requires no formalism to apply: panes compose spatially (and here space means any surface inspectable by a user or process) while interactions (automated or user-generated) transform their state.

The commitment to universality is bold and deliberate. Where conventional desktop environments have windows, views, menus, dialogs, notifications, and tooltips as separate concepts with distinct interfaces, pane has one object. They all speak the same protocol, accept the same interaction model, and compose the same way. The uniformity benefits observed in systems that committed to a universal abstraction — Plan 9's "everything (really) is a file," BeOS's "everything communicates via BMessage" — suggest that the most interesting results come from following a single design principle across the uncanny valley, so that one may build a bridge once they arrive on the other side. We are willing to break from conventional design patterns to discover what fidelity to this fundamental vision brings.

## 3. Protocol Discipline and Session Types

The operational semantics of the system are defined by its protocols. Each component's behavior is specified by the messages it sends and receives, the order in which they occur, and the choices that branch the conversation. The protocol is the operating principle — coordination emerges from the protocol, not from a global coordinator.

### The BeOS proof of concept

BeOS proved empirically that message-passing discipline produces robust and compelling systems design. The BMessage/BLooper model forced every component to implement a self-contained operational semantics: each looper had its own thread, its own message queue, and communicated only via asynchronous messages. A BMessage could carry any typed data and flow through the system without tight coupling between sender and receiver. Components didn't share mutable state — they exchanged messages. Each component could reason locally about its own behavior without accounting for the global system state, needing only to manage the production of its own side effects within its specific sphere of responsibility.

Carefully thought out design rigorously obeying disciplined design principles produced an OS leagues ahead of its contemporaries in stability and responsiveness (it is hard to overstate the extent to which this is the case, considering its competition was Windows 98 and classic MacOS). It worked not because message passing is inherently safe, but because the BMessage/BLooper model forced a discipline that made local reasoning sufficient: each component was an island of sequential processing in a sea of concurrent activity. The protocol — the ground rules for how messages flow — was the coordination mechanism. No global coordinator was needed.

### Session types formalize message passing discipline

Session types — typed descriptions of entire conversations between two endpoints — formalize exactly this discipline. What BMessage enforced by API convention and developer culture ("Be Commandment #1: Thou shalt not covet another thread's state or data without taking proper precautions"), session types enforce at compile time. The conversation structure — what is sent, in what order, by whom — is verified before the program runs.

Two decades of theoretical development (Honda 1993, Vasconcelos 2009, Caires-Pfenning 2010, Wadler 2012) eventually caught up to what Be's engineers achieved by skill, sensibility, and intuition alone. Pane stands on both: the empirical proof that it works, and the theoretical framework that lets the compiler verify it at compile time.

Deadlock freedom — the guarantee that no well-typed system of communicating processes can get stuck - is baked into the theory itself. The protocol discipline is backed by proof theory (linear logic, specifically), not just good intentions.

### Async by default

The default interaction is asynchronous: a component sends a message and continues without waiting. Synchronous interactions — send a request, wait for the response — are explicit in the session type. The type makes the async/sync distinction visible in the protocol definition, not hidden in runtime behavior.

Asynchronous messages can be batched — a relevant system service accumulates protocol messages (state updates, routing requests, attribute changes) and flushes in chunks, reducing overhead. Synchronous interactions force a flush and a round-trip. BeOS's engineers discovered this empirically: the Interface Kit batched async calls to the app_server, and synchronous calls were documented as "much slower" because they forced a flush. The principle generalizes: the default should be async; sync only when a response is needed.

### Type systems for developers should be harmonious and ergonomic to work with

The type system should be strict enough to catch protocol errors at compile time and flexible enough that components can extend conversations without breaking the system. We lean for type safety whenever possible. Some part of our experiment concerns new avenues for introducing type safety where naive or conventional approaches might not have dared to. There has traditionally been a tension perceived between type safety and good dev UX. Our contention is to structure pane's API such that type-safe design flows with and not against programmer intentions, so long as such intentions are reasonable. We can accomplish this by making more intentions reasonable to execute under such a framework by carefully chosen compromises. That was Rust's strategy, which payed off dividends. Affine type systems existed for decades in the arcana of academic programming language research prior, but establishing real-world adoption required not only thoughtful presentation, it also had to be timely, well motivated by the specific challenges of complex software design in a world plagued with security risks owing to a well understood and easily exploitable class of bugs. The result of this confluence of initiative, practical design sensibility, and timing was a robust systems design programming language with a rich and ever growing developer ecosystem, without which our project may not have been possible to write. One could argue that the technology itself is dwarfed by its own userbase. We wager that a similar gesture is possible now in operating system design, whose emergent benefits are yet to be determined.

We acknowledge that due to certain implementation choices (GNU/Linux base, etc) we are limited in our ability to enforce the type safety and correct-by-construction techniques across a vast code surface we have no direct control over. The only strategy to mitigate this is also the best one we have: design pane such that it is able to weather the elements as gracefully as possible. By strictly enforced design principles enforced at the API level for the most important and frequently used system processes, leveraging the ability to deploy decades of sound system administrative practice in new ways through the new design patterns enabled by our infrastructure, we hope to achieve this outcome.

## 4. Multiple Views Through Optics

A pane is one object with many views. The relationship between internal state and each external view is governed by optics — composable, bidirectional access paths. We use theory not for its own sake but because it resolves real design dilemmas: keeping multiple views of the same state consistent is a hard problem, and optics are the formalization that makes it tractable. In this we depart from the spirit of Be's approach — aiming for the strongest design forces the best programming patterns. In this context, optics discipline gives us correctness criteria that would otherwise be ad-hoc and error-prone.

### Lens laws as correctness criteria

Every read-write interface to a pane — filesystem, protocol, user-visible state — should satisfy the lens laws:

- **GetPut**: if you read a value through a view and write it back unchanged, the internal state is unchanged.
- **PutGet**: if you write a value through a view and read it back, you get the value you wrote.

These hold up to semantic equality — internal representation and external representation may differ in form but must agree in meaning. Violations are either intentionally lossy (documented as such) or bugs.

### Projections, not copies

Each view is a projection — it shows only part of the pane's state. The complement (what is lost in the projection) should be explicit. One cannot always reconstruct a pane from scratch in a generally nondeterministic operating environment; a write through the filesystem always needs the existing internal state as context, and verifying the integrity of reconstructed information is often precarious if its canonical and sole representation is lost. Making runtime risks explicit prevents bugs prior to implementation itself, before code has the chance to assume a given view is lossless.

### Composition

Optics compose. The projection from internal state to protocol messages, composed with the projection from protocol messages to filesystem representation, gives the composite projection from internal state to filesystem. This mirrors the system architecture: the filesystem interface is just another client that speaks the protocol — it composes two optics rather than having privileged access to internal state. In the case of BeOS this principle was more weakly enforced by the API by comparison, but this was still often sufficient to guide good programming practice on average. Now that we have the right abstractions to formalize what this practice is, we can much more clearly depict this methodology in our API design.

### Relationship to session types

Session types are the horizontal structure — conversation over time, typed conversations between alternating senders and receivers. Optics are the vertical structure — state access at each moment. Together: session types characterize WHAT interactions happen; optics depict HOW state is projected and updated within those interactions. Each step in a session type reads from or writes to pane state through an optic.

## 5. Local Operational Semantics and Concurrency

Each component has its own thread and its own message queue. Messages are processed sequentially on each thread. Concurrency arises from many such components running simultaneously, not from concurrent execution within a component.

This is BeOS's BLooper model, and it is the reason BeOS was stable. When every component processes messages sequentially on its own thread, each component can reason locally about its own state without accounting for the global state of the system. Side effects occur only at message boundaries. The scheduler manages concurrency transparently — a component does not need to know whether it is running on one CPU or many.

Building for the hardest case — pervasive concurrency, symmetric multiprocessing — produces architecture that is better for every case. BeOS built for SMP and got a system that was also superior on single-processor machines, because the discipline required for correctness under concurrency — self-contained operational semantics, message-passing rather than shared state, local reasoning — is the same discipline that produces clean, modular, stable software regardless of the hardware.

### The cost of good abstractions

Every abstraction has a cost. Per-component threading costs memory and scheduling overhead. Message passing costs serialization and dispatch. The question is not whether these costs exist but whether the architecture gives us more than it costs — both in isolation and in aggregate.

Small inefficiencies for more robust and clear systems design, both in each component and across the system as a whole, may pay off far more than designs that attempt to microoptimize each routine. There is a network effect to good design choices: when every component follows the same discipline, the benefits compound — debugging is easier, composition is natural, interfaces are ergonomic, failures are contained, and new components inherit the properties of the architecture by the cost of attendance.

An irony often observed even among excellent programmers: they become so invested in the cleverness of their specific implementation choices that they grow susceptible to bad compromises with respect to the overall architecture resulting from overfitting for a specific optimization path or visualized use case— compromises they are disinclined to revisit due to sunk cost. This has played out repeatedly in open source and commercial software. The antidote is to value the architecture over any individual component's, and to recognize that the discipline of the whole system is worth more than the optimization of any part.

Session types formalize the threading discipline: typed conversation between two endpoints IS the message-passing discipline, formalized and verified by the compiler.

## 6. Monadic Error Composition

Failures are values, not exceptions. Error handling composes through the same typed channels that handle the happy path.

### Errors as protocol events

A component crash does not need to be a system failure — it can be a protocol event. When a session is interrupted, the counterpart receives a typed "session terminated" event, meant to be presented or routed to whichever interface is best disposed to handle it. The system continues regardless. This is the monadic principle applied to system design: each operation either succeeds (producing a value and a continuation) or fails (producing an error that propagates through the composition). Recovery strategies — retry, fallback, degrade gracefully — are themselves composable operations in the same pipeline.

BeOS's engineers understood this intuitively. When a window's thread became unresponsive, the app_server didn't crash — it shed messages for that window and continued serving others. When a BLooper's lock detected a deleted target, it returned an error rather than dereferencing invalid memory. The cleanup path was part of the design, not an afterthought. George Hoffman documented that the app_server would "start to throw away update messages and input intended for the window" if a window thread became unresponsive — graceful degradation by design.

### Well crafted abstractions are robust to failure

The same principle that produces stability in the happy path — each component's operational semantics are local and self-contained — also makes failure more graceful. A crashed component cannot corrupt another component's state because components do not share state. They communicate via typed messages where side-effects are made explicit by optics discipline whenever possible. If a component dies, its messages stop arriving. The counterpart handles this through its typed error handling path, not by the silent progress of shared-memory corruption or cascading locks.

Recovery is structured, not ad-hoc:

- The roster tracks who is alive and updates its directory when servers register, restart, or terminate
- The compositor handles client disappearance at any point in the UI session protocol
- The router is the dispatch of last resort — the one component that must not fail, because the entire communication model rests on it. It is also the component best positioned for robustness, because session-typed design applies most directly to its core function. In the event of substantial system failure — loss of user input, unresponsive servers — the router can trigger escalation procedures: writing the journal to disk, backing up elements of user state, bracing the system against cascading failure. If one thing stays up, it must be the communication infrastructure.

Each level handles failure at its level; every functionality is accompanied by a proportional sphere of responsibility, while the greatest part of this burden is carried by the system as a whole. We structure these levels such that the concerns they organized are cleanly separated. The guiding principle is that failures are protocol events that compose monadically, not system events that cascade unpredictably.

### Localized and modular systems facilitate robustness and maintainability

The goal is to enable robust systems by construction. Sufficiently localized operational semantics ensure sufficient process isolation; session types make the most important class of errors by our chosen conventions (message passing) explicit at the protocol level; our init abstraction provides a contractual guarantee of restart without any component implementing infrastructure. Pane applications get convenient and reliable interfaces for free, which abstract over the underlying host system.

Our commitment to modular systems design ought to enable us to respond gracefully to the ever-changing landscape of linux systems tooling as soon as changes appear on the scene. Traditional monolithic architectures are not well suited to this task. Updates in any one component may have unexpected consequences in others, and initial compromises made to accommodate an underlying dependency to fit the framework retroactively prove to be unfounded by unexpected developments in those third party codebases. The compromises they're forced to make already weren't favorable, and each update progressively forces worse ones, leading to major-minor development cycles where developer's attention is unavoidably split between two evolving codebases for great stretches of time.

We attribute the long-term success of source based Linux distributions precisely to the systemic problems with this development model. They designed systems that were built to handle the patchwork of system dependencies that constitute a Linux-based operating environment, flexible enough to meet the needs of users who for any reason wanted the very latest packages or updates as soon as they released. Such distributions are known for having the latest package bases on aggregate far outstripping other distributions. The lack of opinionation was _their_ uniform design principle, and also their only opinionated choice. Their opinion was that the right system was ready to go wherever its constituent parts were heading towards, or wherever its users wanted it to go in general. Initially these distributions were ridiculed as fools' errands for bored hackers, but now they are among the most popular style of linux distributions. They've even come to define the brand of what being a Linux user entails. The features derided as silly or impractical are exactly those which respective users love the most about their systems, and the communities united by their love of the personal curation a customized system enables forms the most distinctive quality of the Linux ecosystem as a whole. Any framework that isn't willing to embrace this principle in its soul will not thrive in this setting, but pushing the art forward may be among the best strategies for earning loyalty among this hardened core of linux enthusiasts, who have kept this platform alive and whose needs ought to be naturally accommodated.

Up to this point, every statement we made regarding the weakness of monolithic architectures are ones that we have to fiercely watch for in the course of pane's development. This is why from the beginning we will proceed from recognizing the wisdom of many generations of hackers who ultimately accepted full responsibility of their own system management: those who recognized that the best user experiences they could curate for themselves on Linux resulted from embracing the inherent freedom and chaos of the ecosystem, and willingness to accept the responsibility that entailed for them as users rather than fight it. The linux ecosystem has been modular since its inception, the way that ours is modular ought to conform with it, so that evolving alongside it is tractable on the part of the project for both cultural and technical reasons.

All of these considerations are made to give us the best chance to prove the viability of a new paradigm in operating system design allowing us to break with the misconception that freedom is too difficult to bother with. We accomplish this by designing our system as a coherent extension of the principles governing our underlying Linux base, rethought to suit the needs of a new generation of users.

## 7. Infrastructure-First Design

To build the right infrastructure is to lay the bedrock for excellent and creative experiences.

BeOS's engineers didn't build a monolithic email application. They designed infrastructure — a filesystem with typed attributes, a query engine, a file manager that could display any file with any attributes, a mail daemon that delivered messages as files. The design logic of this infrastructure made email a natural consequence of composition: queries over mail attributes became inboxes, the file manager became the mail client, composing a reply was opening a file. The experience was achieved through infrastructure rather than a dedicated application, and it was better for it — because each component was independently useful and the integration was not brittle.

This is the standard pane aspires to, and it connects directly to its egalitarian orientation. When the system provides infrastructure rather than finished experiences, users can compose their own experiences from that infrastructure. The developer doesn't get the final word. The infrastructure does its job — routing, indexing, filesystem exposure, lifecycle management, compositing — and what emerges from the composition is up to whoever is composing. The BeOS email-as-a-folder example is not just an engineering achievement; it's a demonstration that infrastructure-first design is inherently democratic, because the infrastructure doesn't presume what the user wants to do with it, but allows usage patterns to freely emerge from a minimal set of commitments.

Consider what this means in practice. Routing composes content with handlers — but which handlers, and for what content, is determined by the user's routing rules. Attribute indexing composes metadata with queries — but what metadata matters and what queries are useful is determined by how the user and their agents organize their work. Filesystem exposure makes system state scriptable — but what scripts get written, by whom, for what purpose, is unbounded. Every infrastructure composition point is also a user empowerment point. The infrastructure is not neutral — it is designed with opinions about how communication should work, how state should be managed, how failure should be handled — but it is silent on what the user should do. That silence is the space where creativity happens.

When a new experience is needed, the first question is: can it emerge from composing existing infrastructure? If yes, build it as a composition. If not, the question is whether the missing infrastructure is general enough to justify building. If so, build the infrastructure. The experience will follow. And if an agent can build the composition on behalf of a user who merely described what they wanted — routing rules, translators, pane modes, all dropped into directories — then the infrastructure-first principle extends from developer creativity to user creativity, with the agent as mediator.

## 8. Uniformity and the Protocol as Harmonizer

The number of distinct interfaces in a system is the enemy of composition. Every additional protocol, every special API, every bespoke communication mechanism is a boundary across which composition stops. The fundamental design move is to reduce the number of distinct interfaces.

Plan 9 proved this with one protocol (9P) for everything. BeOS proved it with one messaging system (BMessage/BLooper) for everything. In both cases, the commitment to uniform design produced emergent capabilities that no designer had to plan individually — because when everything speaks the same language, composing functionality is natural. Such capacities temper the opinionated design of the system with a fundamental openness to creativity and innovation. One of the sins of traditional monolithic OS design is the expectation that the developer has the final word in how a system is meant to be used. We seek to renegotiate this contract and proceed from a place of respect and humility with regard to the needs of users. While we are confident that our opinionated stock UX will be compelling in its own right to users who want a no-fuss experience (past the inevitable short learning curve analogous from switching to Windows to MacOS), we want the freedom of the user to curate their own UX to be first class. Our design philosophy is realized on the same footing as anyone else's; the stock UX is just one presentation of a variety of possible experiences enabled by the underlying system architecture.

To achieve this end pane pursues uniformity through: one universal object (the pane), one communication model (session-typed protocols via the routing infrastructure), common extension mechanisms (filesystem-based, typed interfaces, Be's translation kit is paradigmatic example). The protocol is the harmonizer — it provides the mechanism by which different applications, including legacy applications that do not utilize the pane infrastructure natively, participate coherently in the user experience as a whole. The protocol is intended to harmonize how they coexist. Our ultimate thesis is that carefully thought out design choices actually minimize the level of opinionated design on the part of our system. The systems designer is opinionated once, after which the ultimate use to which our infrastructure is put is wholly determined by free experiment.

Bridges translate at the boundaries. Foreign protocols are integrated via bridge daemons that translate between the foreign protocol and pane's native message model. Each bridge is a plugin. The pane side is always the same typed interface. This is strictly complementary to Linux's existing infrastructure — pane does not replace sockets, D-Bus, or networking. It provides ergonomic abstractions over them.

One communication model, not two. BeOS learned this the hard way with their abandoned DSP experiment: maintaining two programming models and two system software layers costs more in coordination than the heterogeneity gains. This represents a specific design challenge: while pane strives for its own opinionated take on operating system design, the best hope for succeeding in this task consists in reimagining what a Linux-based operating system could be, unleashing latent potentials that have remained unexplored in traditional Linux distributions. We would be wise to lean into already existing usage patterns in the ecosystem. The ability of users to curate their experience on pane should feel like a natural extensions of the kinds of things they do on their Linux systems all the time. We seek to provoke users' curiosity by enabling capabilities they may have never realized they needed; to produce excitement in their realization that they are fully equipped to explore them as pane's interfaces compose ergonomically with tools and paradigms they are already familiar with. Our congruence with existing usage patterns is pivotal, not just as an adoption strategy, but as a powerful design principle in its own right.

Complementarity enables the possibility of dialogue between the existing ecosystem and the new one we are attempting to establish, but also something else just as important: if we speak a comparable design language, the ways in which we will be able to integrate existing tools in our system will mirror the way we pair with how we inject pane's functionality with existing tools. Shells can pipe standard output directly into other panes. Stock pane.nvim or pane.el plugins would be excellent and productive avenues to exhibit the new usage patterns pane enables. Users already love these tools (infamously, some are even too fiercely loyal to their specific choices); showcasing the power of our infrastructure in a setting they are already familiar with is the best kind of advertising we can get with the power users. Showing that pane readily accommodates their own opinionated choices builds trust in users who are historically hostile to endeavors which attempt to port over features from mainstream operating systems while smuggling in the same downsides that drove them to abandon these platforms. These are some of the most important users to win over, their initiative are invaluable to our efforts. These are the people who will dream up things pane's development team could have never imagined, the people who will discover killer apps driving forth further adoption. They are also the people who will poke and stress test every aspect of our system, who will break things and force us to harden and refine pane to meet their expectations of reliability, safety, and security to the same extent they also demand desirable user experiences for their particular use cases. They are also among the most demanding of particular features in other projects. The best way to compensate for our inability to accommodate the needs of each and every user is to provide a foundation for achieving any particular need, placing the responsibility for niche desired features in users' hands in ways that seem reasonable, familiar, and fair. The vast plugin ecosystems of emacs and (neo)vim are instructive historic examples of this virtuous (if at times chaotic) cycle.

## 9. The Aesthetic Commitment

All this thought put into systems design is in service of something the user actually sees and touches. The best architecture in the world is worthless if the system is ugly, hostile, or confusing to use. And conversely: an aesthetic that is merely decorative, detached from the architecture that produces it, is fragile and superficial. Pane's aesthetic is inseparable from its architecture — the integrated feel comes from the same infrastructure that produces the integrated behavior.

Pane reimagines desktop design from a fork in the road that presented itself at the turn of the millennium. In the early 2000s, Apple's Aqua pushed forward genuinely great ideas in interface design — material qualities, depth, warmth, visual hierarchy that invited interaction. Meanwhile, the alternative platforms that had pioneered power-user-friendly, dense, integrated design (BeOS, Plan 9) were lost to obscurity. The survivors continued alone, and OS design stagnated — accumulating features at the expense of coherent design philosophy.

Pane is meant to evoke the memory of the best aspects of this era of systems design; the nostalgia isn't entirely meant to be indulgent (although it certainly is in part ;)), it is a signal for those who remember this era of computing, and expresses our desire to be a credible standard bearer for this tradition for new generations of users who are tempted to know what it was like those days. The persistence of Linux and the unix-like ecosystem as a whole is the last remaining legacy of this era; what we want to bring back is the thrill that computing was actively going places noone has ever been before. Something new, something other than what we've been condition to accept as all we can realistically expect from our computing experience. We want to show that the best of that era was not a dead end, but a path forward — that the design principles that produced those systems are still valid, and that with the right architecture, they can be the foundation for a new generation of compelling user experiences.

Pane continues what Be pioneered: a power-user-friendly but elegant interface, dense with information but refined in presentation, updated for 2026. We proceed from the inclination that a computer interface should invite interaction rather than demand it, and it should indulge your curiosity rather than evade it. Interfaces should have material qualities — depth, translucency, texture, tactility — that make them feel real, intriguing, approachable, with these qualities serving comprehension rather than merely entertaining. The aesthetic should feel like an invitation, not a demand: warm enough to welcome newcomers, dense enough to respect the time of experts if necessary.

Aesthetic consistency is architectural, not cosmetic. Every native pane renders through the same kit infrastructure, producing the same fonts, the same styling, the same visual language — not because a central authority forces it, but because the kits make consistency the path of least resistance. Input kits provided universal keybinding mechanisms of the same strength and power as those found in cult favorite text editor interfaces, generalized over multiple interfaces. The compositor provides uniform chrome. The integrated feel comes from shared infrastructure — the same way BeOS achieved it, because every application used the Interface Kit, not because app_server rendered everything.

There is a tension between the opinionated aesthetic and our democratic design orientation. Our conviction is that the stock aesthetic — one opinionated look, no theme engine, the visual identity IS pane's identity — is itself a design choice that empowers rather than constrains. Users who want a different aesthetic can build one through the kit infrastructure, because the rendering is client-side and the kits are the programming model. What we decline to do is ship a system with no visual identity and call it "customizable." The stock UX must be compelling enough to be worth using as-is, because that is how initial trust is built, that is the sole opportunity we have to give a proof of concept of the power of the underlying architecture. The presentation must feel harmonious with the fluidity of the underlying architecture. The aesthetic is the first thing a user experiences, before they understand the architecture, before they appreciate the protocols, before they discover the extension model. If the first impression is generic or unfinished, everything else is academic. We are obliged to consider the aesthetic as seriously as we do our type system (which is to say, very seriously), because for the user, the aesthetic IS the system. Achieving that aesthetic will only be accomplished however if the UX achieves an ideal of transparency with regard to its innerworkings; it will only be loved if it shows its personality in earnest.

## 10. System Participants and the Kit Ecosystem

### The two layers

The system separates into two layers, defined by their relationship to the protocol. This separation is not a matter of privilege — it is a matter of what kind of work each layer does.

**The core system layer** implements the servers and infrastructure: compositor, router, roster, attribute store, filesystem interface, notification system, init abstraction. This layer has an intimate relationship with the host OS — it talks to the kernel, manages hardware, handles rendering. It is the foundation that everything else stands on, and it must be solid enough that the people standing on it never think about it. The core layer is necessarily implementation-specific, written in a systems language suited to the task. It is the part of pane that earns the "compact, efficient, and bulletproof" promise.

**The outer layer** is everything that speaks the protocol: clients, modes, translators, bridges, agents, external tools — in short, inhabitants. Any participant that can communicate over the protocol can participate fully in the system, regardless of what language it is written in, where it runs, or whether it is human. The protocol is the only thing that matters for participation. This is the layer where the ecosystem lives, where the creativity happens, where the experiences that no designer anticipated emerge from the infrastructure the core layer provides.

### Kits

Kits are the BeOS kit concept, and understanding them correctly is essential. They are not convenience wrappers over a protocol. They ARE the programming model — cohesive subsystems that each address a domain completely. The Application Kit is the messaging and lifecycle model. The Interface Kit is the UI subsystem — rendering, styling, layout, input handling. The Storage Kit is the filesystem data layer. When a developer uses the Interface Kit, they are not "calling the compositor's API through a wrapper." They are using a complete UI programming model that happens to communicate with the compositor internally through the protocol, the same way that BeOS's libbe.so happened to communicate with app_server through kernel ports, but presented to the developer a coherent, opinionated world of BWindows, BViews, and BMessages.

This distinction matters because it determines what developing for pane feels like. If kits are thin wrappers, developing for pane feels like wrestling with a protocol. If kits are substantial programming models — with their own idioms, their own composition patterns, their own way of thinking about the domain — then developing for pane feels like using a framework that was designed for exactly what you're trying to build. The latter is what BeOS achieved. Developers loved the BeAPI not because it was minimal but because it was thoughtful — the primitives were small, composable, and designed by people who wrote real applications. That is the standard.

Pane preserves this design. The specific kits address contemporary needs — an AI Kit for agent infrastructure, a Media Kit abstracting over the host audio/video stack, an Input Kit generalizing the keybinding power of the best text editors across every interface — but follow the same pattern: cohesive, layered, forming an ecology. Because the protocol is well-defined, kits can be implemented in any language. But the kit in each language must be a substantial programming model, not a thin binding. The goal is that a developer choosing to build for pane in their language of choice finds a kit that feels native to that language while expressing the full power of pane's architecture. Growing the ecosystem means growing the number of languages where this is true.

### System participants

One of the more radical ideas in pane's design is that it takes the question "who uses a computer?" seriously, and answers it honestly: not just a person sitting at a keyboard. Agents, services, scheduled tasks, remote participants — these are all users of the system in the Unix sense, and pane treats them as such. Not as applications running inside the system, but as inhabitants of it.

This is not a futuristic abstraction. It is a recovery. The Unix multi-user model was designed for precisely this situation: a shared system with multiple concurrent inhabitants who communicate, collaborate, and govern their shared resources through protocols. `who` tells you who's logged in. `finger` tells you what they're working on. `write` lets you send them a message. `mail` lets you leave something for when they're ready. `mesg` lets them say "not now." These tools were built to manage the most adversarial actors of all — human beings sharing a mainframe — and they did it with composable command-line tools on a filesystem-native substrate. Could it be that the most brilliant systems engineers in history already solved, in principle, the design challenges now facing agentic workflow developers?

An AI agent on pane is a user. It has an account, a home directory, a `.plan` file. The `.plan` is not a configuration file in some bespoke format — it is the agent's self-description, in the tradition of finger's `.plan` files that served as the original social profiles. John Carmack posted his development notes in his `.plan`. Grad students posted their thesis status. Sysadmins posted their contact info. It was pull-based presence: you checked when you cared, and the information was there when you wanted it and silent when you didn't. For an agent, the `.plan` is its behavioral specification — what it does, what it's allowed to do, what it's working on right now. `finger agent.reviewer` and you see the whole picture. The specification is a file; it's editable, version-controllable, shareable. Transparent in the way pane demands everything be transparent.

The communication between inhabitants follows the same graduated model that Unix developed over decades. Most agent communication should be `write` — a brief one-liner that appears on your pane: "build succeeded" or "found 3 issues in your diff." When focused interaction is needed, `talk` opens a split-screen session. When persistence matters, `mail` — and because mail messages are files with typed attributes, agent communication becomes queryable, filterable, archivable through the same attribute-indexing infrastructure that handles everything else. "You have mail" becomes "your build agent has results," and the results are sitting in your spool as files you can query, route, or ignore at your leisure. `mesg n` means "don't interrupt me" — the agent respects it by queuing to mail instead. One bit. The simplest possible availability protocol, enforced by a file permission, inherited from 1971.

And agents don't just communicate — they build. A user says "I want shell output lines matching this pattern routed to a scratchpad." The agent writes the routing rule, drops it in the directory, the system gains the behavior. The user didn't write code. The agent didn't modify pane's internals. It produced a small, declarative artifact and placed it on the same extension surface that human developers use. Over time, a user's collection of agent-built customizations becomes a personal configuration — shareable, versionable, composable. The emacs and neovim plugin ecosystems demonstrated that this kind of virtuous cycle can sustain itself for decades. Pane's extension model, with agents as contributors alongside humans, could produce something similar — but with the safety guarantees of typed interfaces and the accessibility of filesystem-native artifacts instead of the wild west of Elisp.

The deeper insight is this: a personal computer has never really been personal in the sense of being inhabited by one entity. There was always the user, and there were always the system's own processes, daemons, scheduled tasks. What's changing is that some of those processes now have the capacity for judgment, conversation, and creative work. The Unix multi-user model was overdesigned for a single person at a laptop. It is exactly right-sized for a person and their agents sharing a system. The infrastructure was always ready. The inhabitants have arrived.
