## 3.1 Vasconcelos: Fundamentals of Session Types

**Source:** Vasconcelos, "Fundamentals of Session Types" (SFM 2009, LNCS 5569; extended version in Information and Computation 217, 2012). Available at <https://www.di.fc.ul.pt/~vv/papers/vasconcelos_fundamental-sessions.pdf>

**Source:** Honda, Vasconcelos, Kubo, "Language Primitives and Type Discipline for Structured Communication-Based Programming" (ESOP 1998, LNCS 1381, pp. 122--138). The original session types paper. ETAPS Test-of-Time Award 2019.

**Source:** Honda, "Types for Dyadic Interaction" (CONCUR 1993). The first session types paper.

### What are session types?

Session types are types for communication channels that describe the _entire protocol_ of interaction between two endpoints, not just individual messages. They were introduced by Honda (1993, for two-party dyadic interaction) and extended by Honda, Vasconcelos, and Kubo (1998, for the pi calculus).

The key conceptual move: rather than typing individual messages, you type the _conversation_. A session type specifies:

- What messages are sent and received, and in what order
- Where choices (branching) occur and who makes them
- How the protocol terminates
- How the protocol may loop

An individual message type tells you "this is an int." A session type tells you "first you send an int, then I send back a bool, then either of us can choose to continue or stop." The entire behavioral contract is in the type.

### Formal grammar

Vasconcelos builds the language gradually: starting from simple input/output, adding choice, then recursive types, then replication, then subtyping. The types are generated by:

```
S ::= !T.S          send a value of type T, continue as S
    | ?T.S           receive a value of type T, continue as S
    | +{l_i: S_i}    internal choice: select a label l_i, continue as S_i
    | &{l_i: S_i}    external choice: offer labels, continue as selected S_i
    | mu X. S         recursive session type
    | X               type variable (for recursion)
    | end             session termination
```

The semicolons encode sequential ordering -- this is a _conversation_, not a set of messages. The sequencing is what makes session types fundamentally different from, say, a protobuf message definition or an enum of request variants. The type describes _time_.

### Duality

The central operation. Every session type S has a dual, written S-bar, obtained by swapping send/receive and internal/external choice:

```
dual(!T.S)          = ?T.dual(S)
dual(?T.S)          = !T.dual(S)
dual(+{l_i: S_i})  = &{l_i: dual(S_i)}
dual(&{l_i: S_i})  = +{l_i: dual(S_i)}
dual(mu X. S)       = mu X. dual(S)
dual(end)           = end
```

Duality is an involution: dual(dual(S)) = S. If one endpoint follows S, the other must follow dual(S). This is the mechanism that guarantees protocol compatibility -- the two sides of a conversation are structurally complementary.

### Linear types and session fidelity

Vasconcelos reconstructs session types inside a linear pi calculus. The key insight: linearly-qualified channel references are used exactly once per step, preventing aliasing. Each channel is characterized by two distinct variables, one for reading and one for writing. Scope restriction binds them together, establishing the correspondence between the two ends.

This gives three guarantees, formalized as theorems:

- **Session fidelity** (subject reduction / type preservation): if a process is well-typed and takes a step, the resulting process is well-typed at the updated session type. The type _advances_ with the conversation. After sending an int on a channel typed `!int.?bool.end`, the channel's type becomes `?bool.end`. The type tracks where you are in the protocol.

- **Communication safety** (progress for typed reductions): no unexpected messages. A channel typed as `?int.S` will receive an int, not a string. A well-typed program cannot send on a channel that expects to receive, or vice versa.

- **Protocol completion**: the linear discipline ensures the protocol is followed to termination -- you cannot silently abandon a session. Every channel must be consumed. This is the linear logic principle: resources cannot be discarded.

### Subtyping

Vasconcelos introduces subtyping on session types. The key principle is that subtyping follows a _covariant/contravariant_ pattern analogous to function types:

- Output is covariant: `!S.T <: !S'.T'` when `S <: S'` and `T <: T'`. You can send a more specific type than expected.
- Input is contravariant: `?S.T <: ?S'.T'` when `S' <: S` and `T <: T'`. You can accept a more general type than expected.
- Internal choice is covariant in the label set: `+{l_i: S_i} <: +{l_j: S_j}` when {l_i} is a subset of {l_j}. Fewer choices is a subtype (you commit to less).
- External choice is contravariant: `&{l_i: S_i} <: &{l_j: S_j}` when {l_j} is a subset of {l_i}. Handling more cases is a subtype.

Subtyping preserves safety: if S <: S', then anything expecting S' can safely interact with a process following S.

### Linear vs shared channels

Vasconcelos distinguishes:

- **Linear channels**: used by exactly one thread at each endpoint. The session type governs the full conversation.
- **Shared (unrestricted) channels**: may be used by multiple threads for connection initiation. Once a session is established over a shared channel, the resulting session channel is linear.

This maps directly to the client-server pattern: a server listens on a shared channel; each accepted connection produces a linear session channel for the specific conversation.

### What session types do NOT do

Session types govern the _protocol structure_ between two endpoints. They do not:

- Specify the content of messages beyond their type (e.g., value constraints, range restrictions)
- Handle more than two parties without decomposition into binary sessions (multiparty session types exist but are a separate theory, introduced by Honda, Yoshida, and Carbone in 2008)
- Address concerns like authentication, encryption, or transport reliability
- Guarantee liveness (only safety -- the protocol won't go wrong, but it might not terminate)

### How this informs pane

Vasconcelos establishes the theoretical soundness of what pane is doing: using linear types to enforce protocol compliance between communicating processes. The linear/shared distinction maps directly to pane's architecture -- servers listen on shared channels, each client connection is a linear session. The duality operation is how `par` derives the compositor's protocol view from the client's. Subject reduction is why a conversation that type-checks at the beginning stays correct throughout.

## 3.2 Caires-Pfenning Correspondence (Linear Logic <-> Session Types)

**Source:** Caires and Pfenning, "Session Types as Intuitionistic Linear Propositions" (CONCUR 2010, LNCS 6269). Extended version: "Linear Logic Propositions as Session Types" (MSCS 2013). Available at <https://www.cs.cmu.edu/~fp/papers/mscs13.pdf>

**Source:** Wadler, "Propositions as Sessions" (ICFP 2012, pp. 273--286; JFP 24(2--3), 2014). Available at <https://homepages.inf.ed.ac.uk/wadler/papers/propositions-as-sessions/propositions-as-sessions.pdf>

### The correspondence

Caires and Pfenning (2010) established a Curry-Howard correspondence between propositions of intuitionistic linear logic and session types. Wadler (2012) independently established a correspondence with classical linear logic. Together, these show that session types are not merely a useful programming abstraction -- they are the computational content of linear logic proofs.

This is the same kind of correspondence that relates types to propositions and programs to proofs in the simply-typed lambda calculus (the original Curry-Howard correspondence). But where Curry-Howard links _sequential computation_ to _intuitionistic logic_, the Caires-Pfenning correspondence links _concurrent communication_ to _linear logic_. The new ingredient -- linearity -- captures the resource-sensitive nature of communication: a message, once sent, is consumed.

The correspondence operates at three levels:

1. **Propositions = Session types.** A linear logic proposition IS a session type. The connective A tensor B IS the session type "send A, continue as B." This isn't an analogy or a mapping -- the mathematical objects are identical, viewed from different perspectives.

2. **Proofs = Processes.** A proof in linear logic IS a well-typed concurrent program. Each inference rule corresponds to a process operation: sending, receiving, choosing, offering. A proof tree IS a process tree. If you can write the proof, the program is correct. If you can't write the proof, the program violates its protocol.

3. **Cut elimination = Communication.** The central theorem of proof theory -- that every proof with cuts can be reduced to a cut-free proof -- corresponds to the operational fact that all communications eventually resolve. A cut in the proof corresponds to a private channel between two processes. Eliminating the cut corresponds to the two processes communicating and resolving. Since cut elimination terminates (a theorem of linear logic), communication completes (a theorem about the process system).

### The correspondence table

**Classical linear logic (Wadler's CP):**

| Linear Logic | Session Type            | Operational Meaning                              |
| ------------ | ----------------------- | ------------------------------------------------ |
| A tensor B   | output A, continue as B | Send a channel of type A, then behave as B       |
| A par B      | input A, continue as B  | Receive a channel of type A, then behave as B    |
| A plus B     | internal choice         | Select one of two branches                       |
| A & B        | external choice         | Offer two branches for the other party to select |
| 1            | close                   | Terminate the session (send close signal)        |
| bottom       | wait                    | Wait for termination (receive close signal)      |
| !A           | server/replication of A | Offer session A to arbitrarily many clients      |
| ?A           | client request for A    | Request a session of type A                      |

Note on naming conventions: there is a confusing inversion between the logical connective names and the session type operations. In Wadler's classical correspondence, tensor (traditionally "multiplicative and") corresponds to _output_, and par (traditionally "multiplicative or") corresponds to _input_. This is because the names come from the logical side, not the operational side. The `par` crate's naming follows operational convention: `Send` sends, `Recv` receives.

**Intuitionistic linear logic (Caires-Pfenning):**

| ILL Connective | Session Type                | Meaning                    |
| -------------- | --------------------------- | -------------------------- |
| A tensor B     | Send A, then behave as B    | Output                     |
| A lolli B      | Receive A, then behave as B | Input (linear implication) |
| A plus B       | Internal choice             | Select a branch            |
| A & B          | External choice             | Offer branches             |
| 1              | Close                       | Terminate                  |
| !A             | Shared channel              | Replicable service         |

The key difference: Caires-Pfenning uses intuitionistic linear logic (no negation, asymmetric sequents), which gives a _provider/client_ distinction. Every process provides exactly one session and uses zero or more others. Wadler uses classical linear logic (symmetric sequents), which treats all channels uniformly -- no privileged "provided" channel. The `par` crate follows Wadler's classical approach.

### What the correspondence buys you

**Deadlock freedom from cut elimination.** The cut rule of linear logic corresponds to parallel composition with a shared channel. Cut elimination (a proof-theoretic property -- every proof with cuts can be reduced to a cut-free proof) corresponds to the fact that all communications eventually resolve. Since cut elimination terminates, so does communication. This gives deadlock freedom as a theorem, not a heuristic.

**Compositional reasoning.** Each process corresponds to a proof. Composing processes (connecting them via a channel) corresponds to composing proofs via the cut rule. The composition is type-safe if and only if the proof composition is valid. This means you can reason about the correctness of a composition from the types alone, without looking at the implementation.

**Conservation of resources.** Linear logic's resource sensitivity (every proposition must be used exactly once) maps to the requirement that every channel endpoint is used exactly once. No leaked connections, no double-use. A message must be consumed by exactly one receiver. A channel must be closed, not abandoned.

### The identity and cut rules as process operations

- **Identity** (axiom A |- A): corresponds to _forwarding_ -- a process that connects two channels directly, acting as a transparent relay. In `par`, this is the `link` method.
- **Cut** (from Gamma |- A and A, Delta |- B, conclude Gamma, Delta |- B): corresponds to _parallel composition_ -- spawn two processes communicating on a private channel of type A. In `par`, this is `fork_sync`.

Cut reduction (simplifying a cut) corresponds to one step of communication between the two processes. The entire cut elimination process corresponds to running the concurrent program to completion.

### Wadler's CP and GV

Wadler defines two calculi:

- **CP** (Classical Processes): directly corresponds to classical linear logic. Proofs are processes. Very close to the logic but alien as a programming language.
- **GV** (a linear functional language): a more conventional programming language with session-typed channels. Looks like a functional language with linear types.

Wadler gives a translation from GV to CP, showing that GV's session types are sound by reduction to the logic. The critical result: _deadlock freedom for GV follows from the translation to CP, which follows from cut elimination in classical linear logic_.

The `par` crate is operationally closer to GV than to CP -- it feels like Rust with session-typed channels, not like a proof calculus. But the guarantees come from the CP layer.

### How this informs pane

The Caires-Pfenning correspondence is why session types are not just a convenient API but a principled foundation. The guarantees (deadlock freedom, protocol fidelity, resource conservation) are theorems of linear logic, not properties of a particular implementation. Any correct implementation of session types inherits these guarantees. Committing to session types means pane's inter-component protocols inherit them by construction.

The `par` crate explicitly cites Wadler's CP as its theoretical basis. The crate is named after the par (⅋) connective of linear logic.

## 3.3 How `par` Implements Session Types in Rust

**Source:** <https://github.com/faiface/par> (v0.3.10, MIT license, by Michal Strba/faiface)
**Source:** <https://docs.rs/par/latest/par/>
**Source:** Source code: lib.rs, exchange.rs, queue.rs, server.rs (read in full)

### Design philosophy

The `par` crate describes itself as "a full implementation of propositional linear logic" as session types for Rust. It is based on Wadler's CP (classical linear logic correspondence). The key claim: no knowledge of linear logic is required to use the library, but the guarantees (protocol adherence, deadlock freedom) follow from the logical foundations.

The crate is built on async/.await and is runtime-agnostic (though it provides optional Tokio integration). It uses no unsafe code.

### The Session trait

The core abstraction:

```rust
pub trait Session: Send + 'static {
    type Dual: Session<Dual = Self>;

    fn fork_sync(f: impl FnOnce(Self::Dual)) -> Self;
    fn link(self, dual: Self::Dual);
}
```

Key properties:

- `Dual` is an involution: `Dual::Dual = Self`. This is the type-level encoding of session duality.
- `fork_sync` creates a matched pair of endpoints. The closure receives one endpoint (the Dual); the function returns the other. This is the operational cut rule -- it creates a private channel between two processes.
- `link` connects a session endpoint to its dual, forwarding all communication. This is the operational identity/axiom rule.
- The unit type `()` implements `Session` with `Dual = ()` -- it is the terminated session (both 1 and bottom, since classical linear logic identifies them under MIX).
- The `Send + 'static` bound means session endpoints can cross thread boundaries. This is essential for concurrent programming but also means the types must be owned values, not references.

### Send and Recv

Defined in the `exchange` module. These are the workhorses:

```rust
pub struct Recv<T, S: Session = ()> {
    rx: oneshot::Receiver<Exchange<T, S>>,
}

pub struct Send<T, S: Session = ()> {
    tx: oneshot::Sender<Exchange<T, S::Dual>>,
}

enum Exchange<T, S: Session> {
    Send((T, S)),
    Link(Recv<T, S>),
}
```

Implementation details:

- Built on `futures::channel::oneshot` channels internally. Each send/recv step is a fresh oneshot channel.
- `Send` is non-blocking: `send(value)` returns immediately with the continuation. Internally, it creates a fresh oneshot pair for the continuation, sends the value and one end of the continuation through the oneshot, and returns the other end.
- `Recv` is async: `recv()` returns a `Future` that resolves to `(T, S)`.
- The `Exchange` enum handles both direct sends and forwarding (linking). When a `Send` is linked to a `Recv`, no actual send/recv occurs -- the linked `Recv` is forwarded through the exchange channel, and the `recv()` loop follows the chain until it finds a real `Send`.
- Both types are `#[must_use]`, producing compiler warnings if dropped. They are also `!Clone` and `!Copy` by virtue of containing oneshot channels.

**Duality is automatic:**

```rust
impl<T, S: Session> Session for Recv<T, S> {
    type Dual = Send<T, S::Dual>;
    // ...
}

impl<T, S: Session> Session for Send<T, S> {
    type Dual = Recv<T, S::Dual>;
    // ...
}
```

So `Dual<Recv<A, Recv<B>>>` = `Send<A, Send<B>>`. The duality recurses through the continuation, flipping each layer.

### Linear logic correspondence in par

| par type                     | Linear logic    | Meaning                  |
| ---------------------------- | --------------- | ------------------------ |
| `Recv<A, B>`                 | A tensor B      | Output A, continue as B  |
| `Send<A, B>`                 | A-perp par B    | Input A, continue as B   |
| `Recv<Enum{V1(S1), V2(S2)}>` | A plus B        | Internal choice (select) |
| `Send<Enum{V1(S1), V2(S2)}>` | A-perp & B-perp | External choice (offer)  |
| `()`                         | 1 / bottom      | Session termination      |

Note the naming inversion: `Recv` corresponds to tensor (output in the logic), `Send` corresponds to par (input in the logic). This is because the names follow the _receiver's_ perspective: a `Recv<A, B>` is a channel from which you receive A. But in the logic, the _provider_ of a tensor channel is the one who sends A. The crate prioritizes programmer intuition over logical naming.

### Branching with enums

Choice/branching uses standard Rust enums rather than dedicated `Choose`/`Offer` types. This is a design decision that avoids introducing new types -- Rust's enum is already a tagged union with exhaustive matching.

```rust
enum Operation {
    CheckBalance(Send<Amount>),
    Withdraw(Recv<Amount, Send<Result<Money, Error>>>),
}

// Client sends a choice:
type ClientSession = Send<Operation>;
// Server receives and matches:
type ServerSession = Recv<Operation>;  // = Dual<ClientSession>
```

The `choose` method provides ergonomic selection. Here is how it works mechanically:

```rust
impl<T> Send<T, ()> {
    pub fn choose<S: Session>(self, choice: impl FnOnce(S) -> T) -> S::Dual {
        S::Dual::fork_sync(|session| self.send1(choice(session)))
    }
}
```

When you write `session.choose(Operation::CheckBalance)`, this is what happens:

1. `Operation::CheckBalance` is an enum variant constructor of type `Send<Amount> -> Operation`.
2. `choose` calls `fork_sync` on `Dual<Send<Amount>>` = `Recv<Amount>`, creating a fresh endpoint pair.
3. Inside the fork closure: the enum variant constructor is called with one endpoint (`Send<Amount>`), wrapping it in `Operation::CheckBalance(Send<Amount>)`. This is then sent through the original session channel via `send1`.
4. `choose` returns the other endpoint: `Recv<Amount>` (the dual).

The result: the client sends the choice AND gets back the continuation channel for that choice's protocol, all in one expression. The server receives the enum, pattern-matches, and extracts the embedded session endpoint for the chosen branch.

### Recursion

Recursive protocols use recursive enum types:

```rust
enum Counting {
    More(Recv<i64, Recv<Counting>>),
    Done(Send<i64>),
}
```

No `Box` or `Arc` needed -- the `Recv`/`Send` wrappers provide the indirection (they contain `oneshot` channels internally, which are heap-allocated). Implementation uses loops with session variable reassignment:

```rust
let mut numbers: Recv<Counting> = /* ... */;
loop {
    match numbers.recv1().await {
        Counting::More(n_session) => {
            let (n, next) = n_session.recv().await;
            total += n;
            numbers = next;
        }
        Counting::Done(report) => break report.send1(total),
    }
}
```

This is idiomatic Rust: a `loop` with reassignment to the session variable, rather than actual recursion. The `Recv<Counting>` in the `More` variant provides the recursion point.

### Queue module

A standardized recursive pattern for streaming:

```rust
pub struct Dequeue<T, S: Session = ()>  // Dual: Enqueue<T, S::Dual>
pub struct Enqueue<T, S: Session = ()>  // Dual: Dequeue<T, S::Dual>
```

Equivalent to the recursive type:

```rust
enum Queue<T, S: Session> {
    Item(T, Dequeue<T, S>),
    Closed(S),
}
```

`Enqueue` provides `push(item)` (returns a new `Enqueue` for more items) and `close()` (returns the continuation S). `Dequeue` provides `pop()`, `fold()`, `for_each()`, and conversion to a `futures::Stream`.

Push is non-blocking, pop is async. FIFO order guaranteed. The module provides convenience methods that eliminate boilerplate: `fold` and `for_each` handle the recursive loop internally, `into_stream` adapts to the standard `Stream` trait.

### Server module

Handles dynamic numbers of clients. Three components:

- **Server<Connect, Resume, ConnectionData>**: Central event loop. `poll()` receives connection and resumption events. `suspend()` creates a `Connection` handle for a client, associating arbitrary connection-local data.
- **Proxy<Connect>**: Handle for initiating new connections. Clonable (via closure pattern to enforce scope separation). `connect()` starts the connection protocol.
- **Connection<Resume>**: Suspended connection. `resume()` re-enters the server's event loop.

Critical invariant: **no two of these three types may exist in the same scope**. Not even two Proxies. This is enforced by the method signatures -- `Proxy::clone` doesn't return a new Proxy, it passes one into a closure:

```rust
proxy1.clone(|proxy2| {
    // proxy2 visible here, proxy1 is not
});
// proxy1 visible here, proxy2 is not
```

The scoping restriction exists for deadlock freedom. If a `Server` and a `Connection` were in the same scope, you could call `resume()` and then `poll()`, but the `resume()` would block waiting for the server to poll, and the poll would never happen because resume hasn't returned. The closure pattern makes this impossible syntactically.

The server module docs cite "Client-Server Sessions in Linear Logic" (2021) by Qian and Kavvos as the theoretical basis. The Server/Proxy/Connection pattern implements _coexponentials_ -- a formal treatment of client-server interaction in linear logic that goes beyond the standard exponentials (!A / ?A).

### MIX rule

The `par` crate describes itself as "linear logic with MIX." The MIX rule in linear logic says: from |- Gamma and |- Delta, conclude |- Gamma, Delta. Operationally, this allows two independent process groups to exist in parallel without sharing any channels. Standard linear logic does not include MIX. Including it means processes can exist side-by-side without being connected -- which is necessary for practical programming (you need to be able to have independent computations running concurrently).

In `par`, MIX manifests as the ability to have multiple independent sessions in scope simultaneously, as long as they are not connected in a cycle. The unit type `()` being self-dual (Dual<()> = ()) is related -- it means a terminated session from either perspective is the same, which is only valid under MIX (in standard linear logic, 1 and bottom are distinct).

### What par does NOT do

- **No transport layer.** Par creates in-memory channel pairs via `futures::channel::oneshot`. It does not handle serialization, sockets, or networking. For pane, this means par handles the _protocol structure_ but pane must build the transport (unix sockets + postcard serialization) separately and map it onto session-typed interactions. The `dialectic` crate (by Bolt Labs) does provide transport-polymorphic session types with a `Transmitter`/`Receiver` backend trait and a `dialectic-tokio-serde` integration layer, demonstrating that the transport bridge _is_ solvable -- but par doesn't solve it.

- **No runtime enforcement of linearity.** Rust has affine types (values can be dropped), not linear types (values must be used). Par's session endpoints are `#[must_use]` and will generate compiler warnings if dropped, but dropping a session endpoint is not a hard compile error. It causes a runtime panic ("sender dropped" / "receiver dropped") in the counterpart. The `#[must_use]` attribute is the best Rust offers, and it covers the common case (forgetting to use a value), but it doesn't prevent deliberate drops or crash-induced drops. - **Not dyn-compatible.** The Session trait has `impl FnOnce` parameters and an associated type with a `Self` bound, preventing use as a trait object. You cannot have `Box<dyn Session>` or dispatch over session types dynamically. Session types are resolved entirely at compile time.

- **No explicit error handling in the protocol.** Par has no built-in mechanism for protocol errors (deserialization failure, transport disconnection). Errors must be modeled as explicit branches in the session type (e.g., `Result<Success, Error>` as a choice) or handled outside the session type system.

### How this informs pane

The `par` crate maps session type theory to Rust idioms effectively. The key implications for pane:

1. **Pane's session types (from the architecture spec) are directly expressible in par.** The `PaneSession` / `PaneActive` / `CompSession` types in the spec are valid par types. The enum-based branching maps naturally to pane's request/event model -- `PaneRequest` and `PaneEvent` are already enums.

2. **Transport is pane's responsibility.** Par provides protocol structure; pane must bridge between par's in-memory channels and unix socket + postcard wire format. This is a non-trivial integration layer but a solvable one (dialectic's existence proves the pattern works).

3. **The server module's Proxy/Connection pattern maps to pane-comp's multi-client model.** Each pane-native client connects via a Proxy and gets a Connection for each session.

4. **Dropping is the gap.** Par relies on Rust's `#[must_use]` + runtime panics for linearity enforcement. A crashed client that drops its session endpoint will cause a panic on the compositor side unless explicitly handled. Pane needs a crash-recovery strategy that accounts for this.

5. **calloop integration.** Par is async-native (futures-based). Pane uses calloop for its event loop. Calloop provides a `futures` module (behind a cargo feature) that includes an executor insertable as an `EventSource`. This executor can drive par's futures within a calloop event loop. The integration path exists but needs prototyping -- the key question is whether calloop's single-threaded executor + par's `fork_sync` (which is synchronous and blocking for the forked closure) interact cleanly. ## 3.4 Deadlock Freedom

**Sources:** Caires-Pfenning (2010), Wadler (2012), Dardha-Gay "A New Linear Logic for Deadlock-Free Session-Typed Processes" (FoSSaCS 2018), Kokke-Dardha "Deadlock-Free Session Types in Linear Haskell" (Haskell 2021), Gay "Linear Logic, Session Types and Deadlock-Freedom" (Simons Institute talk, 2016).

### What is the guarantee?

Deadlock freedom means: in a well-typed system of communicating processes, it is impossible for all processes to be simultaneously waiting for each other, forming a cycle of blocked communications where no process can make progress.

More precisely, the guarantee is **global progress**: if the system has not terminated, at least one process can take a step.

This is a _static_ guarantee: it is checked at compile time, not at runtime. A well-typed program is deadlock-free by construction. No runtime deadlock detector is needed. No timeouts-as-deadlock-workaround.

### How the guarantee works (Caires-Pfenning / Wadler approach)

The guarantee follows from three constraints, each a consequence of the type system:

**1. Tree-shaped process topology.** The cut rule of linear logic composes two processes that share exactly one channel. Repeated application of cut builds a tree: each process is connected to its parent by one channel and to its children by one channel each. There are no cycles in the communication graph.

Since deadlock requires a cycle of dependencies (A waits for B, B waits for C, C waits for A), and the topology is acyclic by construction, deadlock cannot occur.

**2. Linear channel usage.** Each channel endpoint is used by exactly one process. No sharing, no aliasing. This prevents the "multiple waiters" scenario that can create unexpected dependency cycles.

**3. Cut elimination terminates.** Cut elimination in linear logic is strongly normalizing -- it always terminates. Since cut elimination corresponds to communication, all communication eventually completes.

### What constraints does it impose?

**The tree restriction is the big one.** Only tree-shaped process networks are typable. This means:

- Two processes can share at most one channel between them.
- No process can be connected to two other processes that are also connected to each other (that would form a cycle).
- Patterns like Milner's cyclic scheduler, dining philosophers (the classic deadlock example), or any ring topology are rejected by the type system.

In `par`, this is enforced by `fork_sync`: it creates exactly one channel (two endpoints), gives one to the spawned closure, and returns the other. The only way to connect processes is through this mechanism, which structurally produces trees.

The server module enforces this even more strictly: Server, Proxy, and Connection must never be in the same scope. This prevents the formation of cycles between the server event loop and client sessions.

### What does it cost?

**Expressiveness.** Some deadlock-free programs are rejected. A ring of three processes passing tokens is deadlock-free (if all pass in the same direction), but the type system rejects it because the topology is cyclic. The guarantee is conservative -- it rejects some safe programs to ensure no unsafe programs are accepted.

**Architecture.** The topology of communicating processes must be a tree (or forest). Processes cannot form peer-to-peer meshes or rings. If two processes need to communicate bidirectionally, they share one channel, not two.

**Coordinator pattern.** Multi-party interactions must be mediated by a coordinator process that holds all the endpoints. In par, this is demonstrated with a 3-player game example: a coordinator receives all three player sessions and then orchestrates. This is more restrictive than true multiparty session types but is compositionally cleaner (Lane: I think this is where our pane-router will need to shine).

### Alternative: priority-based deadlock freedom (Kobayashi / Dardha-Gay)

Kobayashi (2006) and later Dardha-Gay (FoSSaCS 2018) developed an alternative approach:

- Annotate each channel with a _priority_ (a natural number).
- A process may hold multiple channels, but must always communicate on the lowest-priority channel first.
- This prevents circular wait even in cyclic topologies, because priorities impose a total order on communication.

Dardha and Gay formalized this by replacing the cut rule with a _cycle_ rule (allowing cyclic connections) and adding priority checks. They proved a cycle-elimination theorem that generalizes cut elimination.

The priority approach allows strictly more programs (cyclic topologies are fine if priorities are consistent) but is more complex to use and reason about. The `par` crate does not implement priorities -- it uses the simpler tree restriction. This is the right choice for pane: the tree restriction matches the natural client-server topology, and the simplicity is worth the expressiveness trade-off.

### Practical implications for pane

The tree restriction is not a significant constraint for pane's architecture. Pane's topology is naturally tree-shaped:

```
        pane-comp (compositor)
       /    |    \
  client  client  client     (pane-native apps)

  pane-route (router) <--- clients connect independently
  pane-roster (directory) <--- clients connect independently
```

Each client connects to each server independently. Servers do not form cycles with each other (roster doesn't call route, route doesn't call roster). The topology is a forest of client-server trees.

Where it _could_ bite: if pane ever needs server-to-server communication that forms a cycle (e.g., route queries roster, and roster queries route). The current architecture avoids this by design -- queries flow in one direction. If bidirectional server-to-server communication becomes necessary, it would need to be mediated through a coordinator or restructured to avoid cycles. This is a constraint to maintain as the system grows (Lane: we need to consider this carefully, but we may see benefits from submitting to this constraint in terms of stability; in this case the issue is how do we make this constraint ergonomic to work with).

## 3.5 Synthesis: What Committing to Session Types Means for Pane

### Discipline before theory, theory after discipline

BeOS shipped in 1996. Honda's session types paper was published in 1993. The Caires-Pfenning correspondence -- the result that connects session types to linear logic and gives them their deepest guarantees -- came in 2010. The working implementation of a real-time, stable, pervasively concurrent desktop environment predated the theoretical framework that explains _why_ it worked by over a decade.

This is not a coincidence, and it's not an embarrassment for the theory. It's the normal relationship between engineering practice and formal methods. BeOS's engineers didn't need linear logic to build a stable message-passing system. What they had was _discipline_: the BMessage/BLooper model forced every component to be a self-contained message processor. Each BLooper had its own thread and its own message queue. Communication was asynchronous and typed (BMessage carried typed fields). Components didn't share memory; they exchanged messages. The protocol wasn't formalized, but it was _conventional_ -- every Be developer learned the same patterns, followed the same rules, and the system was stable because those rules were good.

Session types formalize exactly this kind of discipline. What BMessage enforced by API convention ("you send a B_REPLY after receiving a B_REQUEST_COMPLETED"), session types enforce by the compiler ("after Recv<Request>, the type is Send<Reply>"). What the BLooper threading model achieved by runtime structure (one thread per handler, message queue per thread, no shared mutable state), session types achieve by static structure (one channel per conversation, linear ownership per endpoint, no aliasing). The guarantee is the same -- protocol adherence, no stuck states, local reasoning about each component -- but the enforcement moves from developer skill to compiler verification.

Pane stands on both. The empirical proof that message-passing discipline produces stable, compelling desktop systems -- that's BeOS. The theoretical proof that this discipline can be enforced statically, with guarantees that hold by construction -- that's session types. Pane doesn't need session types because BeOS was wrong. Pane needs session types because BeOS was _right_, and now there's a way to get the same result with machine-checked guarantees instead of relying on every contributor maintaining the discipline by hand.

### What compile-time protocol verification changes

A desktop environment is not a web server or a database. It runs continuously, it has dozens of interacting components, and when it stops working the user cannot interact with the machine to diagnose the problem. The cost of a protocol bug in a desktop environment is categorically different from a protocol bug in a request/response service: you can't curl a hung compositor.

Session types change the nature of protocol correctness from a runtime concern to a compile-time one. The conversation structure -- what is sent, in what order, by whom -- is verified before the program runs. This doesn't eliminate all bugs (message _content_ can still be wrong, transport can still fail, timing can still be off), but it eliminates the class of bugs where the system gets into a state that the designer didn't anticipate because two components are out of sync about where they are in their conversation.

For a system like pane, where the compositor, shell, router, roster, and store are all separate processes with independent development trajectories, this matters concretely. When someone changes the compositor's protocol (adds a new request variant, restructures the handshake sequence), every client that doesn't update will fail to compile. Not fail at runtime when the user happens to trigger the changed code path. Fail at compile time, with a type error pointing at the mismatch. Protocol evolution becomes a refactoring operation, not a debugging expedition.

The `ProtocolState` runtime state machine that pane-proto currently uses achieves a weaker version of this: it catches some protocol violations at runtime. Session types subsume it entirely and move the checking to compile time. The state machine becomes the type. The runtime check becomes a type check. The `ProtocolState` enum and its transition logic become dead code.

### What the tree topology means architecturally

The tree topology constraint -- the price of deadlock freedom -- deserves careful thought, not as a limitation but as a design guide.

The constraint says: processes form trees, not graphs. Each process is connected to its parent by one channel and to its children by one channel each. No cycles. Two processes that need to talk share exactly one channel.

For pane, this is not an imposition; it's a description of the architecture that already exists. The compositor is the root. Clients connect to it as children. Each client independently connects to the router, the roster, the store -- each connection is a separate tree. The topology is a forest: multiple independent trees, one per server.

Where the constraint becomes a _design guide_ is in future growth. If pane-route needs information from pane-roster (e.g., "what services can handle this content type?"), the natural impulse is to have route call roster. If roster also needs to call route (e.g., "route this service announcement"), you have a cycle. The tree constraint says: don't do that. Instead, either route queries roster but roster never queries route (unidirectional), or a client mediates (the pane-app kit sends to both and correlates the results), or the information is materialized (roster writes to a file that route watches via pane-notify).

Each of these alternatives is a better architecture than a bidirectional server-to-server dependency. The constraint is pushing toward looser coupling, unidirectional data flow, and materialized views -- all patterns that make the system easier to reason about, easier to test, and more resilient to failure. The tree constraint is not just preventing deadlock; it's preventing architectural complexity that would be hard to manage even without deadlock.

### The open questions

**Transport bridge.** Par operates on in-memory `futures::channel::oneshot` pairs. Pane communicates over unix sockets with postcard serialization. These are different worlds. The gap must be bridged, and this is the hardest integration question.

Three approaches, roughly ordered by how much of par's guarantee they preserve:

1. _Session types as protocol specification only._ Define session types in par, use them for compile-time checking and in-memory testing, but implement the actual socket transport separately with a hand-written state machine that mirrors the session type. The session type is the single source of truth for the protocol structure, but the wire implementation doesn't use par's channel primitives. This is pragmatic and loses the least -- the protocol structure is still verified, the transport is implemented in pane's native idiom (calloop + unix sockets), and par's runtime is not in the hot path.

2. _Adapter layer._ Wrap par's channels with a serialization boundary. Each `send()` serializes with postcard and writes to the socket; each `recv()` reads from the socket and deserializes. This preserves par's runtime guarantees but introduces a per-message serialization/deserialization overhead and requires bridging par's async model with calloop's callback model.

3. _Transport-polymorphic session types._ Use (or build) a session type library that abstracts over the transport, like dialectic's `Transmitter`/`Receiver` backend trait. This is the cleanest solution but would mean either extending par, adopting dialectic, or building a new library.

The first approach is likely where pane starts. The session type is the spec; the transport is the implementation; they're kept in sync by the types (if you change the session type, the transport code stops compiling because it references the same enums and structs). Par's in-memory channels are used for testing. The wire transport is pane's own code.

**Crash handling.** Rust has affine types, not linear types. A process can crash and drop its session endpoints. When this happens, the counterpart's next `send()` or `recv()` will panic ("sender dropped" / "receiver dropped"). For a desktop environment, this is not acceptable -- the compositor cannot panic because a client crashed.

The strategy must be: catch panics at session boundaries (using `std::panic::catch_unwind` or equivalent), treat a dropped session as a "client died" event, and clean up the associated resources. The session type doesn't model this because crash is not a protocol event -- it's a failure of the protocol's preconditions. Pane needs a crash boundary around every client session: something that catches the panic, logs it, and removes the dead client's pane from the layout. This is analogous to how BeOS's BLooper would clean up when a client thread died -- the mechanism is different but the requirement is identical.

**calloop + async integration.** Par is built on futures. Pane uses calloop. Calloop's `futures` module provides an executor that can be inserted as an event source, and par's `fork_sync` is synchronous (it runs the closure to completion before returning). The integration path is: spawn par futures into calloop's executor, dispatch their wakeups through the calloop event loop.

The tension is between calloop's strength (shared mutable state via `&mut State` in callbacks, no interior mutability needed) and par's async model (futures that need to be polled, state captured in closures). A calloop callback that drives a par session would look like: on socket readable, deserialize the message, feed it to the session's next recv, get back the continuation session and the callback's next state. This is a state machine driven by calloop events where the session type tracks the current protocol state. It could work, but it's the kind of integration that needs a prototype before committing to it in a spec.

The `notgull` blog post on calloop notes the core trade-off: calloop is for single-threaded apps that spend most of their time waiting for events, with simple shared mutable state. Async/await is for scalable networking with actor-model patterns. Pane's compositors falls in calloop's sweet spot (single-threaded, event-driven, shared state). Par's sessions are inherently async. The bridge is calloop executing par's futures, not par replacing calloop.

**Relationship to existing pane-proto code.** The current pane-proto crate defines `PaneRequest`, `PaneEvent`, `ProtocolState`, `Value`/`Compute` polarity markers, `ServerVerb`, and typed views. Under session types:

- `PaneRequest` and `PaneEvent` become branches in session type enums (they already are, structurally).
- `ProtocolState` becomes redundant -- the session type _is_ the state machine, enforced at compile time.
- `Value`/`Compute` polarity markers (from the old CBPV framing) are superseded. Session types have their own polarity: `Send` is "positive" (provides data), `Recv` is "negative" (demands data).
- `ServerVerb` and the typed views/builders pattern remain useful for _message content_, but the conversation structure moves to session types.
- The wire format (postcard over unix sockets) is unaffected. Session types govern protocol structure, not serialization.

### What becomes possible

Session types don't just verify existing designs -- they make certain designs _thinkable_ that would be too risky without static guarantees.

**Protocol as specification.** The session type IS the spec. Not documentation that drifts, not a state machine that might be out of sync with the actual message handlers -- the type that the compiler checks. When pane needs a new protocol (say, a session between pane-store and a query client), the designer writes the session type first, and the type IS the specification that both sides must implement. This is "protocol-first development," and it inverts the current approach where code comes first and documentation comes after.

**Protocol versioning as type evolution.** Protocol versions become different session types. A server that supports v1 and v2 offers a choice at connection time:

```rust
enum ProtocolVersion {
    V1(Recv<V1Active>),
    V2(Recv<V2Active>),
}
type Handshake = Send<ProtocolVersion>;
```

The type system ensures that once a version is selected, both parties follow the correct protocol for that version. No "I thought we were speaking v2 but you're sending v1 messages" bugs.

**Testable protocol models.** Create a pair of par endpoints in a test, run both sides in-memory against the session type, verify the conversation completes correctly. No sockets, no serialization, no server processes. This is strictly more powerful than testing individual message serialization round-trips: it tests the _conversation_, not the individual messages. A test can verify that after a CreatePane handshake and three WriteCells operations, a Close completes the session cleanly. The test is short, fast, and covers the protocol logic without infrastructure.

**Composition of protocol fragments.** Session type continuations allow protocol fragments to be defined independently and composed. A "pane lifecycle" fragment (create, configure, activate), an "active pane" fragment (write cells, set tag, receive events), and a "shutdown" fragment (close, cleanup) can be defined separately and composed:

```rust
type PaneSession = Lifecycle<Active<Shutdown>>;
```

Each fragment is testable independently. The composition is type-checked. This maps well to pane's modular design, where different kits handle different phases of interaction.

### Summary

Pane's commitment to session types via `par` is a bet on three things:

1. That the protocol structure of a desktop environment is rich enough to benefit from static verification. The answer is clearly yes -- the compositor, router, roster, store, and clients all have complex, stateful conversations that are currently verified only by testing and review.

2. That the tree topology constraint is compatible with the architecture. The answer is yes by design -- pane's client-server forest IS a tree topology. The constraint additionally prevents architectural decisions that would create hard-to-manage cyclic dependencies.

3. That the transport bridge is solvable. The answer is "yes, with work" -- other crates have solved it, the approach of using par for specification and testing while implementing transport independently is pragmatic, and the calloop integration path exists via the `futures` module.

BeOS proved that message-passing discipline makes desktop systems stable and compelling. Session types prove that this discipline can be enforced by the compiler. Pane inherits both results.
